## 一、系统概述

### 1.1 定位

卖家后台是农脉平台的 B 端管理工具，供入驻商户管理商品、处理订单、售后和查看经营数据。

**核心原则：平台控制客户关系，卖家只负责履约。**

卖家不应获得任何可用于绕过平台直接联系买家的信息。所有买家身份信息、联系方式、详细地址均由平台保管，卖家只看到履约所需的最小数据集。

### 发货流程（先用口语说明）

卖家端的发货不是“自己决定发哪家快递、自己填快递单号”，而是**平台先和合作快递公司谈好，具体这单走哪家由平台决定**。卖家只负责把平台给的面单打出来、贴到包裹上，然后确认发货。

对卖家来说，发货就 4 步：

1. 看到订单进入待发货
2. 点击「生成面单」
3. 平台自动选择合作快递并生成电子面单
4. 卖家打印面单、贴包裹、点击「确认发货」

卖家在整个过程中：
- 看不到买家的完整收货地址和真实姓名
- 不需要自己选择顺丰 / 圆通 / 中通
- 不需要手填快递单号

**举个例子：**

买家下单买了一袋大米。卖家后台看到这单待发货后，直接点「生成面单」。系统会按平台合作策略自动决定这单走哪家快递，比如顺丰，然后返回一张可打印的面单图和一个脱敏后的面单号。卖家把这张面单打印出来贴到包裹上，交给快递员后，再回后台点「确认发货」。这样这笔订单就正式进入“已发货”状态了。

## 二、隐私保护策略（核心改造）

### 2.1 设计目标

**防止卖家挖走客户**。卖家只能看到履约所需的最少信息，无法还原买家真实身份。

### 2.2 信息分级

#### 完全隐藏（卖家不可见）

| 信息 | 隐藏理由 | 替代方案 |
|------|----------|----------|
| 买家昵称 | 可在社交平台搜索定位用户 | 平台生成匿名编号（如"买家#A3x8K"） |
| 买家 userId | 技术标识可关联交叉查询 | 不返回 |
| 买家手机号 | 直接联系方式 | 不返回，通过平台虚拟中转号联系（见 2.7） |
| 买家头像 | 可辅助社交平台搜索 | 不返回 |
| 详细收货地址 | 定位买家住所/公司 | 完全不给卖家，发货通过平台统一面单（见 2.3） |
| 收件人真实姓名 | 真实身份信息 | 完全不给卖家，面单由平台生成 |
| 支付渠道 | 无履约必要 | 不返回 |
| 支付时间明细 | 无履约必要 | 只显示下单日期（不显示精确时间） |

#### 脱敏显示（卖家可见但已处理）

| 信息 | 脱敏规则 | 用途 |
|------|----------|------|
| 收货地区 | 只到省市区（如"广东省深圳市南山区"） | 卖家了解配送区域分布（仅参考，发货靠面单） |
| 物流单号 | 前4后4（如"SF12****7890"） | 卖家核对物流状态 |

#### 完整展示（卖家可见）

| 信息 | 理由 |
|------|------|
| 订单编号 | 业务流转唯一标识 |
| 订单状态 | 履约流程必需 |
| 商品名称、SKU、数量 | 备货发货必需 |
| 商品售价（= 成本 × 加价率） | 卖家可见售价，用于对账 |
| 订单总金额（按售价汇总） | 对账必需 |
| 下单日期（仅日期） | 发货时效管理 |
| 退换货状态与选项式理由 | 售后处理必需 |
| 汇总统计（销量、营收趋势） | 经营分析必需 |
| 商品库存、成本价、售价 | 商品管理必需 |

### 2.3 发货模式：平台统一面单

**关键设计**：卖家完全拿不到收货地址，发货通过平台生成的电子面单完成；具体合作快递由平台统一决定，卖家不参与选择。需要对接快递 API。

```
流程：
1. 买家下单 → 地址存入 Order.addressSnapshot（AES-256 加密存储）
2. 卖家在订单详情页点击「生成面单」
3. 平台后端按合作策略自动选择快递公司，解密地址并调用对应快递电子面单 API → 获取面单号 + 面单图片
4. 前端展示面单图片 → 卖家点击「打印面单」→ 贴包裹
5. 卖家点击「确认发货」→ 订单状态 PAID → SHIPPED
6. 平台自动订阅物流轨迹推送

卖家全程不接触任何收货地址信息，只看到：
- 配送区域（省市区级别，仅供参考）
- 面单图片（打印用，含完整地址但仅物理可见）
- 面单号（脱敏显示，如"SF12****7890"）
- 平台最终分配的快递公司名称（生成后可见）
```

> **当前产品口径说明（2026-03-10）**：卖家侧不应暴露“选择快递公司”这一步。若现阶段实现中仍存在 `carrierCode` 或下拉选择，仅视为过渡方案，后续应统一收敛为平台自动分配合作快递。

#### 快递 API 对接方案

```
支持的快递服务商（初期，平台合作侧）：
- 顺丰速运 (SF)
- 圆通快递 (YTO)
- 中通快递 (ZTO)
- 申通快递 (STO)
- 韵达快递 (YUNDA)
- 京东物流 (JD)
- EMS

API 能力：
- 电子面单下单（获取面单号 + 面单 PDF/图片）
- 物流轨迹订阅（Webhook 推送轨迹更新）
- 面单取消（发货前可撤销）

对接层设计：
- 统一接口抽象 ShippingProvider（Strategy 模式）
- 各快递商实现独立 Adapter
- 平台按合作策略自动选择承运商（如区域、品类、时效、成本、活动档期等）
- 当前为占位实现，接口已定义，方便后续对接真实 API
```

#### 发件人信息

面单上的发件人信息使用卖家企业的经营地址和客服电话（来自企业设置），不暴露平台内部信息。

#### 面单存储

```
面单数据存入 Shipment 记录：
- waybillNo: string      // 快递面单号（快递API返回）
- waybillUrl: string     // 面单图片 URL（存 OSS）
- carrierCode: string    // 快递公司编码
- carrierName: string    // 快递公司名称
```

### 2.4 买家匿名编号机制

每个买家在每个卖家店铺内有唯一的匿名编号，规则：

```
格式：买家#<6位随机字母数字>
生成时机：首次在该店铺下单时生成
存储：独立 BuyerAlias 表（userId + companyId 联合唯一），响应时投影到订单 DTO
      Order 表本身不存储 buyerAlias 字段，避免冗余
特性：
  - 同一买家在同一店铺的所有订单显示相同编号（卖家可识别回头客）
  - 同一买家在不同店铺的编号不同（跨店无法关联）
  - 编号不可逆推出 userId 或手机号
别名冲突重试：
  - 6位字母数字有 36^6 ≈ 21亿 种组合，冲突概率极低
  - 生成时若 alias 唯一约束冲突，自动重试（最多 3 次，每次重新生成随机串）
  - 3 次仍冲突则升级为 8 位（极端情况兜底）
```

### 2.5 退换货理由标准化

将自由文本理由替换为选项式理由（消除买家在文字中泄露联系方式的风险）：

```
退换货理由选项：
- QUALITY_ISSUE    — 质量问题
- WRONG_ITEM       — 发错商品
- DAMAGED          — 运输损坏
- NOT_AS_DESCRIBED — 与描述不符
- SIZE_ISSUE       — 规格不符
- EXPIRED          — 临期/过期
- OTHER            — 其他
```

`OTHER` 选项允许附加文字，但后端自动过滤手机号/微信号等联系方式模式（正则过滤）。

### 2.6 照片安全处理（EXIF + 内容扫描）

买家上传的退换货照片，后端接收时执行两层安全处理：

#### 第一层：EXIF 元数据剥离

剥离 GPS 定位、设备信息等 EXIF 元数据，防止卖家通过照片元数据获取买家位置。

#### 第二层：图片内容联系方式检测（OCR + 二维码）

买家可能在纸条、包装、截图中写入联系方式（手机号、微信号、二维码等），仅清理 EXIF 不够。

```
检测流程：
1. 二维码/条形码扫描：检测图片中的二维码/条形码
   - 命中 → 自动打码（高斯模糊覆盖二维码区域）
   - 工具：jsQR / zxing-wasm（轻量级，无需外部服务）

2. OCR 文字识别：提取图片中的文字
   - 对提取文字执行联系方式正则匹配（同 2.5 的 filterContactInfo 规则）
   - 命中手机号/微信号/QQ号 → 自动打码（模糊对应文字区域）
   - 工具：Tesseract.js（离线 OCR）或 阿里云 OCR（精度更高，占位对接）

3. 处理策略：
   - 自动打码后正常上传（不拒绝，避免影响正常售后流程）
   - 打码记录写入审计日志（便于争议回溯）
   - 如打码失败（识别不准确），图片仍然上传但标记为 needsReview
   - 管理后台可查看 needsReview 的图片进行人工复核

注意：此检测为尽力而为（best-effort），不保证 100% 拦截。
配合 2.8「隐私逃逸边界与处置规则」的卖家违规追责机制形成完整闭环。
```

### 2.7 虚拟中转号（卖家↔买家通话）

卖家有时需要联系买家确认售后细节（如换货规格、配送时间等）。为防止卖家获取买家真实手机号，采用**平台虚拟中转号**方案。

#### 工作原理

```
卖家拨打虚拟号 → 运营商平台 → 转接至买家真实手机号
买家看到来电显示为虚拟号（非卖家真实号码）
双方均不知道对方的真实手机号
```

#### 绑定规则

```
触发场景：卖家在订单详情或换货详情页点击「联系买家」
绑定关系：虚拟号 ↔ (卖家客服电话, 买家手机号, 订单ID)
有效期：绑定后 24 小时自动解绑
限制：
  - 每个订单最多绑定 3 次（防骚扰）
  - 仅 PAID/SHIPPED 状态的订单 或 REQUESTED/UNDER_REVIEW/APPROVED 状态的换货单可发起
  - 通话录音存档（用于争议仲裁）
  - 绑定记录计入审计日志
```

#### 技术实现

```
依赖服务：阿里云隐私号（AXB 模式）/ 腾讯云小号
API 接口：
  POST /seller/orders/:id/virtual-call    — 订单场景，绑定虚拟号
  POST /seller/replacements/:id/virtual-call — 换货场景，绑定虚拟号

返回：
  { virtualNumber: "170xxxx1234", expireAt: "2026-03-11T14:30:00Z" }

前端展示：
  弹窗显示虚拟号 + 有效期倒计时
  "请使用企业客服电话拨打此号码联系买家，有效期24小时"

当前状态：占位实现（返回模拟虚拟号），接口已定义
```

### 2.8 隐私逃逸边界与处置规则

技术手段无法 100% 阻止隐私泄露（如面单截屏、通话中口述联系方式等），因此需要配套**检测 + 追责 + 处罚**机制形成完整闭环。

#### 2.8.1 图片内容拦截（OCR / 二维码）

```
检测范围：
- 买家上传的退换货照片（上传时检测）
- 卖家上传的商品图片（上传时检测，防止在商品图中嵌入联系方式引导买家私联）

检测能力：
- 二维码/条形码 → 自动打码（高斯模糊覆盖）
- OCR 识别文字中的手机号/微信号/QQ号 → 自动打码
- 检测结果写入审计日志，标记 needsReview 的图片推送管理后台人工复核

技术栈：
- 二维码检测：jsQR / zxing-wasm（轻量级，服务端执行）
- OCR：Tesseract.js（离线）→ 后续可升级阿里云 OCR（精度更高）
- 图片处理：sharp（打码 + EXIF 剥离一并完成）

局限性：
- 手写文字识别率有限，无法保证 100% 拦截
- 故意用非标格式（如"壹叁捌"代替"138"）可绕过
- 配合下方违规追责机制弥补技术盲区
```

#### 2.8.2 卖家违规联系买家的追责机制

```
违规行为定义：
- 通过面单截屏获取买家地址后线下联系
- 通过虚拟号通话中索要/透露真实联系方式
- 在商品描述/图片/包裹中放置联系方式引导私联
- 利用技术手段（抓包、OCR 面单图片等）提取买家信息

检测手段：
- 虚拟号通话录音 → 关键词检测（"加我微信"/"手机号是"等）→ 自动标记 + 人工抽检
- 买家举报入口（App 内"商家私联举报"按钮）→ 进入管理后台审核队列
- 面单访问异常检测（短时间大量访问面单打印接口 → 告警）
- 商品审核时检测商品图片/描述中的联系方式

处罚阶梯：
- 第一次：警告 + 记录（企业信用分 -10）
- 第二次：限制虚拟号功能 7 天 + 企业信用分 -20
- 第三次：暂停企业 30 天（Company.status → SUSPENDED）
- 第四次：永久封禁企业（Company.status → BANNED）
- 严重情况（批量爬取、贩卖用户数据）：立即封禁 + 法律追责

信用分模型：
- 初始 100 分
- <80 分：标记为"需关注"，管理后台高亮
- <60 分：限制部分功能（如批量面单、虚拟号）
- <40 分：自动暂停企业，需申诉后恢复
```

#### 2.8.3 敏感操作审计全覆盖

```
所有涉及买家隐私的操作必须记录审计日志：

操作类型                        审计内容
─────────────────────────────────────────────
面单生成/打印                  staffId + orderId + IP + timestamp
面单打印代理访问                staffId + orderId + IP + User-Agent
虚拟号绑定                     staffId + orderId/replacementId + timestamp
虚拟号通话                     callId + duration + 录音文件ID
查看订单详情                   staffId + orderId + timestamp
查看换货详情                   staffId + replacementId + timestamp
批量操作（批量面单/批量发货）    staffId + orderIds[] + timestamp

审计日志保留期限：至少 2 年
管理后台可按 staffId / companyId / 时间范围 / 操作类型 查询审计记录
异常模式自动告警：
  - 同一 staffId 10分钟内查看 >50 个不同订单
  - 同一 companyId 1天内生成 >200 张面单（排除大促期间）
  - 非工作时间段的批量操作
```

#### 发货流程（平台面单模式）

```
1. 卖家点击「生成面单」
2. 平台按合作策略自动选择快递公司
3. 后端解密 addressSnapshot → 调用对应快递电子面单 API → 返回面单图片 URL + 脱敏面单号
4. 前端展示面单预览图
5. 卖家点击「打印面单」→ 浏览器打印面单图片
6. 卖家贴好面单后点击「确认发货」→ 调用 POST /seller/orders/:id/ship
7. 订单状态 PAID → SHIPPED，平台自动订阅物流轨迹
```

#### 关键隐私设计

- **买家编号**：显示 `buyerAlias`（如"买家#A3x8K"），不显示昵称
- **配送区域**：只显示省市区（如"上海市浦东新区"），仅供参考
- **收货地址/收件人**：完全不返回给前端，面单由平台生成
- **联系买家**：通过虚拟中转号，双方不知道对方真实号码
- **金额**：显示商品售价（= 成本 × 加价率），不显示平台利润分配
- **时间**：只显示日期（YYYY-MM-DD），不显示时分秒
- **不显示**：userId、手机号、头像、支付渠道、支付时间

#### 批量发货（OWNER / MANAGER）

OWNER 和 MANAGER 可选择多个待发货订单，使用批量生成面单 + 发货功能：
- 勾选多条 `status=PAID` 的订单
- 平台按每条订单自动分配合作快递
- 点击「批量生成面单」→ 调用 `/seller/orders/batch-waybill`
- 面单全部生成后，点击「批量打印」→ 合并打印所有面单
- 点击「批量确认发货」→ 调用 `/seller/orders/batch-ship`
- 返回每条结果（成功/失败原因）

#### 企业 AI 搜索资料 Card（新增，建议优先级高）

> **新增原因**：当前企业搜索与 AI 企业语义匹配已经开始使用 `企业名 / 简称 / 主营业务 / 地区 / 企业类型 / 特征标签` 等维度；仅靠自由文本 `企业简介 + highlights` 不足以稳定支持诸如”武汉有哪些公司””卖水果的公司””武昌区有机蔬菜合作社”这类查询。

**目标**：把原本散落在 `description / highlights` 里的搜索关键信息，收敛成可维护、可搜索、可排序的结构化资料。

> **与企业信息 Card 的关系**：「企业简介」（`description`）和「经营地址」（`address`）已在企业信息 Card 中维护，本 Card 不重复。如需地址结构化（`province/city/district`），在企业信息 Card 的 `address` 字段上升级，AI 搜索直接读取。

#### 建议新增字段

| 字段 | 类型 | 是否必填 | 用途 |
|------|------|:------:|------|
| 企业类型 | enum | ✅ | `farm / company / cooperative / base / factory / store`，支持”农场/公司/合作社/基地/工厂/店铺”查询 |
| 主营品类 | string[] | ✅ | 平台预置大类枚举（水果 / 蔬菜 / 粮油 / 肉禽 / 水产 / 茶叶 / 蜂蜜 / 乳制品 / 其他），下拉多选 |
| 主营产品关键词 | string[] | 选填 | 品类下的细分产品，自由补充（如”蓝莓 / 草莓 / 富硒粮油 / 有机五常大米”） |
| 服务地区 | string[] | ✅ | 自由输入，后端 trim + 去重 + 过滤空串（如”湖北 / 武汉 / 武昌区”），P2 再升级为结构化地址联动 |
| 产品特征 | string[] | ✅ | 产品本身的属性标签，平台预置（有机 / 可溯源 / 冷链 / 认证） |
| 供给方式 | string[] | 选填 | 交易与配送模式，平台预置（批发 / 零售 / 直供 / 同城配送 / 可预约考察） |
| 认证资质 | string[] | 选填 | 企业持有的认证，平台预置（有机认证 / 绿色食品 / 地理标志） |

**字段边界说明**：
- 「主营品类」是大类枚举，用于粗筛；「主营产品关键词」是品类下的细分，用于精确匹配
- 「产品特征」描述产品属性（有机、冷链）；「供给方式」描述交易模式（批发、直供）；两者不交叉
- 「认证资质」描述企业资质（有机认证），与「产品特征」中的”有机”是不同维度（企业级 vs 产品级）

#### 表单交互建议

- 企业类型：单选下拉
- 主营品类：平台预置枚举，多选 + 可搜索
- 主营产品关键词：TagInput，自由补充
- 服务地区：省 / 市 / 区三级联动选择，可多选
- 产品特征：平台预置标签，多选
- 供给方式：平台预置标签，多选
- 认证资质：平台预置标签，多选

#### 与现有字段的关系

| 当前字段 | 问题 | 改造建议 |
|----------|------|----------|
| `description`（企业信息 Card） | 文本可用，但卖家填写不稳定 | 保留在企业信息 Card，AI 搜索作为补充召回字段读取 |
| `address`（企业信息 Card） | 当前只有 `{text, lat, lng}`，无结构化省市区 | 在企业信息 Card 升级为 `{text, province, city, district, lat, lng}`，AI 搜索直接读取 |
| `profile.highlights.mainBusiness` | 目前只是一段字符串 | 升级为结构化「主营品类 + 主营产品关键词」，`mainBusiness` 保留为展示版摘要（由结构化字段自动拼接生成） |
| `profile.highlights.badges` | 目前语义混杂，既像标签又像营销文案 | 收敛为「产品特征 / 认证资质 / 供给方式」三个明确集合，`badges` 保留为展示版（由三个集合合并生成） |

#### AI 企业搜索对这些字段的直接收益

- `卖水果的公司` → 命中 `主营品类=水果`
- `武汉有哪些公司` → 命中 `服务地区=武汉`（读取企业信息 Card 的结构化地址）
- `武昌区有没有有机蔬菜合作社` → 命中 `服务地区=武昌区` + `主营品类=蔬菜` + `企业类型=合作社` + `产品特征=有机`

```json
}
```

```prisma
model CompanyProfile {
```

### 4.1 买家匿名编号

#### 数据模型变更

```prisma
model BuyerAlias {
  id        String   @id @default(cuid())
  userId    String
  companyId String
  alias     String   @unique  // 格式: "买家#<6位随机>"，全局唯一
  createdAt DateTime @default(now())

  user    User    @relation(fields: [userId], references: [id])
  company Company @relation(fields: [companyId], references: [id])

  @@unique([userId, companyId])  // 同一买家同一店铺只有一个编号
  @@index([companyId])
}
```

**注意**：Order 表不存储 buyerAlias。查询订单列表时，Service 层通过 userId + companyId 查 BuyerAlias 表，将 alias 投影到返回的 DTO 中。

#### 编号生成逻辑（含冲突重试）

```typescript
// 首次下单时自动生成，带冲突重试
async function getOrCreateBuyerAlias(userId: string, companyId: string): Promise<string> {
  const existing = await prisma.buyerAlias.findUnique({
    where: { userId_companyId: { userId, companyId } }
  });
  if (existing) return existing.alias;

  // 冲突重试：最多 3 次 6 位，兜底升级 8 位
  for (let attempt = 0; attempt < 4; attempt++) {
    const length = attempt < 3 ? 6 : 8;
    const alias = '买家#' + generateRandomAlphanumeric(length);
    try {
      const record = await prisma.buyerAlias.create({
        data: { userId, companyId, alias }
      });
      return record.alias;
    } catch (e) {
      if (e.code === 'P2002') continue;  // 唯一约束冲突，重试
      throw e;
    }
  }
  throw new Error('BuyerAlias generation failed after retries');
}
```

#### 面单图片安全策略

面单含完整收货地址，属于**高敏感资源**，必须防止卖家保存/传播：

```
1. 短时效签名 URL
   - 面单图片存 OSS 私有桶，不直接返回 OSS URL
   - 通过平台打印代理接口访问：GET /seller/orders/:id/waybill/print?token=xxx
   - token 为一次性签名，15 分钟过期，每次生成面单时下发新 token
   - 过期后必须重新请求（重新鉴权 + 记录审计）

2. 打印代理接口
   - 后端生成签名 URL → 返回面单图片流（Content-Type: image/png）
   - 响应头禁止缓存：Cache-Control: no-store, Pragma: no-cache
   - 图片上叠加动态水印："仅限订单#xxx履约使用 · {staffId} · {timestamp}"
   - 水印半透明覆盖地址区域，不影响快递扫码

3. 访问审计
   - 每次访问打印代理接口记录审计日志：staffId + orderId + timestamp + IP
   - 异常检测：同一 staffId 短时间大量请求面单 → 触发告警

4. 前端限制
   - 面单展示区域禁用右键菜单（onContextMenu preventDefault）
   - 打印按钮调用 window.print() 直接打印，不提供"另存为"入口
   - 注意：这些前端限制仅为增加门槛，非安全边界（截屏无法防止）
```

## 五、安全体系

### 5.1 认证隔离

```
买家 App  ←→ JWT (JWT_SECRET)       ←→ BuyerAuthGuard
卖家后台  ←→ JWT (SELLER_JWT_SECRET)  ←→ SellerAuthGuard
管理后台  ←→ JWT (ADMIN_JWT_SECRET)   ←→ AdminAuthGuard
```

三端使用完全独立的 JWT Secret，互不相通。卖家 Token 解码后包含 `{ staffId, userId, companyId, role, type: 'seller' }`。

### 5.2 多商户数据隔离

**强制规则**：所有卖家端查询自动注入 `companyId` 过滤。

```typescript
// 每个 Service 方法都从 @CurrentSeller() 获取 companyId
async getOrders(companyId: string, params: QueryDto) {
  return this.prisma.order.findMany({
    where: {
      items: { some: { companyId } },  // 至少有一个商品属于本公司
      ...filters
    }
  });
}
```

- 卖家只能看到自己公司的数据
- 无法通过 API 参数绕过 companyId 过滤

#### 混合订单（跨公司）处理策略

历史上可能存在一个订单包含多家公司商品的情况（混合订单）。明确处理规则：

```
1. 可见性：卖家可以看到包含自己商品的订单，但只能看到属于自己公司的商品行
   - 查询条件：items.some({ companyId }) — 只要有本公司商品就可见
   - 返回数据：items 过滤为仅 companyId 匹配的行
   - 订单金额：按本公司商品行的售价汇总（非整单金额）

2. 售后归属：按子订单项（OrderItem）级别归属
   - 换货请求：关联到具体 orderItemId，哪家公司的商品哪家处理
   - 整单换货：如果所有 items 都属于同一公司才允许整单操作
   - 跨公司整单换货：拒绝，提示买家按商品分别申请

3. 物流归属：每家公司独立发货
   - 一个混合订单可能产生多个 Shipment（每个公司一个）
   - 每个 Shipment 关联该公司负责的 OrderItems
   - 面单按公司分别生成

4. 未来方向：结算时按 companyId 拆分子订单（建议但非当前阶段实施）
```

> **当前实现状态（2026-03-12）**：订单列表/详情、换货、发货、报表统计均已按上述 companyId 规则处理。退款功能暂不开放，代码保留。
