# 后端接口对接清单（以 Repo 层为准）

> 目标：你只需要看这份文档 + `src/types/domain/*`，就能实现后端接口并完成联调。
>
> 范围：当前前端以 `src/repos/*.ts` 为“后端接入入口”，页面不会直接 `fetch()`；未来接入后端时只需要替换 Repo 内部实现即可。

---

## 0. 总体约定

### 0.1 返回结构（建议与前端 `Result<T>` 对齐）

前端通用结构见：`src/types/Result.ts`、`src/types/AppError.ts`。

建议后端所有接口统一返回：

```ts
type ApiResult<T> =
  | { ok: true; data: T }
  | { ok: false; error: { code: 'NETWORK' | 'NOT_FOUND' | 'FORBIDDEN' | 'INVALID' | 'UNKNOWN'; message: string; displayMessage?: string; retryable?: boolean } };
```

说明：
- `displayMessage`：给用户看的短文案
- `message`：给日志/排查用的原始错误信息
- `retryable`：前端可据此决定是否提示“重试”

> 如果后端更倾向于“HTTP 状态码 + 业务错误体”的风格，也可以只保证能映射到上述结构即可（前端在 ApiClient 里统一转换）。

### 0.2 鉴权（占位约定）

当前 Demo 未做登录，但为后端扩展预留：
- 认证头：`Authorization: Bearer <token>`
- `userId` 一律从 token 解析，不建议前端传 `userId`（Demo 中部分方法签名带 `userId`，后端实现时可忽略并以 token 为准）

### 0.3 分页约定

前端分页类型：`src/types/Result.ts` 里配合 `PaginationResult<T>`（见 `src/types`）。

建议统一：
- 入参：`page`（从 1 开始）、`pageSize`
- 出参：`{ items: T[]; nextPage?: number }`

### 0.4 认证（Auth）

对应：`src/repos/AuthRepo.ts`

- `POST /api/v1/auth/login` → `Result<AuthSession>`
  - body：`{ channel: 'phone', mode: 'code'|'password', phone, code?, password? }`
- `POST /api/v1/auth/register` → `Result<AuthSession>`
  - body：同上（注册流程）
- `POST /api/v1/auth/sms/code` → `Result<{ ok: true }>`
  - body：`{ phone }`
- `POST /api/v1/auth/oauth/wechat` → `Result<AuthSession>`
  - body：`{ authCode }`（微信授权码）
- `POST /api/v1/auth/oauth/apple` → `Result<AuthSession>`
  - body：`{ identityToken }`

说明：
- `AuthSession` 定义见 `src/types/domain/Auth.ts`
- 当前前端仅开放手机号登录/注册；邮箱渠道可后续再补

- `GET /api/v1/companies/{id}/events` → `Result<CompanyEvent[]>`
- `GET /api/v1/events/{id}` → `Result<CompanyEvent | undefined>`

### 2.3 预约（Booking）

对应：`src/repos/BookingRepo.ts`

- `GET /api/v1/bookings`（运营审核）→ `Result<Booking[]>`
- `GET /api/v1/companies/{id}/bookings` → `Result<Booking[]>`
- `GET /api/v1/groups/{id}/bookings` → `Result<Booking[]>`
- `POST /api/v1/bookings`（提交预约）→ `Result<Booking>`
  - body：`{ companyId, eventId?, date, headcount, identity, note?, contactName?, contactPhone? }`
- `POST /api/v1/bookings/{id}/review`（审核通过/驳回）→ `Result<Booking>`
  - body：`{ status: 'approved' | 'rejected', note? }`
- `POST /api/v1/bookings/{id}/invite`（发起成团邀请）→ `Result<Booking>`
  - body：`{ groupId }`
- `POST /api/v1/bookings/{id}/confirm-join`（用户确认参团）→ `Result<Booking>`
- `POST /api/v1/bookings/join-group`（一键参团入口）→ `Result<Booking>`
  - body：`{ companyId, groupId, headcount?, identity?, contactName? }`
- `POST /api/v1/bookings/{id}/paid`（支付完成）→ `Result<Booking>`

`identity` 建议枚举：
- `consumer` 消费者
- `buyer` 采购商
- `student` 学生
- `researcher` 科研人员
- `media` 媒体/自媒体
- `investor` 投资/合作方

### 2.4 考察团（Group）

对应：`src/repos/GroupRepo.ts`

- `GET /api/v1/groups` → `Result<Group[]>`
- `GET /api/v1/companies/{id}/groups` → `Result<Group[]>`
- `GET /api/v1/groups/{id}` → `Result<Group | undefined>`
- `POST /api/v1/groups` → `Result<Group>`

- `PATCH /api/v1/groups/{id}/status` → `Result<Group>`
  - body：`{ status }`

## 8. 备注：为什么前端 Repo 是“唯一接入点”

当前代码里：
- 页面/组件只调用 `Repo`，不会直接请求网络
- `Repo` 的返回统一是 `Result<T>`

因此后端同学要做的事情是：
1) 实现本文件列出的接口
2) 前端未来新增 `ApiClient` 后，把 `Repo` 内部的 `simulateRequest(mock)` 改成真实 HTTP 调用即可
