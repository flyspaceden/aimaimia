# 支付宝支付集成方案（收款 + 分润付款）

> **文档定位**：本文档是爱买买平台**支付宝收款与付款（分润出款）集成**的唯一权威来源，覆盖产品选型、申请流程、技术实现、法律合规与实施路线。
>
> **适用范围**：
> - 收款：App 内商品购买、VIP 礼包购买
> - 付款：用户分润奖励提现出款
>
> **不适用范围**：
> - 微信支付（另有独立文档，当前为占位实现，待 App 上架后接入）
> - 平台红包（Coupon）结算抵扣（走内部账本，不涉及真实资金流）

## 第一部分：收款系统（已实现）

### 1.1 产品选型

**已接入产品**：支付宝 **APP 支付**（`alipay.trade.app.pay`）

**选型理由**：
- 爱买买买家端是 React Native App，符合「APP 支付」产品定位
- 支付宝 APP 支付是 C 端支付最成熟的产品，日均交易量庞大，稳定性高
- SDK 完整，官方维护的 `alipay-sdk` Node.js 库覆盖所有接口

**不使用的产品**：
- ❌ 电脑网站支付（PC 场景，不适用于 App）
- ❌ 手机网站支付（H5，我们是原生 App）
- ❌ 当面付（线下二维码场景）
- ❌ 直付通（多商户资金通道，详见第二部分的方案对比）

### 1.2 商户资质与申请流程

#### 前置条件
- 企业营业执照（个体工商户也可，但后续分账/付款产品部分有门槛）
- 企业支付宝主账号（已完成企业实名认证）
- 法人身份证
- 对公银行账户
- App 的基本信息（名称、包名、签名、简介、图标）

#### 申请流程（收款侧）

| 步骤 | 操作 | 预计耗时 |
|---|---|---|
| 1 | 登录 [open.alipay.com](https://open.alipay.com/) 开放平台 | 即时 |
| 2 | 「开发者中心」→「网页&移动应用」→「创建应用」→ 选择「移动应用」 | 10 分钟 |
| 3 | 填写应用信息：App 名称、包名（Android）、Bundle ID（iOS）、应用图标 | 30 分钟 |
| 4 | 上传资质：营业执照、App 截图、应用下载地址（上架前可暂填开发地址） | 1 小时 |
| 5 | 生成并上传 RSA2 应用公钥（私钥本地保存） | 15 分钟 |
| 6 | 应用审核（支付宝侧人工审核） | 1-3 工作日 |
| 7 | 审核通过后，在「产品与功能」里**单独签约「APP 支付」产品** | 30 分钟 |
| 8 | 签约审核 | 1-3 工作日 |
| 9 | 下载应用证书 / 支付宝公钥证书 / 根证书（生产环境必须用证书模式） | 30 分钟 |
| 10 | 配置 `ALIPAY_NOTIFY_URL`（异步通知回调地址，必须 HTTPS 公网可达） | 15 分钟 |

**关键提醒**：
- **签名算法必须是 RSA2**（SHA256withRSA），RSA1 已废弃
- 生产环境**必须用证书模式**，不能用公钥模式
- 回调地址必须是 **HTTPS**（支付宝不允许 HTTP 回调）
- App 上架前可以用沙箱环境（[openhome.alipay.com/platform/appDaily.htm](https://openhome.alipay.com/platform/appDaily.htm)）做开发联调

#### 支付回调处理（payment.service.ts）

**回调入口**：`payment.controller.ts:52-99` → `POST /api/v1/payments/alipay/notify`（NestJS 全局前缀 `api/v1` + controller 路径 `payments/alipay/notify`）

**⚠️ WebhookIpGuard 必须配 IP 白名单**（生产环境）：
- `payment.controller.ts:53` `@UseGuards(WebhookIpGuard)` 拦截所有非白名单 IP 的请求
- `webhook-ip.guard.ts:39-44` 在 `NODE_ENV=production` 且 `WEBHOOK_IP_WHITELIST` 未配时**直接 throw ForbiddenException**，支付宝异步通知会全部 403 → 订单永远不会变 PAID
- 部署生产前必须在 `backend/.env` 配置：`WEBHOOK_IP_WHITELIST="<支付宝/微信回调 IP 段>"`（支持 CIDR）
- 详见 `docs/operations/阿里云部署.md` Step 10

**核心保障**（`payment.service.ts:227-595`）：
1. **Serializable 事务隔离级别**：防止回调并发导致的状态错乱
2. **CAS 状态转换**：Payment 状态 `INIT/PENDING → PAID`，Order 状态 `PENDING_PAYMENT → PAID`
3. **幂等控制**：重复回调直接跳过（L369-371）
4. **异常补偿**：订单已取消但支付成功 → 标记自动退款（L395-446）
5. **自动退款补偿 Cron**：`0 */10 * * * *` 重试失败的自动退款（L91-161）
6. **金额校验**：转换为分（整数）比较，避免 Float 精度丢失

## 第二部分：付款/分润出款系统（待实施）

### 2.1 业务需求背景

**场景描述**：
- 平台采用多层级分润树（VIP 三叉树 + 普通用户树），用户因下级消费获得分润奖励
- 分润结果计入用户的「奖励钱包」（`RewardAccount`），在 App 内可见余额
- 用户可**随时发起提现**，每笔都可以提，也可以攒着一次提
- 只有**用户层的分润**需要真实出款；其他池子（产业基金/慈善基金/科技基金/备用金）**仅做账本记录**，资金暂留公司账户统一调配
- 用户量增大后提现会频繁，需要考虑备付金、风控、对账压力

**明确范围**：
- ✅ 要做：用户 RewardAccount 余额 → 提现 → 真实打到用户支付宝账户
- ❌ 不做：慈善/科技/产业/备用金等池子的真实出款（仅记账）
- ❌ 不做：订单级分账（不需要订单一付款就拆到多方）
- ❌ 不做：卖家结算通道（走另一个独立流程，不在本文档范围）

### 2.2 产品选型：三方案对比

支付宝生态里能做「付款」的产品有三类，评估如下：

| 方案 | 产品名 | 适用场景 | 资质门槛 | 本项目适配度 |
|---|---|---|---|---|
| **A** | 互联网平台直付通 | 多商户电商资金通道，一级平台 + N 级商户 | 注册资本 ≥ 500 万，仅支付宝单通道 | ❌ **不适用**：强制仅支付宝收款，与未来接入微信冲突；且我们不需要多级商户资金拆分 |
| **B** | 交易结算分账 `alipay.trade.order.settle` | 从单笔交易收款账户拆给多个分账接收方 | 商户签约分账产品 | ❌ **不适用**：「慈善/科技/备用金」不是真实账户，无法作为分账接收方；用户层分润涉及多层上溯，一笔订单可能涉及十几个接收方，超出同步分账 5 个接收方的限制 |
| **C** | **商家转账到支付宝账户** `alipay.fund.trans.uni.transfer` | 企业主动给指定用户账户打款 | 商户签约「商家转账」产品 | ✅ **最佳选择**：完全匹配「用户提现主动出款」的业务形态，不改分润引擎，只接最后一公里 |

**最终选型：方案 C（商家转账到支付宝账户）**

**决策依据**：
1. **业务契合度**：用户的"分润奖励"本质是平台发给用户的**佣金/奖励**，以"商户付款"形式发放是最符合业务定性的方式
2. **技术契合度**：现有账本系统（`RewardAccount` + `RewardLedger` + `WithdrawRequest`）完全不需要改动分润引擎，只需在审批环节新增真实打款调用
3. **合规定位清晰**：平台作为电商经营主体收款，用户提现时平台向用户支付佣金/奖励，资金流与业务流完全一致，**不涉及二清**
4. **工作量最小**：只需新增 1 个 Service 方法 + 1 个查询补偿方法 + 数据模型扩展 4 个字段
5. **不绑死通道**：未来可以并行接入微信支付收款，不受直付通「仅支付宝」约束

### 2.3 商家转账产品技术规格

#### 接口信息

| 项目 | 值 |
|---|---|
| 接口名 | `alipay.fund.trans.uni.transfer` |
| 文档地址 | https://opendocs.alipay.com/apis/api_28/alipay.fund.trans.uni.transfer |
| 请求方式 | POST（通过 alipay-sdk 封装） |
| 签名算法 | RSA2（SHA256withRSA） |
| 业务场景码 | `DIRECT_TRANSFER`（直接转账）/ `PAYFUND`（资金付款）等 |
| 收款方类型 | `ALIPAY_LOGON_ID`（支付宝登录号）/ `ALIPAY_USER_ID`（支付宝 UID） |

#### 限额与约束（重点）

| 维度 | 限制 | 说明 |
|---|---|---|
| **单笔上限（个人账户）** | 5 万元/笔 | 超过需拆单 |
| **单笔上限（企业账户）** | 无限制 | 本项目用户都是个人账户，受 5 万限制 |
| **企业账户年度出款** | 理论不限 | API 付款不受个人支付宝「20 万/年」的年度额度约束 |
| **收款方实名要求** | 必须实名 | 收款方必须是已实名认证的支付宝账户 |
| **姓名强匹配** | 必须一致 | 请求参数中 `payee_real_name` 必须与支付宝预留姓名完全一致，否则失败 |
| **出款账户来源** | 商户余额 | 需预先充值商户备付金作为蓄水池 |
| **异步结果返回** | 部分场景 | 大额或需风控审核的订单会异步返回结果，需查询补偿 |
| **查询接口** | `alipay.fund.trans.common.query` | 用订单号或支付宝流水号查询转账状态 |

#### 风控规则（实际运营经验）

支付宝对企业 API 出款虽比个人收款码宽松，但仍有异常监测：
- ⚠️ 同一收款方短时间内多笔、相同金额、规律性出款 → 判定为洗钱/套现可疑
- ⚠️ 账户突然出现大量新收款方 → 触发二次审核
- ⚠️ 平均单笔金额极小（如 1-2 元）但笔数极多 → 异常
- ⚠️ 非营业时段大额出款 → 风险告警

### 2.4 商户资质与申请流程

#### 前置条件
- 已完成支付宝收款侧的企业应用（第一部分 1.2 的流程）
- 企业支付宝主账号余额或备付金充足
- 法人实名、对公账户完整

#### 申请流程（付款侧）

| 步骤 | 操作 | 预计耗时 |
|---|---|---|
| 1 | 登录 [open.alipay.com](https://open.alipay.com/) 开放平台 | 即时 |
| 2 | 进入已创建的应用 → 「产品与功能」→ 搜索「商家转账到支付宝账户」 | 5 分钟 |
| 3 | 点击「立即签约」，填写业务场景说明（**关键：要写清"用户分润奖励提现"场景**） | 30 分钟 |
| 4 | 上传业务证明材料：App 截图显示"奖励钱包"页面、提现流程截图、分润规则说明文档 | 1 小时 |
| 5 | 填写**预计月出款规模**、**单笔平均金额**、**笔数** | 15 分钟 |
| 6 | 支付宝风控审核（可能要求补充材料，如公司银行流水、完税证明等） | 3-10 工作日 |
| 7 | 签约通过后，申请额度提升（默认额度通常不够，需要根据业务量申请上调） | 3-5 工作日 |
| 8 | 往商户号充值首期备付金 | 1-2 工作日 |
| 9 | 联调测试（支付宝提供沙箱环境） | 1-2 周 |

**申请时的重要注意事项**：
- **场景说明务必诚实完整**：隐瞒「多层级分润」可能导致后期资质被撤销
- **预计月出款规模不要低估**：低估会导致额度不够用，后期频繁申请提升额度
- **提前准备完税证明**：支付宝可能要求提供个税代扣代缴的证明材料

#### 需要新增字段（WithdrawRequest）

```prisma
model WithdrawRequest {
  // === 原有字段保持不变 ===
  id                String  @id @default(cuid())
  userId            String
  amount            Float                        // 申请总额（毛额）
  channel           WithdrawChannel              // ALIPAY | WECHAT | BANKCARD
  status            WithdrawStatus
  accountType       String                       // VIP_REWARD | NORMAL_REWARD

  taxRate           Float?                       // 实际预扣率快照
  taxWithheld       Float?                       // 代扣税额

  // === 新增：支付宝收款信息 ===
  payeeAccount      String?                      // 用户支付宝登录号（手机号/邮箱）
  payeeRealName     String?                      // 收款方真实姓名（必须与支付宝预留姓名一致）

  // === 新增：出款执行 ===
  batchId           String?                      // 所属批次 ID（T+1 合并出款用）
  providerPayoutId  String?                      // 支付宝返回的转账订单号
```

```
Cron (每 10 分钟)       Backend                   支付宝             DB

     │                     │  └─WAIT: 继续下轮      │                │
```

#### 流程 4：失败回滚与重试

失败场景分两类：

**可重试失败**：网络错误、支付宝限流、临时风控
- 自动进入下一批次重试
- 最多重试 3 次，超过后人工介入

**不可重试失败**：
- 收款方账号不存在
- 姓名不匹配
- 支付宝账号未实名
- 处理：标记 `FAILED`，**原子回滚**：
  - `RewardAccount.frozen -= amount`
  - `RewardAccount.balance += amount`（退回可用余额）
  - `WithdrawRequest.status = FAILED`
  - `RewardLedger` 新增一条 RELEASE 记录（把 FROZEN 的释放回 AVAILABLE）
  - App 内消息通知用户："提现失败，已退回奖励钱包，请检查账号信息"

### 2.7 备付金管理策略

#### 备付金蓄水池机制

商家转账走的是**预充值**模式，流程是：

```
公司对公账户 ──(充值)──▶ 支付宝商户备付金 ──(API 出款)──▶ 用户支付宝账户
```

#### 运营策略建议

| 策略 | 说明 |
|---|---|
| **首期充值** | 根据预计首月出款规模的 1.5 倍充值（冗余安全垫） |
| **阈值告警** | 余额 < 日均出款 × 3 天 → 发送邮件/短信告警 |
| **自动化告警 Cron** | 每小时查询商户余额，写入 `AlipayPayoutReserve` 快照 |
| **余额不足保护** | 出款前强制校验：若余额 < 批次总额，**拒绝批次执行**，避免出一半失败一半 |
| **月度对账** | 每月对 `AlipayPayoutReserve` 流水与支付宝对账文件做人工核对 |

## 第三部分：法律与合规

> ⚠️ **重要声明**：本部分汇总现行法规要点，**不构成专业法律意见**。正式上线前必须由专业税务师和律师出具书面意见。

### 3.1 电商法相关（《中华人民共和国电子商务法》）

爱买买平台属于《电商法》第九条定义的「电子商务平台经营者」，适用以下义务：

| 条款 | 义务 | 对本项目的影响 |
|---|---|---|
| 第 10 条 | 依法办理市场主体登记 | 平台本身需持有营业执照 |
| 第 27 条 | 核验入驻商家身份、地址、联系方式等 | 卖家入驻（merchant-application 模块）已覆盖 |
| 第 28 条 | 向市场监管部门报送商家身份信息、向税务部门报送商家涉税信息 | **必须对接税务报送**（详见 3.3） |
| 第 38 条 | 对消费者尽到审核义务，否则承担连带责任 | 与售后退换货系统相关 |
| 第 53 条 | 通过第三方支付服务提供者进行支付，需向消费者提供支付信息 | **支付通道必须透明、合规** |

**与分润通道的直接关联**：
- 电商法要求平台「不得通过关联支付进行违规操作」，所以**资金流必须与业务流一致**。用户的提现必须能追溯到具体的分润来源，`RewardLedger` 的 `refType + refId` 设计正好满足这点。

### 3.2 个税代扣代缴义务（重点）

#### 法规依据

- 《中华人民共和国个人所得税法》
- 《中华人民共和国个人所得税法实施条例》
- 《关于互联网平台企业为平台内从业人员办理扣缴申报有关事项的公告》
- 《互联网平台企业涉税信息报送规定》（2025 年 6 月国务院通过）

#### 核心义务

**平台在向个人用户付款时，有法定的代扣代缴义务**。不代扣不仅是违规，而且按《税收征收管理法》第六十九条，可被处以应扣未扣税款 50% 至 3 倍的罚款。

#### 分润定性：三种可能的所得类型

| 所得类型 | 适用场景 | 税率 | 起征点 | 预扣计算 |
|---|---|---|---|---|
| **劳务报酬所得** | 推广佣金、分销返点、用户因下级消费获得的分润 | 20%~40% 累进预扣 | 单次 800 元以上 | 单次 ≤ 4000：(收入 - 800) × 20%<br>单次 > 4000：收入 × (1 - 20%) × 预扣率 |
| **偶然所得** | 抽奖、随机红包、促销活动礼品 | 20% 固定 | 无起征点 | 收入 × 20% |
| **经营所得** | 带货博主、团长等持续经营性质 | 5%~35% 累进 | 年度汇缴 | 按全年综合计算 |

#### 爱买买分润的定性分析（倾向性建议）

基于对业务模型的分析，各类奖励的**建议定性**（**仅供参考，最终以税务师意见为准**）：

| 奖励类型 | 触发方式 | 建议定性 | 建议预扣率 | 说明 |
|---|---|---|---|---|
| **VIP 三叉树分润** | 下级消费上溯 | 劳务报酬 | 20% | 用户向下级推广获得的返佣 |
| **普通用户树分润** | 下级消费上溯 | 劳务报酬 | 20% | 同上 |
| **抽奖奖励** | 抽奖中奖 | 偶然所得 | 20% | 明确的偶然性 |
| **签到/活动红包** | 行为触发 | 偶然所得 | 20% | 非对价性质 |

#### 代扣代缴在数据模型中的体现

已在 2.5 节的 `WithdrawRequest` 新增字段中预留：
- `taxCategory` — 所得类型
- `taxRate` — 预扣率快照（防止后续税率调整影响历史数据）
- `taxWithheld` — 代扣税额
- `netAmount` — 实际到账金额

**计算时机**：用户发起提现时，根据其 `RewardAccount.type` + 账本 `RewardLedger` 的来源，推算所得类型，计算代扣税额。

**示例**：
```
用户申请提现：500 元（VIP 分润）
所得类型：劳务报酬
因单次 < 800 元，不到起征点 → 暂不扣缴（但需在年度汇缴时累计申报）
实际到账：500 元

用户申请提现：1500 元（VIP 分润）
所得类型：劳务报酬
预扣税额：(1500 - 800) × 20% = 140 元
实际到账：1360 元
```

### 3.3 互联网平台企业涉税信息报送（2026 新规）

#### 法规依据
- **《互联网平台企业涉税信息报送规定》**（2025 年 6 月国务院第 61 次常务会议通过）
- 2026 年 4 月 1 日起常态化监管

#### 报送义务

| 项目 | 要求 |
|---|---|
| **报送主体** | 平台内部：电商平台、网约车、外卖、直播电商、分享经济等全覆盖 |
| **报送内容** | 平台内从业人员/商户的身份信息、收入信息、涉税信息 |
| **报送频率** | **季度报送**，季末后 1 个月内 |
| **报送对象** | 国家税务总局电子税务局 |

#### 违规处罚

| 情形 | 处罚 |
|---|---|
| 未按时报送 | 最高罚款 **50 万元** |
| 虚假报送 / 隐瞒 / 拒绝 | 最高罚款 50 万元 + **停业整顿** |
| 影响平台纳税信用 | 后续税务稽查加严 |

#### 实施路径

有两种方案可选：

**方案一：对接阿里云税务报送 API**（推荐）
- 优点：自动化，错误率低
- 缺点：需要额外对接工作

**方案二：每季度人工导出 Excel 提交**
- 优点：实施成本低
- 缺点：人工出错风险，用户量大后不可持续

**本项目建议**：首年（用户量小）走方案二，次年用户量起来后切换方案一。

#### 报送数据结构（需要在系统中沉淀）

应建立一张 `PlatformTaxReport` 表，按季度归档：

```prisma
model PlatformTaxReport {
  id              String   @id @default(cuid())
  quarter         String                          // 2026Q1
  userId          String
  totalPaid       Float                           // 季度累计已付金额
  totalTax        Float                           // 季度累计代扣税额
  incomeCategory  TaxCategory                     // 所得类型
  reportStatus    ReportStatus                    // PENDING | SUBMITTED | ACCEPTED | REJECTED
  submittedAt     DateTime?
  createdAt       DateTime @default(now())

  @@unique([quarter, userId, incomeCategory])
  @@index([reportStatus])

```

### 3.4 分销层级的法律风险（⚠️ 关键提醒）

#### 法规依据
- 《禁止传销条例》（国务院令第 444 号）
- 第 7 条明确的传销三要件：
  1. **入门费**：要求缴纳费用取得加入资格或间接缴纳
  2. **拉人头**：按发展下线人数计酬
  3. **团队计酬**：按下线销售业绩层级计酬

#### 爱买买当前模型的法律风险评估

| 风险点 | 现状 | 风险等级 |
|---|---|---|
| **入门费** | VIP 礼包购买才能进入 VIP 分润树 | 🔴 **高** — VIP 礼包可能被认定为变相入门费 |
| **拉人头** | 推荐码邀请 + 分润树滑落 | 🟡 **中** — 如果 VIP 礼包可被消费掉（有真实商品价值），可以洗清 |
| **团队计酬** | VIP 三叉树 BFS 多层滑落、普通树 k 层上溯 | 🔴 **高** — 多层级团队计酬是传销认定的核心要件 |

#### 司法实践参考

- **2019 年云集微店案**：被市场监管局认定传销，罚款 958 万元。原因：三级分销（核心会员 / 高级主管 / 经理），按下线数量和业绩计酬。
- **2020 年后合规调整趋势**：多数平台将分销层级压缩到 **2 级以内**（自己 + 直接下级），绕开「三级以上团队计酬」的硬红线。
- **京东芬香 / 拼多多「多多进宝」**：均采用二级分销，严格控制层级。

#### 对本项目的直接影响

⚠️ **在花大力气对接支付宝付款通道之前，强烈建议先做分润模型的法律合规审查**。如果模型被裁定需要裁剪层级，分润引擎（`bonus-allocation.service.ts`）需要重做，届时付款通道的对接工作可能要返工。

**强烈建议的前置动作**：

1. **委托专业合规律师**出具《爱买买分润模型合法性意见书》
2. **重点确认**：
   - VIP 礼包是否构成变相入门费？
   - VIP 三叉树 + 普通用户树的层级深度是否突破「二级」红线？
   - 利润六分中的「上溯分配」比例是否会被认定为团队计酬？
3. **根据意见调整**：可能的裁剪方向包括：
   - 将 VIP 礼包改为"购买商品组合"（必须有真实等值商品交付）
   - 上溯分润限定为「自己的直接推荐人」一层
   - 取消层级奖励，改为固定比例平台奖励

### 3.5 资金二清合规

#### 什么是二清

- **一清**：持牌支付机构直接结算给商户（合规）
- **二清**：资金先进入平台账户（非持牌），再由平台拆分给下游商户（**违规**）

#### 爱买买的合规定位

| 场景 | 是否合规 | 说明 |
|---|---|---|
| 商品销售收款 → 进入平台支付宝账户 | ✅ 合规 | 平台本身是电商经营主体，消费者直接与平台交易 |
| 平台统一结算给卖家（卖家不直接收款） | ⚠️ 需审慎 | 严格意义上有二清嫌疑，建议未来用直付通改造 |
| 平台向用户支付分润奖励（本文档范围） | ✅ 合规 | 本质是平台向用户支付佣金/奖励，不是代第三方收款 |

#### 分润出款的合规定性

本项目的「分润提现」场景**不涉及二清**，理由：
1. 资金先入平台账户，属于平台的**经营收入**
2. 分润是平台向用户支付的**奖励/佣金**，是平台的成本支出
3. 资金流与业务流完全一致：消费者 → 平台 → 平台奖励给推广者
4. 和「代卖家结算」（二清）有本质区别：平台不是在代任何第三方转钱

**结论**：用商家转账 API 付款给用户**没有二清风险**，可以放心实施。
