## 1. 核心目标

将当前“抽象的 VIP 升级”改造成一个更容易转化的购买流程：

1. 用户在“我的”页点击 VIP 按钮
2. 弹出 VIP 权益弹窗
3. 点击“购买 VIP 礼包”
4. 进入“VIP 赠品选择页”
5. 用户选择一个赠品方案
6. 进入结账页
7. 已登录用户直接下单
8. 未登录用户先通过“手机号 + 验证码”完成自动登录/注册
9. 支付成功后，系统为用户开通 VIP，并发起赠品履约

该方案要同时满足以下业务约束：

1. 购买 VIP 礼包是成为 VIP 的入场费
2. VIP 礼包订单不计入普通树有效消费首单
3. VIP 礼包订单不计入 VIP 上溯有效消费首单
4. 用户成为 VIP 后，后续真实商品消费单才开始参与分润计数
5. VIP 必须以支付成功为准开通，不能在点击购买时提前生效
6. VIP 购买不支持退款
7. VIP 赠品必须从管理后台的奖励商品中选择
8. 奖励商品可以同时用于 VIP 赠品与抽奖奖品
9. 奖励商品由管理后台维护，但发货仍然走现有 seller / 平台公司发货链路
10. 奖励商品是独立商品域，不参与普通商城陈列

## 2. 业务口径

### 2.1 购买对象

用户买的不是“普通商品”，而是：

- 一个 `VIP 资格包`
- 外加一个用户可选的 `赠品履约商品`

前端展示上是“选一个 VIP 赠品方案”。

后端业务上仍然是“购买同一类 VIP 礼包”，只是附带不同赠品。

### 2.2 身份确认口径

收货信息不等于登录信息。

必须明确区分：

1. 账号身份：手机号 + 验证码
2. 收货信息：收货人姓名、收货手机号、收货地址

规则：

1. 已登录用户：
   收货手机号只属于地址，不影响账号手机号
2. 未登录用户：
   必须通过手机号验证码先完成身份确认
3. 后端判断：
   - 手机号已注册：直接登录
   - 手机号未注册：自动注册并登录

### 2.3 VIP 生效口径

VIP 只在支付成功后开通。

不能在以下时机开通：

1. 点击“购买 VIP 礼包”时
2. 进入结账页时
3. 创建 CheckoutSession 时
4. 订单待支付时

只能在支付回调成功、订单状态确认成功后开通。

### 2.4 分润口径

VIP 礼包订单仅用于：

1. 开通 VIP
2. 生成 VIP 购买记录
3. 发放“购买 VIP”相关的推荐奖励
4. 触发赠品履约

VIP 礼包订单不得用于：

1. 写入 `VipEligibleOrder`
2. 写入 `NormalEligibleOrder`
3. 增加 `VipProgress.selfPurchaseCount`
4. 增加 `NormalProgress.selfPurchaseCount`
5. 设置普通消费意义上的首单资格

### 2.5 退款口径

VIP 礼包订单不退款。

明确规则：

1. 不支持用户自助退款
2. 不支持常规客服人工退款
3. 不进入普通售后退款链路
4. 不应在订单详情页展示“申请退款”入口

唯一例外只能是技术事故级应急处理，例如：

1. 支付渠道重复回调造成重复扣款
2. 系统异常造成重复建单

这类场景不属于业务退款，而属于运维/财务异常处置，不作为常规产品能力设计。

### 2.6 运费口径

VIP 礼包订单包邮。

明确规则：

1. `VIP_PACKAGE` 订单运费固定为 `0`
2. `VIP_PACKAGE` 不参与普通运费模板或满减包邮门槛计算
3. `preview`、`checkout`、支付回调建单三处都必须保持一致

## 3. 术语定义

### 3.1 业务对象

- `VIP 礼包`：用户购买 VIP 资格的业务商品
- `VIP 赠品方案`：某个 VIP 礼包下可选的一种赠品组合
- `VIP 礼包订单`：用于开通 VIP 的特殊订单
- `普通商品订单`：平台日常商品订单
- `奖励商品`：由管理后台维护、仅用于 VIP 赠品与抽奖奖品投放的独立商品池

### 3.1.1 奖励商品域说明

`奖励商品` 是独立商品域。

明确规则：

1. 不参与普通商城首页、分类页、商品详情页陈列
2. 不作为普通用户直接购买商品
3. 仅服务于：
   - VIP 赠品
   - 抽奖奖品
4. 由管理后台统一维护

### 3.2 建议的订单业务类型

当前代码里主表 `Order` 没有类型字段，只有：

1. `VipPurchase`
2. `VipEligibleOrder`
3. `NormalEligibleOrder`

这能区分部分语义，但不足以在订单主链路上快速识别“这是 VIP 礼包订单”。

建议新增明确业务类型字段：

```ts
enum OrderBizType {
  NORMAL_GOODS
  VIP_PACKAGE
  // 后续可扩展 GROUP_BUY / CAMPAIGN / PRESALE / EXCHANGE 等
}
```

同样建议给 `CheckoutSession` 增加同类字段：

```ts
enum CheckoutBizType {
  NORMAL_GOODS
  VIP_PACKAGE
  // 后续可扩展 GROUP_BUY / CAMPAIGN / PRESALE / EXCHANGE 等
}
```

这样支付回调、订单创建、确认收货、退款回滚时，都能直接按业务类型分流。

### 3.3 默认值兼容说明

如果为 `Order` 与 `CheckoutSession` 增加 `bizType @default(NORMAL_GOODS)`，需要同时确认：

1. migration SQL 会为历史数据安全落默认值
2. 旧代码不会把 `bizType` 当作可空字段处理
3. 历史订单默认视为 `NORMAL_GOODS` 与现有逻辑兼容

### 3.4 VIP 礼包单的 Order 结构

采用最简方案：

1. `Order.bizType = VIP_PACKAGE`
2. `OrderItem` 只保留 1 条
3. 这 1 条就是用户选中的赠品 SKU
4. `quantity = 1`
5. `unitPrice = VIP 统一价格`
6. `shippingFee = 0`

不采用“VIP 虚拟商品 + 赠品商品”双条目方案。

原因：

1. `VIP 资格` 语义已经由 `Order.bizType + VipPurchase` 承担
2. 订单结构更简单
3. 发货仍然可以复用现有实物订单链路

## 4. 整体流程

## 4.1 用户主流程

### A. 用户入口

1. 用户进入“我的”页
2. 点击 VIP 卡片或 VIP 按钮
3. 打开 VIP 权益弹窗
4. 点击弹窗底部“购买 VIP 礼包”
5. 进入“VIP 赠品选择页”

### B. 赠品选择

1. 页面展示多个 VIP 赠品方案
2. 每个方案价格均为 VIP 礼包价格
3. 用户只能选择一个方案
4. 点击方案卡片后进入结账页

### C. 结账前登录判断

1. 若用户已登录：
   直接进入正常结账流程
2. 若用户未登录：
   先弹出“手机号 + 验证码”确认

### D. 未登录用户认证流程

1. 用户输入手机号
2. 点击发送验证码
3. 输入验证码
4. 后端校验手机号：
   - 已存在账号：登录
   - 不存在账号：自动创建账号并登录
5. 登录成功后返回当前结账页
6. 之前已选择的 VIP 赠品方案必须保留

### E. 地址与下单

1. 用户填写或选择收货地址
2. 提交订单
3. 创建 `CheckoutSession`
4. 发起支付

### F. 支付成功

1. 支付回调成功
2. 创建 VIP 礼包订单
3. 生成 `VipPurchase`
4. 开通 VIP
5. 绑定 VIP 树节点
6. 触发推荐奖励
7. 创建赠品履约信息

### G. 后续发货

1. 赠品进入正常发货链路
2. 商家或平台发出赠品
3. 用户确认收货
4. 该订单不触发普通消费分润计数

## 5. 页面设计方案

## 5.1 页面一：VIP 权益弹窗

### 目标

承接用户从“我的”页进入 VIP 购买主流程的第一步。

### 当前基础

当前“我的”页已经存在 VIP 权益弹窗和“购买VIP礼包”按钮。

### 调整建议

保留现有视觉方向，但优化为更明确的销售弹窗。

### 信息结构

1. 顶部标题
   - `VIP 会员权益`
2. 核心卖点区
   - 专属分润
   - 邀请收益
   - 专属礼包
   - 优先客服
3. 价格区
   - `限时开通价 ¥399`
4. 行为按钮
   - 主按钮：`购买 VIP 礼包`
   - 次说明：`开通后可任选一份赠品`

### 交互规则

1. 不要求用户在这里先登录
2. 点击主按钮统一进入赠品选择页
3. 已登录和未登录都可进入赠品选择页
4. 登录判断延后到结账阶段

这样做的原因：

1. 用户先看到赠品，更容易转化
2. 先登录再选礼包会增加流失

## 5.2 页面二：VIP 赠品选择页（VIP 专属空间）

### 页面定位

这是新增页面，是整个方案里最关键的新增页面。

**设计理念**：不是”商品列表页”，而是”VIP 专属空间”。用户走进的是一个深色、金色微光、粒子飘落的会员空间，在这里选择属于自己的专属礼遇。视觉上必须和普通商品页面有本质区别。

### 建议路由

`app/vip/gifts.tsx`

原因：

1. 它不是”我的资料页”的一部分
2. 它是独立的购买流程页
3. 后续更容易扩展活动入口

### 页面目标

让用户感受到 VIP 的尊享感，明确自己买的是 VIP 资格 + 专属礼遇，而不是普通商品。

### 视觉语言

#### 整体氛围

1. **背景**：深墨绿（#0A1F1A）→ 纯黑（#0D0D0D）纵向渐变，和 App 已有的 AI 视觉风格统一但更高级
2. **粒子效果**：金色微粒缓慢飘落，复用已有的 `react-native-reanimated` 粒子动画能力，调整为金色调（#C9A96E → #E8D5A3）
3. **文字色**：标题用金色渐变（#C9A96E → #E8D5A3），正文用暖白（#F5F0E8）
4. **卡片**：深色半透明背景（rgba(255,255,255,0.06)）+ 金色微光边框（1px, rgba(201,169,110,0.3)），选中时边框亮度提升至 0.8 + 外发光

### 各区域详细设计

#### 1. 导航栏

- 完全透明，和背景融为一体
- 左侧返回箭头用暖白色
- 无标题文字（标题在页面内部）

#### 2. 标题区

- 主标题：`✦ VIP 会员专属空间 ✦`，金色渐变，字号 24px，居中
- 装饰符号 ✦ 带微弱脉冲动画（opacity 0.6 ↔ 1.0，周期 3 秒）
- 副标题：`所有礼遇，仅为 VIP 准备`，暖白，字号 14px

#### 3. 推荐人提示条（条件显示）

- 仅当用户无推荐人时显示
- 半透明背景（rgba(201,169,110,0.08)）+ 左侧 3px 金色竖线
- 文案：`扫描好友邀请码，绑定专属推荐人`
- 点击后唤起扫码

#### 4. 赠品卡片轮播区（核心）

**布局**：横向滑动，一次显示一张卡片居中，两侧露出邻卡边缘。

**卡片规格**：
- 宽度：屏幕宽度 × 0.82
- 圆角：16px
- 背景：卡片背景色
- 边框：1px 金色微光边框

**卡片内部结构**：
1. 上部：赠品实拍图（高度占卡片 55%，圆角上半部分）
2. 下部信息区：
   - 赠品名称（金色，18px，单行）
   - 赠品描述（暖白，14px，最多两行）
   - 市场参考价（划线，次级灰，14px）
   - 标签（金色边框小胶囊，12px）

**卡片交互**：
- 左右滑动切换，snap 到卡片中心
- 当前卡片：scale 1.0，opacity 1.0，金色边框
- 非当前卡片：scale 0.85，opacity 0.5，边框隐藏
- 选中动画：点击当前卡片时，边框亮度提升 + 外发光扩散（200ms）

**指示器**：
- 金色圆点，当前点放大
- 显示 `1 / 3` 文字（暖白，12px）

#### 5. VIP 权益横排

- 6 个权益图标横向排列，等间距
- 每个：金色线性图标（24px）+ 暖白文字（10px）
- 权益项：专属分润 / 邀请收益 / 专属礼包 / 优先客服 / 包邮特权 / 消费奖励
- 作用：持续提醒用户 ¥399 买的不仅是赠品，还有 VIP 全部权益

#### 6. 底部固定栏

- 深色毛玻璃背景（和 App 已有的底栏风格一致）
- 左侧：`开通 VIP` + 价格 `¥399`（金色大字）
- 右侧：`立即开通` 按钮（金色渐变背景 #C9A96E → #E8D5A3，深色文字）
- 按钮下方小字：`包邮 · 支付即开通 VIP`（次级灰，12px）

## 6. 关键业务逻辑

## 6.1 登录状态逻辑

### 已登录用户

1. 进入赠品页后可直接选择礼包
2. 去结账时无需验证码
3. 直接走地址 + 下单 + 支付流程

### 未登录用户

1. 进入赠品页后可正常浏览和选择
2. 去结账后，提交前必须验证手机号
3. 验证码通过后才允许创建地址和下单

---

## 6.2 自动登录 / 自动注册逻辑

### 后端判断规则

1. 收到手机号 + 验证码
2. 如果手机号已存在：
   返回登录 Session
3. 如果手机号不存在：
   自动创建：
   - User
   - UserProfile
   - AuthIdentity
   - MemberProfile
   然后返回登录 Session

### 前端要求

1. 登录成功后立即持久化 token
2. 不丢失当前已选择的 VIP 赠品方案
3. 继续停留在当前结账页

实现要求补充：

1. `VipPackageSelection` 不能只放内存态
2. 需要放入持久化 store
3. 底层应使用 AsyncStorage 持久化

原因：

1. 验证码登录成功后会触发认证态切换
2. React Query 可能立即刷新用户数据
3. 如果只放内存 store，礼包选择有丢失风险

---

## 6.3 地址逻辑

### 规则

1. 地址必须归属于当前登录用户
2. 未登录状态不能直接创建正式地址
3. 收货手机号可以与账号手机号不同

### 下单顺序

1. 未登录用户完成验证码认证
2. 创建地址
3. 生成 `CheckoutSession`
4. 支付

---

## 6.4 VIP 开通逻辑

### 触发时机

只能在支付成功时触发。

### 幂等要求

支付回调可能重复推送，VIP 开通链路必须具备幂等保护。

建议：

1. 使用 `checkoutSessionId` 或 `orderId` 作为幂等键
2. 整个“支付成功 -> 开通 VIP -> 写 VipPurchase -> 发奖励 -> 履约”流程放在单个 `Serializable` 事务中
3. 同一幂等键重复回调时直接返回已处理结果

### 开通内容

1. 创建 `VipPurchase`
2. 更新 `MemberProfile.tier = VIP`
3. 写入 `vipPurchasedAt`
4. 创建或初始化 `VipProgress`
5. 分配 `VipTreeNode`
6. 如有推荐人，发放 VIP 推荐奖励
7. 冻结普通树进度

### 并发防护

同一用户可能在多个设备上重复发起 VIP 购买。

必须至少做两层保护：

1. 创建 VIP 结算会话时，校验用户当前不是 VIP，且不存在进行中的 VIP 购买会话
2. 支付回调时，事务内再次校验该用户尚未完成 VIP 开通

建议增加数据库约束：

1. `VipPurchase.userId` 唯一约束
2. 或等价的“同一用户仅允许一条成功 VIP 购买记录”的唯一方案

### 绝对不能做的事

1. 不能在下单时先开通
2. 不能在待支付状态先开通
3. 不能把该订单当作 VIP 有效消费首单

---

## 6.5 分润隔离逻辑

### 规则

`VIP_PACKAGE` 订单：

1. 不参与普通树
2. 不参与 VIP 上溯
3. 不产生 `VipEligibleOrder`
4. 不产生 `NormalEligibleOrder`
5. 不增加任何消费计数
6. 不进入普通运费规则计算

### 允许发生的奖励

如果业务需要，允许仅发生：

1. VIP 开通推荐奖励

该奖励必须归类为：

- `购买 VIP 成功触发的邀请奖励`

而不是：

- `普通订单消费分润`

---

## 6.6 发货与履约逻辑

### 原则

赠品不能只停留在展示层，必须进入正式履约。

### 处理方式

建议两种方案二选一。

#### 方案 A：VIP 礼包单本身承担履约

即订单里直接带赠品 SKU，发货、物流、收货都走同一订单。

优点：

1. 复用现有订单发货链路
2. 不需要新建赠品履约单体系

缺点：

1. 需要在主订单层明确区分 `VIP_PACKAGE`
2. 需要在分润入口显式跳过

#### 方案 B：支付成功后拆出赠品履约单

即：

1. 主业务单是 VIP 资格单
2. 支付成功后再生成一个赠品履约订单

优点：

1. 概念更干净
2. VIP 资格与赠品履约强解耦

缺点：

1. 要新增一套“履约单”或“派发单”逻辑
2. 开发成本更高

### 推荐

优先采用方案 A。

原因：

1. 当前项目已有成熟 `Order`、`Shipment`、`Refund` 链路
2. 只要在业务类型和分润入口做好隔离即可
3. 改造成本最低

补充说明：

虽然复用现有订单链路，但 VIP 礼包订单本身不参与退款链路。

### 发货责任方

VIP 赠品来自奖励商品（`Company.isPlatform=true`），但履约发货不由管理后台直接执行。

不新建独立的"平台订单"页面，而是在现有管理后台订单列表页增强：

1. 新增 `VIP礼包` 订单类型筛选（`bizType=VIP_PACKAGE`）
2. 订单详情页识别 VIP 礼包订单后：
   - 隐藏退款按钮
   - 显示 `VIP 开通礼包 · 不支持退款` 标签
3. 发货操作不走 admin 直接发货
4. 发货仍走现有 seller / 平台公司发货链路
5. 平台公司作为奖励商品的归属主体，使用现有 seller 端订单发货能力完成履约

### Seller 端展示要求

seller / 平台公司订单页也必须识别 VIP 礼包订单。

建议展示：

1. `VIP礼包` 标签
2. `包邮`
3. `不支持退款`
4. 与普通订单不同的说明文案
5. 增加快捷筛选：
   - `VIP礼包订单`
   - `抽奖奖品订单`
   - `普通订单`

## 7.3 建议新增模型：VIPGiftOption

用于定义“VIP 赠品选择页”里可选的方案。

```prisma
enum VipGiftOptionStatus {
  ACTIVE
  INACTIVE
}

model VipGiftOption {
  id          String   @id @default(cuid())
  title       String
  subtitle    String?
  coverUrl    String?
  skuId       String
  sku         ProductSKU @relation(fields: [skuId], references: [id])
  marketPrice Float?
  badge       String?
  sortOrder   Int      @default(0)
  status      VipGiftOptionStatus @default(ACTIVE)
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}
```

用途：

1. 控制赠品页展示内容
2. 控制上架/下架
3. 控制排序
4. 控制每个方案绑定哪个 SKU

赠品来源规则：

1. `VipGiftOption` 绑定的 SKU 必须来自管理后台”奖励商品”
2. 奖励商品可以同时被用于：
   - VIP 赠品
   - 抽奖奖品
3. VIP 与抽奖共用同一奖励商品时，使用同一 SKU 库存口径
4. VIP 赠品资格与抽奖奖品资格是两套独立配置，不互相替代

原因：

1. 奖励商品本身就是运营池资源
2. VIP 赠品与抽奖奖品都属于运营资源投放
3. 共享商品池更符合当前业务组织方式

### 7.3.1 VIP 统一价格配置

由于第一版所有 VIP 赠品方案价格相同，不在 `VipGiftOption` 上单独存价格。

VIP 统一价格使用现有的 `VIP_PRICE` 系统配置项，该配置已在管理后台的 `VIP 系统设置 > VIP 基础设置 > VIP 礼包价格` 中管理。

规则：

1. 前台展示价格统一从 `VIP_PRICE` 系统配置读取
2. 结账金额统一从 `VIP_PRICE` 系统配置读取
3. `VipGiftOption` 只负责”选什么赠品”，不负责”卖多少钱”
4. 管理后台”购买VIP赠品”页面顶部说明卡展示当前 VIP 统一价格，不可在此页面修改

## 7.4 建议扩展 VipPurchase

当前 `VipPurchase` 只记录：

1. userId
2. orderId
3. amount
4. status

建议扩展：

```prisma
model VipPurchase {
  ...
  giftOptionId String?
  giftSkuId    String?
  giftSnapshot Json?
  source       String? // APP_VIP_PACKAGE / ADMIN_GRANT / ACTIVITY
  activationStatus String? // PENDING / ACTIVATING / SUCCESS / FAILED / RETRYING
  activationError  String?
}
```

这样可以清楚知道：

1. 用户当时买的是哪种礼包方案
2. VIP 激活链路当前处于什么状态

### 7.4.1 激活状态机建议

建议固定状态机：

1. `PENDING`
   支付已确认，待进入激活流程
2. `ACTIVATING`
   正在执行 VIP 开通事务
3. `SUCCESS`
   开通成功
4. `FAILED`
   激活失败，等待人工处理或重试
5. `RETRYING`
   自动补偿重试中

这样后台排查“已支付未开通”问题时更清楚。

## 11. 推荐人机制

### 11.1 核心规则：最后推荐人生效 + 购买锁定

推荐关系在 VIP 支付成功前可被覆盖，支付成功后永久锁定。

```
B 扫了 A 的码 → 注册 → referrer = A
    ↓
B 没买 VIP，过了 1 个月
    ↓
B 扫了 C 的码 → referrer 覆盖为 C
    ↓
B 购买 VIP 支付成功 → referrer 永久锁定为 C，C 拿奖励
```

规则：

1. 注册时绑定第一个推荐人，但未购买 VIP 前推荐人可被覆盖
2. 每次扫码或触发新邀请码，覆盖当前推荐人
3. 支付成功瞬间锁定，之后不可更改
4. App 内”我的”页可以看到当前推荐人是谁（购买前）
5. 推荐关系暂不设过期天数（后续可按需增加，管理后台可配置）

### 11.2 VIP 推荐奖励

1. 推荐奖励为直接发钱给推荐人
2. 奖励金额在管理后台「VIP 系统设置」页面配置
3. 仅在被推荐人 VIP 支付成功时触发
4. 与普通消费分润完全隔离

### 11.3 邀请码传递机制：服务端设备指纹匹配

#### 设计目标

用户全程零输入：扫码 → 下载 → 打开 App → 推荐关系自动建立。

#### 完整流程

```
VIP 用户 A → App 内点”邀请好友” → 生成二维码（内嵌 h5.nongmai.app/invite/{referralCode}）

新用户 B 扫码
    ↓
微信/浏览器打开 H5 落地页
    ↓
落地页做两件事：
  ① 展示 App 介绍，引导下载
  ② 静默记录设备指纹 → POST /api/v1/referral/track
     { referralCode, ip, screenWidth, screenHeight, devicePixelRatio, language, timezone, createdAt }
     存入 Redis/数据库，TTL = 48 小时
    ↓
B 点”下载 App” → 应用商店 → 安装
    ↓
B 首次打开 App
    ↓
App 静默收集同样的设备指纹 → POST /api/v1/referral/match
    { ip, screenWidth, screenHeight, devicePixelRatio, language, timezone }
    ↓
后端用指纹哈希 + 48 小时时间窗匹配
    ↓
匹配成功 → 返回 referralCode → App 自动绑定推荐关系
匹配成功后删除该记录（防止重复匹配）
    ↓
B 全程无感知
```

#### 指纹匹配维度

| 维度 | 浏览器可获取 | App 可获取 | 稳定性 |
|------|------------|-----------|--------|
| IP 地址 | 是（后端取） | 是（后端取） | 同 WiFi 下一致 |
| 屏幕宽度 | screen.width | Dimensions.get('screen').width | 设备固定 |
| 屏幕高度 | screen.height | Dimensions.get('screen').height | 设备固定 |
| 设备像素比 | devicePixelRatio | PixelRatio.get() | 设备固定 |
| 系统语言 | navigator.language | 系统语言 API | 通常不变 |
| 时区 | Intl.DateTimeFormat().resolvedOptions().timeZone | 同上 | 通常不变 |

预估匹配准确率：80-85%。

主要失败场景：扫码时和安装后不在同一网络环境（如外出扫码、回家安装）。

#### 匹配失败兜底

不做”输入邀请码”的独立入口。

在 VIP 赠品选择页顶部做轻提示：如果当前用户没有推荐人，显示”扫描好友的邀请码可绑定专属推荐人”。此时用户已装 App，扫码直接唤起 App 并带上 referralCode，100% 成功。

```
VIP 赠品选择页顶部（仅无推荐人时显示）：
┌──────────────────────────────────────┐
│  扫描好友邀请码，绑定专属推荐人        │
└──────────────────────────────────────┘
```

已安装 App 的用户扫码 → Universal Link / Deep Link 直接唤起 App → URL 携带 referralCode → 直接绑定，不需要再走指纹匹配。

## 12. 流程补丁：已确认的逻辑漏洞修复

### 补丁 A：已是 VIP 但未登录的用户

**问题**：未登录用户走完选礼包、填地址、输验证码，登录后才发现已经是 VIP，全部流程白费。

**修复**：验证码登录成功后、创建 CheckoutSession 前，立即校验 VIP 状态。已是 VIP 则弹出友好提示并引导到 VIP 权益页，不在结账页报错。

### 补丁 B：CheckoutSession 无过期机制

**问题**：创建会话时校验”不存在进行中的 VIP 购买会话”。若用户放弃支付，会话永远挂着，永久阻塞该用户再次购买 VIP。

**修复**：VIP 类型的 CheckoutSession 必须有过期时间，过期时间在管理后台可配置（建议默认 30 分钟）。过期后自动释放，不再阻塞新会话创建。创建新会话时，先清理该用户所有已过期的 VIP 会话。

### 补丁 C：价格快照

**问题**：VIP 统一价格从系统配置读取，但未指定锁定时机。管理员在用户结账过程中改价会导致金额不一致。

**修复**：创建 CheckoutSession 时将当前 `VIP_PRICE` 快照写入 `bizMeta.snapshotPrice`。后续支付金额以快照为准，不再重新读配置。支付回调建单时金额也使用快照值。

### 补丁 D：库存软预留

**问题**：用户选品到支付之间有时间窗口，期间抽奖或其他 VIP 订单可能消耗完库存。

**修复**：创建 CheckoutSession 时对赠品 SKU 做软预留（扣减可用库存但不扣真实库存），会话过期时释放预留。如果项目已确认”超卖容忍”策略，也可接受不预留，但支付成功页需告知用户可能延迟发货。

## 14. 风险点

## 14.1 业务风险

### 风险 A：VIP 礼包订单误进入分润链路

后果：

1. 被当成首单
2. 错误递增消费次数
3. 错误发放上溯奖励

优先级：最高

### 风险 B：支付前提前开通 VIP

后果：

1. 用户未付款已获得 VIP
2. 树节点和奖励记录污染

优先级：最高

### 风险 C：赠品库存与礼包配置不一致

后果：

1. 前台可选，支付后无货
2. 履约失败

优先级：高

补充防护：

1. VIP 与抽奖共用奖励商品时，必须明确共享库存口径
2. 下单与发奖前都要做库存有效性校验

### 风险 D：未登录认证后丢失礼包选择状态

后果：

1. 用户完成验证码后回不到当前礼包
2. 造成流失

优先级：高

补充防护：

1. 使用 AsyncStorage 持久化 `VipPackageSelection`

### 风险 E：支付回调重复导致重复开通 VIP

后果：

1. 重复创建 `VipPurchase`
2. 重复发放推荐奖励
3. 树节点和账户状态污染

优先级：最高

### 风险 F：同一用户并发重复购买

后果：

1. 多个设备同时创建会话
2. 支付后重复开通
3. 多笔 VIP 订单状态冲突

优先级：最高

### 风险 G：奖励商品/SKU 被下架或删除后仍被 VIP 方案引用

后果：

1. 前台赠品页展示异常
2. 结账时方案失效
3. 运营误删后产生线上事故

优先级：高

补充防护：

1. 若某个 SKU 被 `ACTIVE` 的 `VipGiftOption` 引用，则奖励商品页删除/下架该 SKU 时必须拦截或强提示
2. 若某个奖励商品被用于抽奖奖品或 VIP 赠品，应显示”被运营活动引用”状态

## 14.2 推荐机制安全风险

### 风险 H：指纹碰撞导致错误绑定推荐人

**描述**：同一 WiFi 下两个人使用同型号手机，先后扫了不同推荐码并安装，指纹相同导致后安装者匹配到前者的推荐码。

**后果**：用户被绑定到错误的推荐人，推荐人获得不应得的奖励。

**优先级**：中

**防护**：
1. 匹配时优先取最近一条记录（时间戳最接近的）
2. 匹配成功后立即删除记录，缩小碰撞窗口
3. 指纹维度尽可能多（IP + 屏幕 + 像素比 + 语言 + 时区 = 5 维），降低碰撞概率
4. 如果同一指纹 48 小时内存在多条不同 referralCode 的记录，全部作废不匹配，让用户走兜底流程（App 内重新扫码）

### 风险 I：恶意刷推荐绑定

**描述**：攻击者伪造大量 `/referral/track` 请求，写入虚假指纹记录，试图让随机新用户匹配到自己的推荐码。

**后果**：攻击者非法获取大量推荐关系和推荐奖励。

**优先级**：高

**防护**：
1. `/referral/track` 接口做频率限制（同一 IP 每分钟最多 5 次，同一 referralCode 每小时最多 50 次）
2. 落地页加入简单的人机验证（如滑块或隐形 reCAPTCHA），防止脚本批量调用
3. 后端校验 referralCode 必须真实存在且对应用户确实是 VIP
4. 同一指纹哈希如果已存在记录，覆盖而非新增（防止同一设备被写入多条）
5. 推荐奖励发放前做异常检测：单个推荐人短时间内大量新用户绑定应触发风控告警

### 风险 J：指纹伪造劫持推荐关系

**描述**：攻击者通过逆向 App 或抓包，获取受害者的设备指纹，抢先调用 `/referral/track` 将自己的 referralCode 绑定到该指纹上。

**后果**：受害者安装 App 后被匹配到攻击者的推荐码。

**优先级**：中

**防护**：
1. `/referral/track` 必须来自 H5 落地页，校验 Referer / Origin 头
2. 落地页请求时附带一个短期有效的签名 token（后端生成，嵌入页面），防止接口被直接调用
3. 指纹匹配只是初始绑定，用户购买 VIP 前可随时通过 App 内扫码覆盖推荐人，最终以支付锁定为准

### 风险 K：推荐人自刷（自己推荐自己）

**描述**：VIP 用户用自己的邀请码生成链接，用另一个手机号注册新账号并购买 VIP，给自己刷推荐奖励。

**后果**：推荐奖励被薅。

**优先级**：中

**防护**：
1. 同一设备指纹不允许既是推荐人又是被推荐人（设备级排重）
2. 推荐人和被推荐人不能使用同一收货地址（地址级排重，在支付成功发奖前校验）
3. 单个推荐人的推荐奖励设每日/每月上限（管理后台可配置）
4. 人工审核能力：管理后台可查看推荐关系明细，支持冻结异常推荐人的奖励

---

## 15. 推荐实施顺序

### 第一步

先定死业务口径：

1. VIP 礼包是否算有效消费：不算
2. VIP 何时生效：支付成功后
3. 赠品如何履约：复用订单链路

### 第二步

补后端业务类型：

1. `Order.bizType`
2. `CheckoutSession.bizType`

### 第三步

新增礼包方案配置与前台赠品页。

这里建议并行推进：

1. 后端先做 `bizType + VipGiftOption + 管理端接口`
2. 前端可先用 mock 数据开发 VIP 赠品选择页
3. 管理后台同时新增 `购买VIP赠品` 页面
4. 抽奖奖品与 VIP 赠品共用奖励商品选择器时，接口可共用

### 第四步

推荐人机制与 H5 落地页：

1. 后端实现 `/referral/track` 和 `/referral/match` 接口
2. 部署 H5 落地页（纯静态，OSS/CDN）
3. App 端实现首次启动指纹匹配 + Deep Link 唤起处理
4. VIP 赠品页实现”无推荐人时引导扫码”提示

### 第五步

打通结账与支付成功开通逻辑。

含：VIP 状态校验、CheckoutSession 过期机制、价格快照、库存软预留、推荐人锁定与奖励发放。

### 第六步

最后补不可退款口径限制、订单展示和异常单查询能力。

## 16. 最终建议

这个方案值得做，原因如下：

1. 它比直接卖”抽象 VIP”更容易被理解
2. 它能把 VIP 购买动作包装成更具体的消费体验
3. 它和你现有”结账页 + 地址 + 订单 + 发货”体系可以自然衔接
4. 推荐人机制让 VIP 用户主动拉新，形成裂变闭环

但要注意：

1. 必须把 VIP 礼包单与普通消费单严格分开
2. 必须把”开通 VIP”的时机压到支付成功
3. 必须把”赠品商品”当作正式履约对象处理
4. 必须在分润入口显式跳过 VIP 礼包单
5. 推荐人指纹匹配必须做好防刷和碰撞处理
6. 推荐奖励发放必须有风控兜底

如果以上六点不先做好，后面越做越难收拾。
