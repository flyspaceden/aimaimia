## 规则 1：退货窗口起算点

- 从订单状态变为 `DELIVERED`（物流签收）的时间开始计算
- 系统记录 `deliveredAt` 时间戳作为起算基准
- 7 天 = 168 小时，精确到时分秒（例如 3月1日 14:30 签收 → 3月8日 14:30 截止）
- 如果订单在 DELIVERED 之前买家主动确认收货（跳到 RECEIVED），仍以 DELIVERED 时间为准
- 如果物流异常没有 DELIVERED 记录，以买家主动确认收货时间为准

## 规则 2：不可退商品的判定方式（两级机制）

### 分类级别

- `Category` 模型新增 `returnPolicy` 字段，可选值：`RETURNABLE`（可退）/ `NON_RETURNABLE`（不可退）/ `INHERIT`（继承父分类）
- 顶级分类默认 `RETURNABLE`
- 子分类默认 `INHERIT`，沿树向上查找最近的非 INHERIT 祖先分类的值
- 管理员在分类管理页面设置

### 商品级别

- `Product` 模型新增 `returnPolicy` 字段，可选值：`RETURNABLE` / `NON_RETURNABLE` / `INHERIT`（继承分类设置）
- 默认 `INHERIT`
- 卖家发布商品时可选择，管理员可在审核时修改

### 最终判定逻辑

1. 先看商品自身的 `returnPolicy`
2. 如果是 `INHERIT`，向上查分类的 `returnPolicy`
3. 分类如果也是 `INHERIT`，继续向上查父分类，直到找到明确值
4. 最终兜底为 `RETURNABLE`

### 不可退声明展示

- 商品详情页：小字提示（可退商品显示「支持7天无理由退换」，不可退商品显示「签收后24小时内如有质量问题可申请售后」）
- 结账页：用户首次结账时弹出统一退换货规则协议（含普通商品 + 生鲜 + VIP/奖品条款），确认一次后永久记录不再弹出
- 订单详情页：小字提示对应退货政策

## 规则 3：不可退商品的质量问题仍可退

- 标记为"不可退"的商品（如生鲜海鲜），仅豁免"七天无理由退货"
- 如果商品存在质量问题（变质、损坏、发错货等），买家仍可发起"质量问题退货退款"或"质量问题换货"
- 需要提供照片/视频凭证
- 生鲜商品质量问题的申报时限为签收后 24 小时（可配置，规则 10）
- 审核流程不变：卖家审核 → 平台仲裁

## 规则 4：审核模式 — 卖家审核 + 平台仲裁

### 第一级：卖家审核

- 买家提交售后申请后，首先由卖家审核
- 卖家可以：开始审核（REQUESTED → UNDER_REVIEW）→ 同意（→ APPROVED）或驳回（→ REJECTED）
- 卖家审核需要查看买家提供的照片凭证、原因描述
- 仅 OWNER / MANAGER 角色可审核
- 卖家只能处理自己公司商品的售后申请

### 第二级：平台仲裁

- 触发条件：卖家驳回后，买家不满意可升级到平台仲裁
- 管理员可查看双方举证，做出最终裁决（同意/驳回）
- 管理员仲裁结果为最终结果
- 管理员可在 REQUESTED / UNDER_REVIEW / REJECTED 状态下介入仲裁

### 特殊情况

换货后再退货（规则 22）直接走平台仲裁，跳过卖家审核。

## 规则 5：价格阈值决定是否需要寄回

### 配置项

- 后台管理员可配置全局参数 `RETURN_NO_SHIP_THRESHOLD`（默认值建议 50 元）
- 存储在系统配置表中，可随时调整

### 适用场景：质量问题退货退款 + 质量问题换货

- 退货/换货商品的金额（单价 × 数量） ≤ 阈值：拍照审核通过后直接退款/补发，不用寄回
- 退货/换货商品的金额 > 阈值：拍照 + 必须寄回商品

### 不适用场景

七天无理由退货（规则 8 强制寄回，不受阈值影响）

### 寄回运费承担

- 质量问题需要寄回：平台承担运费（规则 9）
- 七天无理由需要寄回：买家自付运费（规则 8）

## 规则 6：退款只退商品价格，不退运费

- 退款金额 = 退货商品的实付金额（扣除该商品分摊的红包优惠后）
- 原订单的运费不退（无论退多少商品）
- **唯一例外**：如果订单所有商品全部退货且为质量问题，运费也应退还；七天无理由全退不退运费
- 计算公式：`退款金额 = 商品单价 × 退货数量 - 该商品分摊的红包优惠`

---

## 规则 7：平台红包按比例分摊

### 分摊算法

- 订单使用了平台红包（CouponInstance）抵扣了 D 元
- 订单总商品金额 = G 元
- 退货商品金额 = R 元
- 该商品分摊的红包优惠 = D × (R / G)
- 实际退款金额 = R - D × (R / G)

### 举例

- 订单商品：A=100, B=200, C=300，总额 600
- 红包抵扣 60 元，实付 540
- 退商品 A（100 元）：分摊红包 = 60 × (100/600) = 10 元，退款 = 100 - 10 = 90 元

### 红包不退回

- 退货后已使用的红包不退回给用户
- 红包的 CouponInstance 状态保持 USED，不恢复为 AVAILABLE

## 规则 8：七天无理由退货规则

### 适用条件

- 商品的 returnPolicy 最终判定为 `RETURNABLE`（非生鲜、非不可退商品）
- 在签收后 7 天内（从 DELIVERED 起算）
- 商品应当完好（未拆封使用、配件齐全、不影响二次销售）

### 流程要求

- 必须拍照（证明商品完好状态）
- 一律需要寄回商品（不受价格阈值影响）
- 退回运费由买家自付
- 买家填写退回物流单号

### 不适用

- 生鲜/鲜活/易腐商品（returnPolicy = NON_RETURNABLE）
- 已拆封食品、已使用商品
- VIP_PACKAGE 订单
- 抽奖奖品（isPrize = true，包括折扣购和满额赠品）
- 超过 7 天窗口期

---

## 规则 9：质量问题退货运费平台承担

- 质量问题退货/换货中，如果需要寄回（商品金额 > 阈值），运费由平台承担
- 实现方式：买家寄快递时选择到付，平台承担到付运费
- 平台全额承担退回运费，不设上限

---

## 规则 10：生鲜商品特殊规则

### 不适用七天无理由退货

- 生鲜/鲜活/冷冻/冰鲜商品属于法定豁免类别
- 通过分类或商品级 returnPolicy = NON_RETURNABLE 标记
- 商品详情页和下单页明确告知"不支持7天无理由退货"

### 质量问题可退（唯一售后途径）

- 仅在签收后 24 小时内（可配置，参数 `FRESH_RETURN_HOURS`，默认 24）
- 必须提供清晰凭证：商品全貌照、快递面单照、变质/破损/漏发照片
- 可退情形：变质、异味、严重化冻、包装破损泄漏、重量严重短缺、发错货、漏发
- 不可退情形：非质量问题的个人原因、签收超时导致变质、无有效凭证、已拆封加工食用

### 处理方式

- 按阈值决定是否需要寄回（与普通商品规则一致，防止假照片骗退）
- 核实后可补发、换货或退款

---

## 规则 11：非生鲜商品质量问题申报时限

- 后台可配置参数 `NORMAL_RETURN_DAYS`，默认 7 天
- 从 DELIVERED（物流签收）时间起算
- 在此期间内买家可以发起"质量问题退货退款"或"质量问题换货"
- 超过此期限后不再受理质量问题售后
- 该时限不能超过 7 天（所有售后窗口 ≤ 7 天，与退货保护期对齐）

---

## 规则 12：统一售后入口 — 三种售后类型

### 买家 App 售后入口

- 订单详情页一个"申请售后"按钮
- 进入后选择要售后的商品（每次选一个商品，多商品需分别提交）
- 然后选择售后类型

### 三种售后类型

| 类型 | 标识 | 结果 | 展示条件 |
|------|------|------|---------|
| 七天无理由退货 | `NO_REASON_RETURN` | 退款 | 商品可退 + 在 7 天窗口内 |
| 质量问题退货退款 | `QUALITY_RETURN` | 退款 | 在质量问题时限内（生鲜 24h / 非生鲜可配置） |
| 质量问题换货 | `QUALITY_EXCHANGE` | 补发 | 在质量问题时限内（生鲜 24h / 非生鲜可配置） |

### 动态展示逻辑

- 如果商品是不可退的（生鲜等），隐藏"七天无理由退货"选项
- 如果超过七天窗口，隐藏"七天无理由退货"选项
- 如果超过质量问题时限，隐藏两个质量问题选项
- 如果所有选项都不可用，显示"该商品已超过售后申请期限"

### 原因子类型（质量问题时必选）

`QUALITY_ISSUE` / `WRONG_ITEM` / `DAMAGED` / `NOT_AS_DESCRIBED` / `SIZE_ISSUE` / `EXPIRED` / `OTHER`

## 规则 13：部分退货支持

- 一个订单包含多个商品时，买家可以分别对不同商品提交售后申请（每次申请只选一个商品）
- 每个售后申请关联一个具体的 OrderItem
- 退款金额按选中商品的实付金额计算（扣除红包分摊）
- 部分退货后，订单中未退货的商品正常履约
- 同一个 OrderItem 同一时间只能有一个进行中的售后申请（规则 21）

## 规则 14：退货/换货后奖励归平台

### 触发条件

订单中任何一个商品发生了退货退款或换货（无论什么原因）。

### 处理逻辑

- 该订单的**全部**分润奖励（所有上级用户的奖励）归平台所有
- 不是删除奖励记录，而是将收益方从"上级用户"改为"平台账户"
- 后台仍然正常计算退货后的订单实际金额和对应的分润比例数据
- 这些数据仅用于平台统计报表，不实际发放给上级用户

### 计算逻辑（用于统计）

- 退货后订单实际金额 = 原订单商品金额 - 退货商品金额
- 按退货后的实际金额重新计算分润比例
- 记录在奖励分配记录中，标记为"平台收入"

### 时序

- 所有售后窗口 ≤ 7 天，与退货保护期完全对齐
- 退货只能发生在 7 天保护期内 → 奖励从 RETURN_FROZEN 直接转为平台收入
- 7 天后奖励已安全转为 FROZEN，不存在需要回收的场景

---

## 规则 15：分润奖励两层冻结机制

### 第一层：退货保护期冻结（新增）

| 阶段 | 时间 | 奖励状态 | 用户钱包可见 | 可提现 |
|------|------|---------|------------|--------|
| 订单 RECEIVED | T+0 | 计算并分配，标记为 RETURN_FROZEN | 否 | 否 |
| DELIVERED 起 7 天内 | T+0 ~ T+7 | RETURN_FROZEN | 否 | 否 |
| 7 天到期无退货 | T+7 | 转为 FROZEN（进入第二层） | 是 | 否 |
| 7 天内发生退货 | 随时 | 转为平台收入（规则 14） | — | — |

### 第二层：原有第 x 单冻结解冻（现有机制不变）

- 奖励从 RETURN_FROZEN 转为 FROZEN 后，进入现有的"第 x 单完成"冻结/解冻机制
- 满足条件后 FROZEN → AVAILABLE
- AVAILABLE 后用户可在钱包中提现

### 需要新增的 RewardLedgerStatus 枚举值

- `RETURN_FROZEN` — 退货保护期冻结（用户不可见）

## 规则 16：完整状态机

### 状态定义

| 状态 | 含义 |
|------|------|
| `REQUESTED` | 买家提交售后申请 |
| `UNDER_REVIEW` | 卖家开始审核 |
| `APPROVED` | 卖家/管理员同意 |
| `REJECTED` | 卖家驳回 |
| `PENDING_ARBITRATION` | 平台待仲裁（买家升级/换货后再退直接进入） |
| `RETURN_SHIPPING` | 买家已寄回商品，等待卖家收货 |
| `RECEIVED_BY_SELLER` | 卖家确认收到退回商品 |
| `SELLER_REJECTED_RETURN` | 卖家验收不通过（买家可仲裁或接受关闭） |
| `REFUNDING` | 退款处理中 |
| `REFUNDED` | 退款完成 |
| `REPLACEMENT_SHIPPED` | 换货新商品已发出 |
| `COMPLETED` | 售后完成（换货买家确认收货 / 退款到账） |
| `CLOSED` | 买家接受驳回/验收不通过，主动关闭 |
| `CANCELED` | 买家主动撤销（审核前） |

### 退货退款 — 需要寄回

```
REQUESTED → UNDER_REVIEW → APPROVED → RETURN_SHIPPING
→ RECEIVED_BY_SELLER → REFUNDING → REFUNDED → COMPLETED
                  ↓
        SELLER_REJECTED_RETURN（买家可仲裁）
```

### 退货退款 — 不用寄回（低于阈值 / 生鲜）

```
REQUESTED → UNDER_REVIEW → APPROVED → REFUNDING → REFUNDED → COMPLETED
```

### 换货 — 需要寄回

```
REQUESTED → UNDER_REVIEW → APPROVED → RETURN_SHIPPING
→ RECEIVED_BY_SELLER → REPLACEMENT_SHIPPED → COMPLETED
                  ↓
        SELLER_REJECTED_RETURN（买家可仲裁）
```

### 换货 — 不用寄回

```
REQUESTED → UNDER_REVIEW → APPROVED → REPLACEMENT_SHIPPED → COMPLETED
```

### 驳回 → 仲裁路径

```
REJECTED → 买家升级仲裁 → PENDING_ARBITRATION → 管理员裁决 APPROVED 或 REJECTED
SELLER_REJECTED_RETURN → 买家升级仲裁 → PENDING_ARBITRATION → 管理员裁决
```

### 关闭路径（买家接受驳回/验收不通过）

```
REJECTED → 买家接受 → CLOSED
SELLER_REJECTED_RETURN → 买家接受 → CLOSED
```

### 撤销路径

```
REQUESTED / UNDER_REVIEW → CANCELED（买家主动撤销）
```

## 规则 17：卖家验收不通过处理

### 触发条件

买家寄回商品后，卖家收到商品检查发现不符合退货条件。

### 卖家操作

- 状态从 `RECEIVED_BY_SELLER` → `SELLER_REJECTED_RETURN`
- 卖家必须填写拒绝原因（文字说明）
- 卖家必须上传举证照片（证明商品不符合退货条件）
- 卖家需要将商品寄回给买家（填写退回物流单号）

### 买家选择

- 接受拒绝 → 状态变为 `CLOSED`，售后终止，商品由卖家寄回买家
- 不接受 → 升级仲裁，状态变为 `PENDING_ARBITRATION`
- 平台管理员查看双方举证后做最终裁决（APPROVED 或 REJECTED）

## 规则 18：超时自动处理机制

### 全部超时参数后台可配置

| 场景 | 参数名 | 默认值 | 超时处理 |
|------|--------|--------|---------|
| 卖家不审核 | `SELLER_REVIEW_TIMEOUT_DAYS` | 3 天 | 自动同意（REQUESTED/UNDER_REVIEW → APPROVED） |
| 买家不寄回 | `BUYER_SHIP_TIMEOUT_DAYS` | 7 天 | 自动关闭售后（APPROVED → CANCELED） |
| 卖家不验收 | `SELLER_RECEIVE_TIMEOUT_DAYS` | 7 天 | 自动视为验收通过（RETURN_SHIPPING → RECEIVED_BY_SELLER），继续后续流程 |
| 换货买家不确认 | `BUYER_CONFIRM_TIMEOUT_DAYS` | 7 天 | 自动确认收货（REPLACEMENT_SHIPPED → COMPLETED） |

### 执行方式

- 定时任务（Cron）定期扫描超时的售后申请
- 扫描频率：每小时一次
- 超时处理在 Serializable 事务内执行，防止并发

## 规则 19：买家撤销规则

### 可撤销阶段

- `REQUESTED`（刚提交，卖家还没看）
- `UNDER_REVIEW`（卖家在审核中，但还没做决定）

### 不可撤销阶段

- `APPROVED` 及之后的所有状态（流程已启动，涉及物流和资金操作）

### 撤销后

- 状态变为 `CANCELED`
- 该商品可以重新发起售后申请（如果仍在时限内）
- 不影响订单状态和奖励状态

---

## 规则 20：订单状态与售后的关系

### 部分退货

- 订单状态保持 `RECEIVED` 不变
- 订单关联的售后申请中可以查看哪些商品退了
- 前端通过售后记录显示"部分退款"标记
- 订单的 `totalAmount` 不变，通过售后记录计算实际最终金额

### 全部退货（订单所有商品都完成退款）

- 订单状态变为 `REFUNDED`
- 质量问题全退：同时退还运费（规则 6 的例外情况）
- 七天无理由全退：不退运费

### 换货完成

- 订单状态保持 `RECEIVED`
- 售后申请状态为 `COMPLETED`

### 售后进行中的订单展示

- 前端订单列表中标记"售后中"标识
- 订单详情页显示关联的售后申请进度

---

## 规则 21：多售后并行规则

### 同一订单、不同商品

- 可以同时存在多个售后申请
- 例如：商品 A 申请退货 + 商品 B 申请换货，互不影响

### 同一个 OrderItem

- 同一时间只能有一个进行中的售后申请
- "进行中" = 状态为 REQUESTED / UNDER_REVIEW / APPROVED / RETURN_SHIPPING / RECEIVED_BY_SELLER / SELLER_REJECTED_RETURN / REFUNDING / REPLACEMENT_SHIPPED
- 如果上一个售后申请已终结（COMPLETED / REFUNDED / CANCELED / REJECTED 且未升级仲裁），可以重新发起

### 同一订单的奖励影响

- 只要订单下有任何一个售后申请最终成功（退款完成或换货完成），整单奖励归平台（规则 14）

---

## 规则 22：换货后再退货限制

### 背景

防止"换货拿新品 → 无理由退货退款"的薅羊毛路径。

### 规则

- 换货完成（状态 COMPLETED）后收到的新商品
- **不允许**申请"七天无理由退货"
- **仅允许**申请"质量问题退货退款"（新商品确实有质量问题）

### 审核方式

- 跳过卖家审核，直接走平台仲裁
- 平台管理员审核照片凭证，判断是否属实

### 实现

- 售后申请记录中标记 `isPostReplacement = true`
- 系统自动跳过 UNDER_REVIEW 阶段，状态直接到平台待仲裁

---

## 规则 23：换货也按阈值决定是否寄回

### 逻辑与退货一致

- 换货商品金额 ≤ `RETURN_NO_SHIP_THRESHOLD`：不用寄回旧商品，卖家直接补发新商品
- 换货商品金额 > `RETURN_NO_SHIP_THRESHOLD`：买家需要寄回旧商品，卖家收到后再发新商品

### 运费承担

- 质量问题换货需要寄回：运费平台承担（与退货一致）

### 流程差异

- 不用寄回：APPROVED → 卖家直接发出新商品 → REPLACEMENT_SHIPPED → 买家确认 → COMPLETED
- 需要寄回：APPROVED → 买家寄回 → RETURN_SHIPPING → 卖家验收 → RECEIVED_BY_SELLER → 卖家发新商品 → REPLACEMENT_SHIPPED → 买家确认 → COMPLETED
