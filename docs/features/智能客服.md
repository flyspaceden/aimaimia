## 1. 功能概述

智能客服是爱买买平台的买家↔平台沟通通道，采用 **三层路由** 架构：

```
买家发消息
   ↓
[第一层] FAQ 关键词匹配（0 LLM 调用，<50ms）
   ↓ 未命中
[第二层] AI 意图理解 + 自然语言回复（Qwen-Flash，~500ms）
   ↓ 无法理解 / 用户主动转人工 / 情绪激动
[第三层] 转人工坐席（Socket.IO 实时聊天）
```

**核心特性**：
- ✅ 三层路由：FAQ → AI → 人工，绝大部分问题在前两层消化
- ✅ 实时通讯：Socket.IO 双向实时聊天，支持买家↔坐席
- ✅ 上下文感知：从订单/售后页进入时，AI 自动获取相关订单信息
- ✅ 会话超时：2 小时无活动自动关闭旧会话
- ✅ 自动转人工：用户主动要求 / AI 连续 2 次无法理解 / 检测到情绪激动
- ✅ 工单系统：转人工自动生成工单，AI 自动生成对话摘要
- ✅ 满意度评价：会话结束后用户评分

## 3. 数据模型

数据库共 **8 个表 + 10 个枚举**，定义在 `backend/prisma/schema.prisma` 末尾的 N 域。

### 3.1 模型关系

```
CsTicket (工单)
   │ 1:N
   ↓
CsSession (会话) ──→ CsRating (评价, 1:1)
   │ 1:N
   ↓
CsMessage (消息)

CsAgentStatus (坐席状态)  - 独立表，按 adminId 跟踪
CsFaq (FAQ 规则库)        - 独立表
CsQuickEntry (快捷入口)   - 独立表（前端展示用）
CsQuickReply (坐席回复)   - 独立表（坐席工作台用）
```

### 3.2 关键模型字段

**CsSession**（会话）：
- `id`、`ticketId?`、`userId`
- `status`：`AI_HANDLING` / `QUEUING` / `AGENT_HANDLING` / `CLOSED`
- `source`：`MY_PAGE` / `ORDER_DETAIL` / `AFTERSALE_DETAIL`
- `sourceId?`：关联订单或售后单 ID
- `agentId?`、`agentJoinedAt?`、`closedAt?`、`createdAt`

**CsMessage**（消息）：
- `id`、`sessionId`、`senderType`：`USER` / `AI` / `AGENT` / `SYSTEM`
- `contentType`：`TEXT` / `RICH_CARD` / `ACTION_CONFIRM` / `ACTION_RESULT` / `IMAGE`
- `content`、`metadata?`、`routeLayer?`（命中的路由层）

**CsAgentStatus**（坐席状态）：
- `adminId @unique`、`status`（ONLINE/BUSY/OFFLINE）
- `currentSessions`、`maxSessions`（默认 5）

**CsFaq**（FAQ 规则）：
- `keywords[]`、`pattern?`（正则）、`answer`、`answerType`、`priority`、`enabled`

完整字段参考 schema.prisma 第 N 域部分。

### 4.2 三层路由详解（cs-routing.service.ts）

**输入**：消息文本、AI 上下文（含订单信息）、连续失败次数

**优先级判断顺序**：

1. **转人工关键词**（最高优先）：检测 `转人工`、`找客服`、`找人工`、`人工客服`、`真人客服` → 立即返回 layer 3
2. **情绪/投诉关键词**：检测 `投诉`、`骗子`、`欺诈`、`举报`、`报警`、`工商`、`消协`、`12315` → 返回道歉消息 + 转人工
3. **第一层 FAQ**：调用 `CsFaqService.match()` → 关键词命中或正则命中 → 返回 layer 1 + FAQ 答案
4. **第二层 AI**：调用 Qwen-Flash 意图分类 → 返回 6 种意图之一：
   - `query_logistics`：查物流
   - `query_aftersale`：查退款进度
   - `apply_aftersale`：引导申请退货
   - `cancel_order`：取消订单（需用户二次确认）
   - `query_coupon`：查优惠券
   - `general_qa`：平台规则问答
5. **AI 失败兜底**：连续 2 次无法理解 → 自动转人工；单次失败 → 返回提示用户重新描述
6. **AI 超时保护**：DashScope 调用 10 秒超时（AbortController）

**安全机制**：
- LLM Prompt 注入风险：`contextInfo` 直接拼入 system prompt（**待优化**）
- API Key 缺失时跳过 AI，直接走第三层

### 4.3 坐席分配（cs-agent.service.ts）

**自动分配（assignAgent）**：使用原子 SQL，避免竞态：
```sql
UPDATE "CsAgentStatus"
SET "currentSessions" = "currentSessions" + 1,
    "lastActiveAt" = NOW()
WHERE "adminId" = (
  SELECT "adminId" FROM "CsAgentStatus"
  WHERE status = 'ONLINE' AND "currentSessions" < "maxSessions"
  ORDER BY "currentSessions" ASC
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
RETURNING "adminId"
```

**手动接入（agentAcceptSession）**：
1. 先用 raw query 检查 `currentSessions < maxSessions` 并原子递增
2. 容量校验通过后用 `updateMany WHERE status=QUEUING` 做 CAS 更新会话状态
3. 任一步失败都回滚坐席计数

**坐席断线处理（handleDisconnect）**：
1. 将该坐席的 `AGENT_HANDLING` 会话退回 `QUEUING`（agentId=null）
2. 坐席状态标记 `OFFLINE`，`currentSessions=0`
3. Gateway 30 秒宽限期：坐席重连则取消断线，否则才执行上述清理

### 4.4 会话生命周期（cs.service.ts）

**状态转换图**：
```
                  createSession
                       │
                       ▼
              ┌──────────────┐
              │ AI_HANDLING  │
              └──────┬───────┘
                     │
       ┌─────────────┼─────────────┐
       │             │             │
   AI 解决      shouldTransfer   超时(2h)/
   closeSession    =true        关闭
       │             │             │
       │             ▼             │
       │      ┌──────────┐         │
       │      │ QUEUING  │◄──── 坐席断线
       │      └────┬─────┘
       │           │
       │      assignAgent
       │      或 manual accept
       │           │
       │           ▼
       │      ┌──────────────┐
       │      │AGENT_HANDLING│
       │      └──────┬───────┘
       │             │
       │       closeSession
       │             │
       ▼             ▼
       └──────► CLOSED
```

**关键方法**：
- `createSession(userId, source, sourceId)` - 创建或恢复会话；超过 2 小时空闲自动关闭旧会话
- `handleUserMessage(sessionId, userId, content)` - 保存消息 + 路由 + 返回 `{ userMessage, aiReply, transferred }`
  - **重要**：`AGENT_HANDLING` 和 `QUEUING` 状态都跳过路由（只保存消息），防止重复转人工
- `transferToAgent(sessionId)` - CAS 更新 status=AI_HANDLING→QUEUING 后才创建工单（防并发重复工单）
- `closeSession(sessionId)` - 释放坐席计数 + 标记会话 CLOSED + 标记工单 RESOLVED
- `submitRating(sessionId, userId, score, tags, comment)` - 提交满意度评价

**会话空闲超时**：
- 配置：`SESSION_IDLE_TIMEOUT_MS`（cs.service.ts 第 24 行）
- 当前值：`5 * 1000` (5 秒，**测试用**)
- 上线前改回：`2 * 60 * 60 * 1000`（2 小时）
- 触发时机：用户重新调用 `createSession` 时检查最后消息时间，超时则关闭旧会话开新会话

### 4.6 HTTP→Socket.IO 桥接（核心机制）

**场景**：买家通过 HTTP 发消息，坐席通过 Socket.IO 实时收消息。

**实现**（`cs.controller.ts:30-73`）：
```typescript
@Post('sessions/:id/messages')
async sendMessage(...) {
  const result = await this.csService.handleUserMessage(...);

  // HTTP→Socket.IO 桥接
  const server = this.csGateway.server;
  if (server) {
    server.to(`session:${sessionId}`).emit('cs:message', result.userMessage);
    if (result.aiReply) {
      server.to(`session:${sessionId}`).emit('cs:message', result.aiReply);
    }
    if (result.transferred) {
      // 通知坐席接入
      server.to(`agent:${agentId}`).socketsJoin(`session:${sessionId}`);
      server.to(`session:${sessionId}`).emit('cs:agent_joined', {...});
    } else if (result.routeResult?.shouldTransferToAgent) {
      // 排队通知
      server.to('agent:lobby').emit('cs:new_ticket', {...});
    }
  }
  return result;
}
```

## 5. Socket.IO Gateway（cs.gateway.ts）

### 5.1 命名空间和认证

- **命名空间**：`/cs`
- **CORS**：从 `process.env.ALLOWED_ORIGINS` 读取（默认 localhost:3000/5173/8081）
- **认证**：连接时通过 `client.handshake.auth.token` 提供 JWT
  - 先尝试用 `JWT_SECRET` 解析买家 token
  - 失败则尝试 `ADMIN_JWT_SECRET` 解析坐席 token
  - 都失败则 `client.disconnect()`
- **ValidationPipe**：所有 `@SubscribeMessage` 自动校验 payload

### 5.2 房间设计

| 房间 | 谁加入 | 用途 |
|------|--------|------|
| `user:{userId}` | 买家连接时 | 个人通知 |
| `agent:{adminId}` | 坐席连接时 | 个人通知 |
| `agent:lobby` | 所有坐席 | 排队通知广播 |
| `session:{sessionId}` | 会话双方 | 消息双向同步 |

### 5.3 事件清单

**客户端 → 服务端**：
| 事件 | 发送方 | 用途 |
|------|--------|------|
| `cs:send` | 买家/坐席 | 发送消息 |
| `cs:accept_ticket` | 坐席 | 领取排队会话 |
| `cs:close_session` | 坐席 | 结束会话 |
| `cs:typing` | 双方 | 正在输入 |
| `cs:agent_status` | 坐席 | 更新在线状态 |

**服务端 → 客户端**：
| 事件 | 接收方 | Payload | 用途 |
|------|--------|---------|------|
| `cs:message` | 会话房间 | CsMessage | 新消息推送 |
| `cs:agent_joined` | 会话房间 | `{sessionId, agentName}` | 坐席接入通知 |
| `cs:session_closed` | 会话房间 | `{sessionId}` | 会话关闭通知 |
| `cs:new_ticket` | agent:lobby | `{sessionId, userId, userNickname, ...}` | 新排队会话 |
| `cs:queue_update` | agent:lobby | `{queueCount}` | 排队数变化 |
| `cs:typing` | 会话房间 | `{sessionId, senderType}` | 对方正在输入 |
| `cs:error` | 单个 client | `{message}` | 错误通知 |

### 5.4 重要细节

- **去重机制**：坐席发送消息时使用 `client.to(room).emit`（排除发送者自己），前端已本地添加，避免重复
- **断线处理**：30 秒宽限期，重连则恢复，超时才标记离线 + 退回会话
- **重连恢复**：坐席重连时自动重新加入正在处理的 session 房间（`getActiveSessionIds`）
- **错误处理**：所有 handler 包裹 try-catch，错误通过 `cs:error` 推给客户端

### 6.3 用户操作流程

#### 流程 A：从「我的」进入（无订单上下文）

```
我的页面 → 点击「联系客服」工具
   ↓
进入 /cs?source=MY_PAGE
   ↓
useEffect 调用 CsRepo.createSession('MY_PAGE') → 后端返回 sessionId
   ↓
显示 AI 欢迎语 + 快捷操作 + 热门问题
   ↓
用户点击「查物流」或输入"我的快递到哪了"
   ↓
CsRepo.sendMessage(sessionId, "...")
   ↓
后端 handleUserMessage → 三层路由 → 返回 { userMessage, aiReply, transferred }
   ↓
本地添加 userMessage（即时显示）
等待 API 返回 → 添加 aiReply
   ↓
用户继续对话或点击「结束会话」按钮
   ↓
如点结束 → 调用 closeSession(sessionId) → 显示评价弹窗 + "重新开始对话"按钮
```

#### 流程 B：从订单详情进入（有订单上下文）

与流程 A 相同，但 `source=ORDER_DETAIL&sourceId=${orderId}`，AI 在意图识别时会获取该订单的状态、商品、金额等信息。

#### 流程 C：用户离开后回来

```
用户在客服页聊天 → 按返回键 → 回到上一级页面
   ↓
... 间隔时间 ...
   ↓
再次点击「联系客服」
   ↓
CsRepo.createSession() 调用后端
   ↓
后端检查活跃会话：
  - 距上次消息 < 2 小时 → 返回 isExisting=true，恢复历史消息
  - 距上次消息 ≥ 2 小时 → 自动 closeSession 旧会话，创建新会话，返回 isExisting=false
```

### 6.4 「重新开始对话」按钮

会话结束后（`sessionClosed=true`），底部输入栏被替换为绿色「重新开始对话」按钮：
```typescript
onPress={() => {
  setSessionClosed(false);
  setSessionId(null);
  setMessages([]);
  CsRepo.createSession(source, sourceId).then((res) => {
    if (res.ok) setSessionId(res.data.sessionId);
  });
}}
```

## 8. 跨系统数据流

### 8.1 完整买家发消息流程

```
[1] 买家 App
    用户输入"我的快递到哪了" → handleSend()

[2] CsRepo.sendMessage(sessionId, content)
    POST /api/v1/cs/sessions/:id/messages
    Body: { content: "我的快递到哪了", contentType: "TEXT" }

[3] 后端 cs.controller.ts: sendMessage()
    ├── 调用 csService.handleUserMessage()
    │     ├── 校验会话归属
    │     ├── 保存 USER 消息到数据库
    │     ├── (如果 AI_HANDLING) 调用 routingService.route()
    │     │     ├── 检查转人工/情绪关键词
    │     │     ├── 第一层：FAQ 匹配
    │     │     ├── 第二层：调用 Qwen-Flash 意图分类
    │     │     └── 返回 CsRouteResult
    │     ├── 保存 AI 回复消息（routeLayer 标注哪一层）
    │     └── (如果 shouldTransferToAgent) 调用 transferToAgent()
    └── HTTP→Socket.IO 桥接
          ├── server.to('session:xxx').emit('cs:message', userMsg)
          ├── server.to('session:xxx').emit('cs:message', aiReply)
          └── (如转人工) server.to('agent:lobby').emit('cs:new_ticket', ...)

[4] HTTP 响应返回 App
    { userMessage, aiReply, transferred }

[5] App 渲染
    setMessages([...prev, aiReply])
    （userMessage 已在发送时本地添加）

[6] 同时管理后台坐席收到 Socket.IO 推送
    socket.on('cs:message', (msg) => 更新会话消息列表)
    socket.on('cs:new_ticket', (data) => 排队列表新增项目)
```

### 8.2 转人工流程

```
买家说"转人工" → 后端检测到 → 路由返回 layer=3, shouldTransferToAgent=true
   ↓
csService.transferToAgent(sessionId)
   ├── CAS: UPDATE csSession WHERE status=AI_HANDLING SET status=QUEUING
   │   （防并发重复触发）
   ├── csTicketService.createTicket(sessionId)
   │   ├── 查询会话消息
   │   ├── 调用 Qwen-Flash 生成 AI 摘要（10s 超时）
   │   └── 创建 csTicket 记录
   ├── csAgentService.assignAgent()
   │   └── 原子 SQL：FOR UPDATE SKIP LOCKED 选取最少会话的在线坐席
   ├── (有可用坐席) UPDATE csSession SET status=AGENT_HANDLING, agentId=...
   └── (无可用坐席) 保持 QUEUING 状态
   ↓
HTTP→Socket.IO 桥接广播
   ├── 已分配 → cs:agent_joined 给会话房间
   └── 排队中 → cs:new_ticket 给 agent:lobby
   ↓
坐席工作台收到事件 → 排队列表新增 → 坐席点「接入」
   ↓
emit('cs:accept_ticket', { sessionId })
   ↓
csService.agentAcceptSession(sessionId, adminId)
   ├── 原子 SQL 容量检查：UPDATE csAgentStatus WHERE currentSessions < maxSessions
   ├── (容量满) 抛 BadRequestException
   ├── CAS: updateMany csSession WHERE status=QUEUING SET status=AGENT_HANDLING
   └── (CAS 失败) 回滚坐席计数
   ↓
广播 cs:agent_joined
```

### 8.3 会话超时自动关闭流程

```
买家在 App 客服页聊天 → 按返回 → 离开页面
   ↓
... 2 小时后（测试为 5 秒）...
   ↓
买家再次点击「联系客服」
   ↓
CsRepo.createSession(source, sourceId)
   ↓
后端 csService.createSession()
   ├── 查询活跃会话（status in [AI_HANDLING, QUEUING, AGENT_HANDLING]）
   ├── 找到旧会话 → 计算 idleMs = now - 最后消息时间
   ├── (idleMs > SESSION_IDLE_TIMEOUT_MS) 调用 closeSession() 静默关闭
   │   ├── releaseAgent（如有坐席）
   │   ├── csSession SET status=CLOSED
   │   └── csTicket SET status=RESOLVED（如有工单）
   └── 创建新会话 csSession.create()
   ↓
管理后台通过 cs:queue_update 事件 + 10s 轮询感知列表变化
```

## 9. 已修复的 Bug 历史

### 9.1 第一轮：CRITICAL + HIGH 安全/逻辑（2026-04-08）
1. **缺少 POST 消息端点** → 新增 `POST /cs/sessions/:id/messages`
2. **HTTP→Socket.IO 桥接缺失** → Controller 调 Gateway.server 广播
3. **Socket 房间无归属校验** → 校验后才 `client.join`
4. **Socket 无输入校验** → 添加 ValidationPipe + 内容长度限制
5. **ReDoS 风险** → `isSafeRegex()` 检测嵌套量词 + 截断输入
6. **坐席分配竞态** → `UPDATE...RETURNING + FOR UPDATE SKIP LOCKED`
7. **DTO 无长度限制** → 全字段 `@MaxLength`
8. **API 调用无超时** → AbortController 10s
9. **CORS `origin: '*'`** → 改为环境变量配置
10. **sourceId null/undefined 不一致** → 统一 `|| null`
11. **Quick Entry/Reply 无审计** → 添加 `@AuditLog`
12. **FAQ 无缓存** → 1 分钟内存缓存

### 9.2 第二轮：跨系统协同（2026-04-08）
1. **sendMessage 返回结构不匹配** → 新增 `CsSendMessageResult` 类型
2. **结束会话不通知后端** → 新增 `closeSession` 端点 + App 调用
3. **QUEUING 消息重复建工单** → `handleUserMessage` 跳过 QUEUING
4. **agentAcceptSession 竞态+不递增** → CAS + upsert 递增
5. **管理列表过滤 enabled=true** → 去掉过滤
6. **agentHandled 误标为响应时间** → 改为"人工处理 次"

### 9.3 第三轮：深度审查（2026-04-08）
1. **坐席断线会话变孤儿** → `handleDisconnect` 退回 QUEUING + 重连恢复 session 房间
2. **并发重复工单** → `transferToAgent` CAS 防并发
3. **手动接入无容量检查** → raw query 加 maxSessions 检查
4. **@CurrentUser → @CurrentAdmin** → 对齐项目约定

### 9.4 第四轮：FAQ 匹配 bug
- **空字符串关键词匹配所有消息** → `kw.length > 0` 跳过

### 9.5 第五轮：实际测试发现
1. **管理后台菜单需双击 + 闪屏** → 菜单分组路径从 `/cs` 改为 `/customer-service`（避免与子路由冲突）
2. **坐席消息显示两次** → `client.to(room).emit` 排除发送者
3. **结束会话后无重新开始入口** → 替换为「重新开始对话」按钮
4. **管理后台结束会话状态不清理** → 清除 activeSessionId + localMessages
5. **会话超时不通知管理后台** → createSession 后向 agent:lobby 广播

## 12. 待优化项 / 已知问题

### 12.1 高优先级（建议尽快处理）

1. **LLM Prompt 注入风险**
   - 文件：`cs-routing.service.ts:101-110`
   - 问题：用户消息和订单 ID 直接拼入 system prompt，未做转义
   - 影响：恶意用户可能操纵 AI 回复或绕过意图分类
   - 修复方向：使用 `JSON.stringify()` 转义，或将上下文放入 user message 而非 system prompt

2. **没有前端测试**
   - 影响：UI 交互问题、组件回归、跨系统契约都靠手动测试发现
   - 修复方向：用 React Testing Library 给关键组件写测试，重点是 `app/cs/index.tsx` 的状态机和 `workstation.tsx` 的 Socket.IO 集成

3. **没有 E2E 测试**
   - 影响：无法验证完整链路（买家发消息 → 后端路由 → 坐席接收）
   - 修复方向：用 supertest + 真实数据库做 HTTP 端点测试，用 socket.io-client 测试实时事件

### 12.2 中优先级

4. **会话清理 cron**
   - 当前：超时清理只在用户回访时触发
   - 问题：用户永不回访的话，旧会话永远是 `AI_HANDLING`/`AGENT_HANDLING`，污染统计
   - 修复方向：加 `@nestjs/schedule` 定时任务，每小时清理超过阈值的空闲会话

5. **坐席快速回复模板缺少分类管理 UI**
   - 当前：分类靠手动输入字符串
   - 修复方向：分类做成预设枚举或独立管理

6. **AI 摘要生成阻塞 transferToAgent**
   - 当前：摘要生成失败时降级跳过，但成功路径会等待 LLM 调用（最多 10s）
   - 修复方向：摘要生成异步化，工单先创建（summary=null），后台异步补全

7. **FAQ 优先级重复处理**
   - 当前：同一关键词命中多条 FAQ 时，按 priority desc 取第一条，但代码用 `for...of` 遍历，未利用数据库 ORDER BY 之外的优化
   - 影响：100+ FAQ 时性能可能下降
   - 修复方向：构建关键词倒排索引

### 12.3 低优先级

8. **管理后台坐席状态管理 UI**
   - 当前：没有专门的坐席在线状态管理页面
   - 修复方向：在工作台或独立页面展示所有坐席的实时状态、当前会话数、忙闲

9. **快捷回复按使用频率排序**
   - 当前：按 sortOrder 静态排序
   - 修复方向：记录每条快捷回复的使用次数，自动按热度排序

10. **AI 失败兜底文案太机械**
    - 当前："抱歉我没太理解您的意思，能再描述一下您的问题吗？"
    - 修复方向：根据上下文生成更自然的兜底回复

11. **没有客服会话历史导出功能**
    - 影响：合规审计时不方便导出
    - 修复方向：管理后台增加导出 Excel/PDF 入口

12. **没有黑名单/封禁机制**
    - 当前：恶意用户可以无限发消息
    - 修复方向：基于 user 维度的限流和封禁

### A.2 坐席发送消息后看到两条相同消息

**已知 bug**：已修复（commit f9736dc）
**机制**：Gateway 用 `client.to()` 排除发送者
**回归测试**：`cs.gateway.spec.ts` 中的"坐席发消息"测试用例

### A.3 管理后台菜单点击需要双击/闪屏

**已知 bug**：已修复（commit b2d7d8b）
**根因**：菜单分组 path `/cs` 与子路由 `/cs/workstation` 冲突
**修复**：分组改为虚拟路径 `/customer-service`

### A.4 用户回到客服时看到旧对话，期望是新会话

**预期行为**：2 小时内回访恢复旧会话；超过 2 小时自动开新会话
**配置**：`SESSION_IDLE_TIMEOUT_MS`（cs.service.ts:24）
**测试模式**：当前为 5 秒，便于快速测试

### A.5 坐席接入会话失败"会话不在排队状态或已被其他坐席接入"

**正常行为**：CAS 防竞态机制，两个坐席同时点接入时只有一个成功
**前端处理**：捕获错误后刷新列表

### A.6 AI 回复不出现，只有用户消息

**可能原因**：
1. `DASHSCOPE_API_KEY` 未配置 → AI 路由跳过，返回兜底
2. DashScope API 超时（>10s）→ 检查后端日志
3. AI 返回 confidence < 0.6 → 降级为兜底回复
4. 网络问题 → 检查后端是否能访问 dashscope.aliyuncs.com

### A.7 工单 AI 摘要为空

**正常行为**：摘要生成失败时静默跳过，工单仍创建但 summary 字段为 null
**可能原因**：DASHSCOPE_API_KEY 未配置、API 超时、API 返回非预期格式
**日志**：搜 `AI 摘要生成失败` 关键字
