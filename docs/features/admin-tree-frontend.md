# 管理后台分润树页面改造方案

## 1. 文档目标

本文档用于明确管理后台两个树页面的重构方向：

- `VIP 奖励树` `/bonus/vip-tree`
- `普通奖励树` `/bonus/normal-tree`

目标不是单纯优化样式，而是让运营、财务、客服能够：

- 快速定位任意用户在树中的位置
- 看明白该用户的上级、下级和所处层级
- 看明白某笔奖励为什么发给这个人，或为什么没有发给这个人
- 看明白 VIP 树和普通树"奖励规则相同，但入树规则不同"
- 从会员详情、订单、流水等页面快速跳转并定位到树中的具体用户

---

## 2. 核心业务事实

这两个页面的设计必须建立在以下业务前提上，不能混淆。

### 2.1 共性：奖励分配规则基本一致

VIP 和普通用户的分润奖励，在"奖励从哪里往哪里走"这件事上是同构的：

- 都是**从下到上**
- 用户第 `k` 次有效消费，对应第 `k` 层祖辈
- 目标祖辈必须先完成 `k` 次消费，才能解锁该层奖励
- 未解锁则进入冻结状态
- 超过冻结时限后，奖励归平台
- 若没有合格祖辈，奖励归平台

也就是说，这两个页面的"奖励路径解释""冻结/解锁说明""奖励去向说明"应尽量共用。

### 2.2 差异：树的生成方式不同

两套系统真正不同的是"树从上到下怎么长出来"。

#### VIP 树生成规则

- 系统根节点为 A1-A10 等高管根
- 有推荐人时优先挂到推荐人子树
- 子树满位后按 BFS / 滑落逻辑继续落位
- 无推荐人时落到系统根节点下

#### 普通树生成规则

- 单个平台系统根节点
- 用户首笔有效消费确认收货后自动入树
- 无推荐码
- 采用**轮询平衡插入**
- 同一层要先让每个父节点分到第 1 个子节点，再轮到第 2 个，再轮到第 3 个

所以，这两个页面的"树结构解释""为什么落到这个位置""位置生成原因"必须分开表达。

### 2.3 设计结论

树页面应采用：

- **同一套分润解释内核**
- **两套入树解释外壳**

不能把两个页面做成完全独立的两套产品，也不能做成只是换个颜色的同一页面。

## 3. 当前实现现状

### 3.1 代码结构

当前两个页面的实现情况：

| 文件 | 行数 | 说明 |
|------|------|------|
| `vip-tree.tsx` | 547 | VIP 树页面，主题色蓝色 `#1E40AF` |
| `normal-tree.tsx` | 559 | 普通树页面，主题色绿色 `#2E7D32` |
| `components/TreeNode.tsx` | 194 | 共享节点卡片组件 |
| `components/NodeDetail.tsx` | 154 | 共享详情面板组件 |

两个页面代码 **~95% 重复**，仅主题色和 API 端点不同。这是最大的技术债务。

### 3.2 当前已有能力

- 搜索用户（手机号/ID/昵称）带自动补全
- 面包屑导航（祖先链上溯到根节点，带循环保护 64 跳）
- 节点卡片显示：状态点、昵称、VIP 标签、自购次数、累计奖励、子节点数
- 缩放控制（40%-120%，支持 Ctrl+滚轮）
- 深度控制（1-5 层，但后端实际只返回 2 层）
- 双击重新定位
- 可折叠右侧详情面板
- 懒加载子节点接口（`getChildren`）已存在

### 3.3 后端已有接口

| 接口 | 说明 | 返回内容 |
|------|------|----------|
| `GET /admin/bonus/vip-tree/search` | 搜索用户 | `userId, nickname, phone, tier, hasVipNode` |
| `GET /admin/bonus/vip-tree/context` | 树上下文 | `breadcrumb, parent, current, children`（含孙节点） |
| `GET /admin/bonus/vip-tree/:userId/children` | 懒加载子节点 | 直接子节点列表 |
| `GET /admin/bonus/normal-tree/search` | 搜索用户 | 同 VIP，过滤已入普通树用户 |
| `GET /admin/bonus/normal-tree/context` | 树上下文 | 同 VIP 结构 |
| `GET /admin/bonus/normal-tree/:userId/children` | 懒加载子节点 | 同 VIP |
| `GET /admin/bonus/members/:userId` | 会员详情 | 含 wallet、tree、最近 20 条 ledger、最近 10 条 withdrawal |

## 4. 当前页面主要问题

### 4.1 看树能力不够

- 现在本质上只看"父节点 + 当前节点 + 子节点 + 孙节点"
- 虽然页面上有深度输入（1-5），但后端上下文接口实际只返回 2 层后代
- 运营无法连续查看更深层结构
- 运营很难判断用户在整个树中的真实位置

### 4.2 看得到结构，看不到原因

- 现有页面更偏静态结构展示
- 缺少"为什么这笔奖励给了这个人/没给这个人"的解释
- 缺少"冻结原因、解锁条件、归平台原因"的可视化说明
- 缺少"第 k 次消费对应第 k 层祖辈"的路径演示

### 4.3 普通树语义不够独立

- 普通树前端当前基本复用 VIP 树页面与节点详情
- 这会导致普通树页面没有突出"轮询平衡入树"
- 没有突出"成为 VIP 后普通树位置保留但不再接收新奖励"
- 没有突出"平台根节点 / 自然停止 / 归平台"的普通树特有语义

### 4.4 跳转链路不顺

- 会员详情已提供跳转树页面入口
- 但树页面应支持自动读取 URL 中的 `userId` 并立即定位
- 需要支持从更多地方带条件跳入，如：
  - 会员详情
  - 订单详情
  - 奖励流水
  - 提现审核

### 4.5 详情面板价值不够

- 右侧面板当前偏"基础资料卡"（4 个统计卡 + 基础信息）
- 缺少奖励分析、路径解释、关联订单、冻结明细
- 不能支撑运营处理实际问题

### 4.6 交互可发现性差

- 双击重新定位是当前页面最重要的导航方式，但完全不可发现
- 没有右键菜单或显式操作按钮
- 新用户不知道节点可以交互

### 4.7 代码重复严重

- VIP 和普通两个页面各 ~550 行，95% 逻辑一致
- 任何改动都要在两个文件中同步修改
- 增加维护成本，容易产生不一致

## 5. 设计原则

### 5.1 先解释业务，再展示结构

树页面不是图形展示页，而是分润分析页。

用户首先要知道：

- 这个人为什么在这里
- 这个人能不能收奖励
- 这笔奖励为什么在这里

然后才是看树结构本身。

### 5.2 共性统一，差异显性

以下模块两页尽量共用：

- 搜索栏
- 面包屑
- 树画布
- 节点卡片基础结构
- 路径高亮
- 奖励记录列表
- 冻结/解锁说明

以下内容必须区分：

- 页面标题与说明文案
- 入树规则解释
- 节点补充字段
- 平台归因说明

### 5.3 默认服务运营排查场景

页面默认要支持以下高频场景：

- 查一个用户为什么没收到奖励
- 查一个用户为什么奖励被冻结
- 查一笔奖励为什么归平台
- 查一个用户是怎么落到这个位置的
- 查某个 VIP 用户上溯关系是否正确
- 查普通用户在成为 VIP 后是否还在普通树收奖励

### 5.4 先把定位和解释做好，再做高级动画

优先级顺序应为：

1. 可定位
2. 可解释
3. 可分析
4. 可视化增强

不要先做动画和大图，再补核心业务信息。

### 5.5 下一阶段体验优化优先级

在当前 6 个 Phase 已完成的基础上，下一轮优化应继续围绕"降低进入门槛"和"提高高频排查效率"展开。

#### P0：优先立即做

1. **根节点快捷入口**
   - VIP 树首屏直接展示 `A1-A10` 十棵根树入口卡片
   - 每张卡片显示：根节点 ID、节点总数、活跃率、本周新增
   - 普通树首屏展示 `平台根节点` 单张入口卡片，一键进入
   - 这是最高优先级，因为它直接解决"不知道从哪开始"

2. **空状态引导**
   - 当搜索框为空且未定位用户时，显示明确引导区
   - 至少提供两个动作：
     - `从根节点开始浏览`
     - `搜索昵称 / 手机号 / ID`
   - 不能只放一段静态说明文案

3. **搜索框自动补全增强**
   - 输入 1-2 个字符即可触发联想
   - 支持昵称模糊匹配、手机号后四位、用户 ID 前缀
   - 每条结果显示：头像、昵称、等级、状态标签、是否已入当前树

#### P1：第二层增强

4. **搜索历史 + 最近浏览**
   - 搜索框聚焦时展示最近搜索关键词和最近浏览节点
   - 使用 `localStorage` 持久化即可

5. **异常优先快捷筛选**
   - 优先放：
     - `最近加入`
     - `即将冻结`
     - `沉默用户`
     - `已出局`
     - `升级 VIP 后停收普通奖励`
   - 点击后先展示用户列表，再决定跳转哪个节点

6. **统计概览卡片**
   - 建议只保留 4 个最强指标：
     - 总节点数
     - 活跃率
     - 本周新增
     - 冻结 / 异常节点数
   - 每个数字都应可点击，并展开对应用户列表

#### P2：进一步打磨

7. **当前视角提示**
   - 例如：`当前：VIP树 / A3根树 / 子树模式 / 来源：会员详情`

8. **路径面包屑复制**
   - 节点详情中显示简化路径：
     - `A3 > 上级节点 > 当前用户`
   - 支持一键复制，便于客服和运营沟通

9. **同屏列表模式**
   - 增加"当前层节点列表 / 当前子树节点列表"
   - 支持按收入、状态、加入时间排序

## 6. 页面总体方案

两个页面采用同一套大框架：

```text
页头说明区
  标题 / 体系说明 / 跳转来源提示
  根节点快捷入口 / 统计概览卡片（下一阶段）

顶部操作区
  搜索 / 回到根节点 / 回到当前用户 / 展开深度 / 刷新
  搜索历史 / 最近浏览 / 快捷筛选（下一阶段）

主体区：左右布局
  左侧：树画布
  右侧：分析面板（可折叠，小屏幕变为抽屉）

底部辅助区
  图例 / 规则说明
```

### 6.1 两种查看模式

初期两个页面支持两种模式，第三种模式（奖励路径模式）在 Phase 5 中单独实现。

#### A. 定位模式（默认）

用于快速找到某个用户。

展示重点：

- 祖先链（面包屑）
- 当前节点（高亮）
- 当前节点的直接下级
- 所在层级

适合：

- 客服查询用户位置
- 运营快速确认是否入树

#### B. 子树模式

用于看一个用户往下的若干层结构。

展示重点：

- 当前节点作为子树根
- 向下展开 2~4 层
- 支持逐层展开/收起
- 只展示当前分支，不展示全树无关部分
- 通过 `getChildren` 懒加载接口按需加载更深层

适合：

- 运营查看某个 VIP 子树
- 查看普通树某层扩张状态

## 7. 代码复用策略

### 7.1 提取 TreeViewer 共享组件

当前两个页面 ~95% 代码重复，**Phase 1 首先要解决这个问题**。

提取共享的 `TreeViewer` 组件（预计 ~400 行），包含：

- 搜索栏 + 自动补全
- 面包屑导航
- 树画布渲染（节点卡片 + 连线）
- 缩放控制
- 深度控制
- 详情面板框架
- URL 状态同步

VIP 和 Normal 页面瘦身为 ~100 行配置文件：

```typescript
// vip-tree.tsx 瘦身后的结构
const VipTreePage = () => (
  <TreeViewer
    treeType="vip"
    themeColor="#1E40AF"
    title="VIP 分润树"
    subtitle="入树规则：推荐优先，子树满位后按 BFS 滑落"
    rewardRule="奖励规则：第 k 次有效消费，上溯到第 k 层祖辈"
    searchApi={searchVipTreeUsers}
    contextApi={getVipTreeContext}
    childrenApi={getVipTreeChildren}
    // VIP 特有配置
    extraNodeFields={vipNodeFields}
    extraDetailTabs={vipDetailTabs}
    placementExplain={VipPlacementExplain}
  />
);
```

### 7.2 组件拆分

共享组件（`admin/src/pages/bonus/components/`）：

| 组件 | 职责 |
|------|------|
| `TreeViewer` | 页面主容器，组合所有子组件 |
| `TreeCanvas` | 树画布渲染 + 拖拽/滚动 + 空位占位 |
| `TreeNodeCard` | 节点卡片（升级现有组件） |
| `TreeBreadcrumb` | 面包屑导航 |
| `NodeDetailPanel` | 右侧分析面板（Tab 结构，升级现有 NodeDetail） |
| `TreeLegend` | 底部图例说明 |

各页面独有组件：

| 组件 | 页面 | 职责 |
|------|------|------|
| `VipPlacementExplain` | VIP | 推荐落位解释卡 |
| `NormalPlacementExplain` | 普通 | 轮询落位解释卡 |

### 7.3 组件拆分原则

共享组件负责：

- 布局
- 基础树渲染
- 通用交互
- 通用图例和路径高亮

各页面负责：

- 文案
- 业务字段映射
- 特有解释卡

## 8. 共用模块设计

### 8.1 页面头部

头部应包含：

- 页面标题
- 一句话解释该树的生成方式
- 一句话解释奖励是如何从下到上分配的
- 当前定位来源标签（如"来自：会员详情 - 张三"）

示例：

#### VIP 奖励树

- 标题：`VIP 分润树`
- 副标题：`推荐优先 + BFS 滑落落位，奖励按第 k 次消费上溯到第 k 层祖辈`
- 视角提示：`当前：VIP 树 / A1-A10 根树入口 / 来源：会员详情`

#### 普通奖励树

- 标题：`普通分润树`
- 副标题：`首单入树 + 轮询平衡落位，奖励按第 k 次消费上溯到第 k 层祖辈`
- 视角提示：`当前：普通树 / 平台根节点 / 来源：订单详情`

### 8.1.1 根节点快捷入口

这部分应位于页头说明区下方、顶部工具栏上方，作为首屏入口层。

#### VIP 树

- 展示 `A1-A10` 十棵根树入口卡片
- 每张卡片至少显示：
  - 根节点 ID（如 `A3`）
  - 节点总数
  - 活跃率
  - 本周新增
- 点击卡片后：
  - 自动切换到该根节点
  - 更新 URL 中的 `userId`
  - 清空搜索框，进入定位模式

#### 普通树

- 展示 `平台根节点` 单张入口卡片
- 附带总节点数、活跃率、本周新增
- 点击后直接进入普通树平台根视图

#### 设计要求

- 首次进入页面时，根节点入口必须可见
- 不应要求运营先知道某个具体用户 ID
- 这层入口优先级高于统计图表

### 8.2 搜索栏

搜索支持：

- 手机号
- 用户 ID
- 昵称
- 手机号后四位
- 用户 ID 前缀

触发策略：

- 输入 1-2 个字符即可弹出候选
- 聚焦时优先展示：
  - 最近搜索
  - 最近浏览
  - 快捷筛选结果入口（下一阶段）

搜索结果下拉中应显示：

- 头像
- 昵称
- 脱敏手机号
- 身份标签：`VIP` / `普通`
- 状态标签：`活跃 / 沉默 / 冻结 / 已出局 / VIP停收`
- 是否已入当前树
- 层级信息

如果用户未入树，下拉中仍允许展示，但点击后给出明确状态提示：

- `该用户尚未进入 VIP 树`
- `该用户尚未进入普通树`

### 8.3 顶部工具条

工具条统一包含：

- `回到根节点`
- `回到当前用户`
- `展开层数`（1-5，配合后端深度参数）
- `模式切换`（定位 / 子树）
- `刷新`
- `显示空位`（开关）
- `复制定位链接`

工具条下方可预留一行快捷标签区域（下一阶段）：

- `最近加入`
- `即将冻结`
- `沉默用户`
- `已出局`
- `升级 VIP 后停收`

这些标签点击后不应直接跳树，而应先展开一个轻量结果列表供选择。

### 8.4 树画布

#### 技术方案

继续使用 **CSS flexbox 布局** + `transform: translate()` 实现拖拽，配合 `overflow: auto` 做滚动。

理由：
- 当前实现已基于 CSS，改造成本最低
- 管理后台不需要上千节点同屏（每次只展示局部子树，50-100 个节点以内）
- 引入 `react-flow` 或 `@antv/g6` 会大幅增加包体积和学习成本，对管理后台不值得

如果未来节点数量增长到 CSS 无法承受，可在 Phase 6 中考虑引入图形库。

#### 画布能力

树画布不再只展示固定三层，而应支持：

- 当前视角居中
- 横向拖拽 / 纵向滚动（`overflow: auto` + 拖拽手势）
- 节点展开 / 收起（点击展开按钮，通过 `getChildren` 懒加载）
- 只高亮当前支路
- 空位占位显示

#### 性能约束

- **单次渲染节点上限**：100 个可见节点
- 超出时自动收起最深层，提示"点击展开更多"
- 深度展开采用**分批懒加载**：利用已有的 `getChildren` 接口，每次只加载一层子节点
- 面包屑上溯采用已有的循环保护（64 跳上限 + visited set）

#### 空位占位必须展示

空位对理解两棵树都很重要：

- VIP 树：可帮助理解推荐人节点是否已满（三叉树，最多 3 个子节点）
- 普通树：可帮助理解轮询平衡落位为什么会落到另一个父节点

空位节点渲染为虚线边框的空卡片，显示"空位"文字。

### 8.5 节点卡片

节点卡片采用统一骨架，不同页面展示不同补充字段。

统一基础字段：

- 昵称 / 用户 ID
- 状态标签
- 层级
- 自购次数
- 累计奖励
- 冻结金额
- 子节点数

统一状态标签：

- `可领奖励`（active）
- `冻结中`（frozen）
- `沉默`（silent，无消费记录）
- `已出局`（exited）
- `普通树停收`（仅普通树，已升级 VIP）
- `平台节点`（系统根节点）

#### 节点交互

- **单击**：右侧分析面板切换到该节点
- **节点上的操作按钮**（替代不可发现的双击）：
  - 🎯 "以此为中心"按钮 → 以该节点为中心重新加载树
  - 展开按钮（当子节点未加载时显示 `+N`，N 为 `childCount`）
- **右键菜单**：
  - 以此为中心
  - 查看子树
  - 跳转会员详情
  - 复制用户 ID
- **Hover**：显示 tooltip

tooltip 至少显示：

- 用户 ID
- 手机号
- 入树时间
- 当前层级
- 最近一次有效消费时间

### 8.6 URL 状态管理

树页面的导航状态需要同步到 URL，支持浏览器前进/后退和链接分享。

URL 格式：

```
/bonus/vip-tree?userId=xxx&mode=locate&depth=3
/bonus/normal-tree?userId=xxx&mode=subtree&depth=4
```

参数说明：

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `userId` | string | 无 | 当前聚焦的用户 ID |
| `mode` | `locate` \| `subtree` | `locate` | 查看模式 |
| `depth` | 1-5 | 2 | 展开深度 |

行为：

- 进入页面时读取 URL 参数，自动定位到目标用户
- 切换用户、切换模式、改变深度时自动更新 URL（使用 `replaceState` 避免堆积历史）
- 从其他页面跳转时带上 `userId` 参数即可
- "复制定位链接"按钮直接复制当前 URL

### 8.7 右侧分析面板

右侧面板统一做成 Tab 结构，不再只做基础详情。

小屏幕（宽度 < 1200px）时变为抽屉（Drawer）模式。

#### Tab 1：节点概况

展示：

- 基础身份
- 当前树位置
- 上级 / 下级数量
- 自购次数
- 当前可解锁层级
- 冻结金额
- 可用余额
- 跳转按钮：`查看会员详情 →`

#### Tab 2：奖励记录

展示最近奖励列表（查询 `RewardLedger`，筛选对应账户类型）：

- 收到奖励
- 冻结奖励
- 过期奖励
- 归平台替代记录

每条奖励要显示：

- 来源用户
- 来源订单
- 第几层奖励（从 `meta.scheme` 中提取）
- 金额
- 状态
- 原因

#### Tab 3：路径解释（Phase 5 实现）

用于回答"为什么这样分"。此 Tab 在 Phase 1-4 中显示为占位提示"即将上线"，Phase 5 实现后激活。

展示：

- 当前选中奖励的来源用户
- 消费第几次
- 应命中的祖辈层级
- 当前用户为什么收到 / 没收到
- 若没收到，是因为：
  - 未解锁
  - 已成为 VIP / 已退出普通收奖
  - 超过最大层级
  - 无祖辈
  - 归平台

- 触发其收到奖励的下级订单
- 导致解锁的关键订单

### 8.8 加载与异常态

所有异步数据加载必须实现三态：

- **加载中**：树画布显示骨架屏（3-5 个占位节点卡片），面板显示 Spin
- **空态**：搜索无结果时显示"未找到用户"；用户未入树时显示明确提示文案
- **错误态**：接口异常时显示 Result 组件 + 重试按钮

空态必须同时提供明确动作：

- `从根节点开始浏览`
- `搜索昵称 / 手机号 / ID`

不能只显示一段被动说明文字。

## 12. 数据接口调整建议

### 12.1 接口改造优先级

前端要达到上述效果，后端接口需要分阶段补充。按价值和难度排序：

| 优先级 | 接口 | 改动类型 | 难度 | 价值 | 对应 Phase |
|--------|------|----------|------|------|------------|
| **P0** | context 接口支持 `descendantDepth` 真递归 | 修改现有 | 中 | **高** | Phase 3 |
| **P0** | `GET reward-records` | 新增 | **低** | **高** | Phase 2 |
| **P1** | 节点视图模型增加字段 | 修改现有 | 低 | 中 | Phase 4 |
| **P2** | `GET path-explain` | 新增 | **高** | 中 | Phase 5 |
| **P2** | `GET placement-explain` | 新增 | **高** | 中 | Phase 5 |

**不做**的接口：
- ~~`viewMeta.positionExplain`~~：由前端用已有数据拼出说明文案，无需后端计算
- ~~`viewMeta.rewardExplainSummary`~~：同上

### 12.2 P0：奖励记录接口（Phase 2 需要）

新增两个接口，复用已有的 `RewardLedger` 数据：

```
GET /admin/bonus/vip-tree/:userId/reward-records?page=1&pageSize=20
GET /admin/bonus/normal-tree/:userId/reward-records?page=1&pageSize=20
```

返回：

```typescript
{
  items: Array<{
    id: string;
    entryType: 'FREEZE' | 'RELEASE' | 'VOID';
    amount: number;
    status: 'FROZEN' | 'AVAILABLE' | 'VOIDED';
    refType: string | null;      // ORDER, REFUND 等
    refId: string | null;        // 关联订单 ID
    sourceUserId: string | null; // 来源用户（从 meta 提取）
    layer: number | null;        // 第几层奖励（从 meta.scheme 提取）
    createdAt: string;
  }>;
  total: number;
  page: number;
  pageSize: number;
}
```

实现方式：查询 `RewardLedger` 表，按 `accountType` 区分 VIP/NORMAL，从 `meta` JSON 中提取 `sourceUserId` 和 `scheme` 信息。难度低，不需要新建表。

### 12.3 P0：context 接口支持真深度递归（Phase 3 需要）

当前 `getVipTreeContext` / `getNormalTreeContext` 虽然接受 `descendantDepth` 参数，但实际只返回 children + grandchildren（固定 2 层）。

需要改为真正按 `descendantDepth` 递归返回，同时增加安全约束：

- `descendantDepth` 最大值限制为 5
- 单次返回节点总数上限 100，超出截断并返回 `truncated: true`
- 每层子节点通过 `getChildren` 懒加载接口按需补充

### 12.4 P1：节点视图模型扩充（Phase 4 需要）

在 `buildNodeView` / `buildNormalNodeView` 中补充以下字段：

统一节点视图增加：

- `joinedTreeAt`：入树时间
- `position`：在父节点下的位置编号
- `unlockedLevel`：当前已解锁层级

普通树额外字段：

- `normalRewardEligible`：是否仍可接收普通奖励
- `frozenAt`：冻结时间（已有）
- `upgradedToVipAt`：升级 VIP 时间
- `stoppedReason`：停收原因（`UPGRADED_VIP` / `NATURAL_STOP` / `null`）

VIP 树额外字段：

- `referrerUserId`：推荐人 ID
- `entryMode`：入树方式（`REFERRAL` / `BFS_SLIDE` / `SYSTEM`）
- `exitedAt`：出局时间

### 12.5 P2：路径解释与落位解释接口（Phase 5 需要）

这两个接口需要较大的后端开发量，**仅在 Phase 5 实现**。

```
GET /admin/bonus/vip-tree/:userId/path-explain?ledgerId=xxx
GET /admin/bonus/normal-tree/:userId/path-explain?ledgerId=xxx
```

返回指定奖励记录的完整流转路径：消费用户 → 第 k 层上溯 → 命中/未命中祖辈 → 原因。

```
GET /admin/bonus/vip-tree/:userId/placement-explain
GET /admin/bonus/normal-tree/:userId/placement-explain
```

返回该用户的落位原因：入树时间、推荐人/轮询状态、为什么落在当前父节点下。

## 14. 实施顺序总结

```
✅ Phase 1: 代码重构 + 定位能力        ← 消除重复代码，打通 URL 跳转
✅ Phase 2: 右侧分析面板重构           ← Tab 化，展示奖励记录（后端新增 reward-records 接口）
✅ Phase 3: 树画布能力升级             ← 展开收起、空位、拖拽（后端 context 支持真深度）
✅ Phase 4: VIP / 普通差异化表达       ← 落位解释卡、补充字段（后端扩充节点视图）
✅ Phase 5: 奖励路径模式               ← 路径高亮、解释 Tab（后端新增 path-explain 接口）
✅ Phase 6: 体验增强                   ← 图例、截图、小地图
```

理由：

- Phase 1 先消除技术债，后续所有改动只需改一处
- Phase 2 不依赖树画布改造，但能立刻提升分析价值（价值高、风险低）
- Phase 3 树画布改造工作量最大，放在面板之后，避免阻塞核心功能
- Phase 4-5 逐步增加业务深度
- Phase 6 纯体验打磨，最后做

## 15. 验收标准

改造完成后，两个页面至少应满足以下验收标准：

### 通用验收

- 能从会员详情直接跳转并定位到用户（URL 带 `userId` 参数）
- 能清楚看到用户祖先链和子树（面包屑 + 可展开的子树）
- 能看到某个节点当前是否具备收奖资格（状态标签）
- 能查看冻结奖励及原因（奖励记录 Tab）
- 能解释奖励为什么发放 / 未发放 / 归平台（路径解释 Tab，Phase 5 后）
- 节点操作可发现（🎯 按钮、+N 按钮、右键菜单）
- URL 支持分享和浏览器前进后退
- 代码无重复（共享 TreeViewer 组件）

### VIP 页面验收

- 能解释该用户为什么落在这个位置（推荐落位解释卡）
- 能解释推荐人与 BFS 滑落的关系
- 能看清该用户在 VIP 子树中的上溯路径
- 系统根节点（A1-A10）明确标识

### 普通页面验收

- 能解释该用户为何落在当前父节点下（轮询落位解释卡）
- 能解释轮询平衡插入逻辑
- 能解释用户升级 VIP 后为什么停止接收普通奖励
- 能解释普通奖励归平台的原因
- 平台系统根节点明确标识

## 16. 结论

本次改造的关键，不是把两个页面做得更花，而是把两个页面做成真正可用于排查和解释业务的后台工具。

最终目标应是：

- `VIP 分润树`：重点解释推荐落位和 VIP 子树分润关系
- `普通分润树`：重点解释轮询平衡落位和普通树停收逻辑
- 两页共用同一套"奖励从下到上"的路径解释框架
- 两页共用同一套代码基础（TreeViewer 组件），差异通过配置表达

只有这样，这两个页面才会既"好操作"，也"看得清"。

## 18. 下一阶段建议路线

当前 6 个 Phase 已完成，后续建议不再继续扩展"树画布炫技"，而是聚焦入口层和高频运营效率。

### 18.1 推荐实施顺序

```text
P0: 根节点快捷入口 + 空状态引导 + 搜索自动补全增强
P1: 搜索历史 / 最近浏览 + 异常优先快捷筛选 + 统计概览卡片
P2: 当前视角提示 + 路径面包屑复制 + 同屏列表模式
```

### 18.2 为什么这样排

- `根节点入口` 和 `空状态引导` 直接解决"第一次进入页面不知道怎么用"
- `自动补全增强` 直接改善最高频操作路径
- `搜索历史 / 最近浏览` 非常适合后台重复排查场景
- `异常优先筛选` 比"收入 Top10"更符合运营工作流
- `统计卡片` 应服务筛选，不应变成装饰仪表盘

### 18.3 不建议优先做的内容

- 不建议优先做复杂动画或 3D/力导图
- 不建议优先做过多排行榜类卡片
- 不建议先堆很多统计图再补搜索和入口

下一轮改造仍应遵循一个原则：

- **先让用户知道从哪开始，再让用户看清楚，再让用户看得更快**
