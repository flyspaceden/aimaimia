## 一、认证与授权

### 1.1 JWT 令牌安全

| 风险 | 说明 | 防御 |
|------|------|------|
| Token 泄露 | JWT 存储在前端（AsyncStorage），被恶意 App/XSS 读取 | 使用 `expo-secure-store`（Keychain/Keystore）存储，不用 AsyncStorage |
| Token 有效期过长 | Access Token 过期时间太长，被盗后攻击窗口大 | Access Token 15min，Refresh Token 7-30天，支持强制下线 |
| Refresh Token 无限刷新 | 攻击者拿到 Refresh Token 可永久保持会话 | 设置 Refresh Token 最大续命次数（如 30 次）或绝对过期时间 |
| 多端共享 Token | 同一 Token 在多设备使用无法追踪 | 每个设备一个 Session，支持「踢掉其他设备」 |
| JWT 密钥管理 | Secret 硬编码在 `.env` 或源码中 | 使用密钥管理服务（KMS），定期轮换 Secret |
| 算法混淆攻击 | 攻击者将 `alg` 从 RS256 改为 HS256，用公钥签名 | 服务端强制指定算法，忽略 Header 中的 `alg` |
| Token 注入跨端 | 用买家 Token 调用管理端 API | 已做三套独立 Secret（`JWT_SECRET` / `ADMIN_JWT_SECRET` / `SELLER_JWT_SECRET`），确保每个 Guard 只校验对应 Secret ✅ |

**当前代码状态（2026-02-25）**: `部分实现`。买家端、卖家端、管理员端 JWT Strategy 均已补 `Session` 有效性检查（新版 token 使用 `sessionId` 精确匹配；旧 token 兼容降级为主体维度会话检查）✅。卖家端/管理员端也会校验主体状态（禁用后拒绝访问）✅。买家 Access Token 默认配置已收紧为 `15m` ✅；卖家/管理员默认仍为 `8h`，与文档建议基线存在差距（`建议未落地`）。三端 Session 均已补 `absoluteExpiresAt`（90天最大续期上限），首次登录时生成、Refresh 时继承旧值（不重置），Refresh 时同时检查滑动过期和绝对过期 ✅。

### 1.2 验证码安全

| 风险 | 说明 | 防御 |
|------|------|------|
| 验证码爆破 | 6 位数字只有 100 万种，可暴力枚举 | 同一手机号 5 次错误后锁定 15 分钟；验证码有效期 5 分钟；OTP 使用 `crypto.randomInt` 密码学安全随机生成 ✅ |
| 短信轰炸 | 攻击者用脚本对任意手机号反复发送验证码 | IP 维度限频（每 IP 每分钟 5 条）+ 手机号维度限频（每号每天 10 条）+ 图形验证码/滑块 |
| 验证码重放 | 同一验证码被多次使用 | 买家端与卖家端已做 CAS 原子消费（S07）✅；管理员端如新增 OTP 流程也应复用同样模式 |
| 语音验证码绕过 | 只防了 SMS 通道，语音通道不防 | 统一对所有验证码通道做限频 |

### 1.3 OAuth / 第三方登录

| 风险 | 说明 | 防御 |
|------|------|------|
| 微信 openId 伪造 | 攻击者不走微信 OAuth 流程，直接传假 openId | 必须用 `code` 换取 `access_token`，再用 `access_token` 获取用户信息，不信任客户端传来的 openId |
| State 参数缺失 | OAuth 回调无 state 参数，容易被 CSRF | 生成随机 state，回调时严格校验 |
| 账号关联劫持 | 攻击者先绑定受害者手机号，再用 OAuth 登录接管 | 绑定第三方登录前必须验证当前用户身份（短信验证码） |

### 1.4 权限控制

| 风险 | 说明 | 防御 |
|------|------|------|
| 越权访问 | 用户 A 可查看/操作用户 B 的订单/地址/钱包 | 所有数据查询必须带 `userId` 或 `companyId` 过滤（纵深防御） |
| 权限提升 | 普通用户调用管理端 API | 已做 Guard 隔离（`AdminAuthGuard` / `SellerAuthGuard`），确保无遗漏 ✅ |
| RBAC 绕过 | 管理员权限检查遗漏某些端点 | 端点级权限白名单（默认拒绝），而非黑名单 |
| 超级管理员滥用 | 超管绕过所有权限且无审计 | 超管操作也必须记录审计日志，关键操作（删除用户、修改余额）需二次验证 |
| 高危操作误操作/盗用 | 提现审核、批量退款、分润规则修改被盗号后直接执行 | 高危操作要求二次认证（MFA/短信/OTP）+ 双人审批（四眼原则） |
| 卖家跨店访问 | 商户 A 的员工操作商户 B 的商品/订单 | 所有卖家端 API 强制 `companyId` 过滤（已有 `@CurrentSeller()` 装饰器）✅ |

**当前代码状态（2026-02-25）**: `部分实现`。管理端鉴权已做端隔离（独立 Secret + `AdminAuthGuard`）✅；`PermissionGuard` 已改为数据库实时读取管理员当前角色/权限并覆盖 JWT 内缓存权限，角色变更后无需等待 Access Token 过期即可生效 ✅。仍待补：高危操作二次认证（MFA/短信/OTP）与双人审批（`建议未落地`）。

## 二、资金与金融安全

### 2.1 支付安全

| 风险 | 说明 | 防御 |
|------|------|------|
| 0 元购 | 篡改前端传入的金额创建订单 | 后端完全忽略前端传入的价格，只按 SKU 数据库价格计算 ✅ |
| 支付回调伪造 | 攻击者模拟微信/支付宝回调 | IP 白名单 + 签名验证（已修复 S04）✅ |
| 回调重放攻击 | 合法的支付回调被截获后重放 | 回调处理幂等（同一 `merchantOrderNo` 只处理一次）+ timestamp 窗口校验（5 分钟内）；回调事务已补 Serializable 隔离级别 + P2034 重试 ✅ |
| 账务对账缺失 | 支付成功/退款成功/分润回滚与内部账本长期偏差未发现 | 建立日对账任务（支付、退款、分润、提现四类）+ 差异告警 + 人工复核流程（内部账务骨架版已完成：日报聚合 + 提现/奖励一致性检查 + 日志告警；渠道对账单接入待补） |
| 金额溢出 | 超大金额导致浮点精度丢失 | 数据库用 `Decimal` 类型，或统一使用分为单位（Int），计算时避免浮点 |
| 退款超额 | 多次部分退款总额超过订单金额 | 事务内校验累计退款 ≤ 订单金额（已修复 S19）✅ |
| 负数金额 | 传入负数退款金额导致余额增加 | DTO 层 `@Min(0)` 校验 + 数据库 CHECK 约束 |
| 支付渠道降级 | 微信支付不可用时的处理 | 多支付渠道自动切换 + 超时检测（30 秒无回调则标记为待确认） |

### 2.2 奖励系统安全

| 风险 | 说明 | 防御 |
|------|------|------|
| 奖励双重使用 | 并发请求用同一奖励下两个订单 | Serializable + CAS（已修复 S02）✅ |
| 奖励金额篡改 | 前端传入奖励抵扣金额不等于实际奖励面额 | 后端忽略前端传入的金额，从数据库读取奖励面额 |
| 奖励无限生成 | 攻击者批量领取奖励 | 奖励发放需要触发条件（购买/分润），不能凭空领取；防止重复发放（幂等） |
| 过期奖励复活 | 修改客户端时间使过期奖励可用 | 过期检查用服务端时间（`new Date()`），不信任客户端 |
| 奖励嗅探 | 枚举他人奖励 ID 尝试使用 | 奖励使用时必须校验 `userId` 归属 |
| 奖励金额精度 | 0.1 + 0.2 ≠ 0.3 浮点问题 | 计算时用整数分，展示时除 100（或使用 `Decimal` 库） |

### 2.3 分润/奖金系统安全（核心风险区）

| 风险 | 说明 | 防御 |
|------|------|------|
| 分润重复发放 | 同一订单触发多次分润 | 分润前检查 `RewardAllocation` 表是否已存在该订单记录（S08 补偿机制已有去重）✅ |
| 分润金额计算错误 | 百分比规则变更导致已完成订单被重新计算 | 分润比例在订单创建时快照冻结，不受后续规则变更影响 |
| VIP 树篡改 | 攻击者修改自己的上级节点获取更多分润 | VIP 树节点 `parentId` 只在创建时设置，后续不可修改；API 不暴露修改接口 |
| 分润资金池透支 | 分润总额超过平台实际收入 | 每笔分润计算后校验：`sum(所有层级分润) ≤ 订单金额 × 最大分润比例`；设置总分润上限 |
| 虚假交易刷分润 | 自买自卖/关联交易赚分润 | 规则引擎检测：同设备/同 IP/同地址的买卖关系；设置新用户分润冷却期 |
| 提现恶意攻击 | 并发大量提现请求掏空余额 | 提现申请链路使用 Serializable 隔离 + `balance >= amount` 校验 + 数据库 CHECK ✅；提现审批/拒绝链路补状态 CAS（`REQUESTED -> APPROVED/REJECTED`）+ 事务隔离/并发保护 ✅ |
| 提现拒绝服务 | 大量小额提现（¥0.01）消耗运营资源 | 最低提现金额 ¥1 + 每日最多 3 次提现限制（Serializable 事务内检查，防并发绕过）✅ |
| 分润延迟攻击 | 确认收货后立即提现，再退款 | 分润冻结期（如 7 天），冻结期内退款自动回滚分润 |
| 层级关系环路 | 数据异常导致 A→B→C→A 循环分润 | 分润遍历设置最大深度限制 + 循环检测（已访问节点 Set）✅；BFS 遍历补 10000 次迭代上限 + 20 层最大树深度限制 + 系统根节点搜索补 20 节点上限 ✅ |
| 退款后分润未回收 | 退款后上级已提现的分润无法回收 | 回滚写入负向流水（已确认 S09）✅，已提现的标记为待追回 |

**当前代码状态（2026-02-25）**: `部分实现`。买家”发起提现”链路已用事务内二次余额校验（并发安全基础较好）✅；管理员”审批/拒绝提现”链路已补状态 CAS（`where status='REQUESTED'`）与显式 `Serializable`，降低并发重复处理风险 ✅。提现已补最低金额限制（¥1）和每日次数限制（3次，Serializable 事务内检查防并发绕过）✅。VIP 树方面，后台 VIP 树 breadcrumb 构建已补循环保护（visited Set + 最大跳数）✅，分润路径递归 CTE 也已补 path 去环（防脏数据环路导致错误祖先）✅，BFS 滑落遍历已补 10000 次迭代上限 + 20 层最大树深度限制 + 系统根节点搜索补 20 节点上限 ✅。仍待完善项主要是业务侧反作弊策略（关联交易/刷分润识别）（`建议未落地`）。

> 详细问题清单见 `tofix-safe.md`（截至 2026-02-25：20 个并发与一致性问题已完成修复）

以下为延伸场景：

| 风险 | 说明 | 防御 |
|------|------|------|
| 库存超卖 | 并发下单扣减同一 SKU | Serializable + CAS + CHECK（已修复 S01/S03）✅ |
| 订单状态机竞态 | 支付/取消/退款同时到达 | CAS `updateMany` 锁定来源状态 ✅ |
| 分布式锁缺失 | 多实例部署时内存锁失效 | 未来多实例时需要 Redis 分布式锁（`SETNX`）或 Advisory Lock |
| 定时任务重复执行 | 多实例部署时 Cron 在每个实例都跑 | 使用 `@nestjs/schedule` + 分布式互斥（Redis 锁 / DB 行锁 / Advisory Lock）；对账 Cron 已补 DB 行锁互斥基础版 ✅ |
| 批处理内存溢出 | 定时扫描大量数据导致 OOM | 定时批处理查询加 `take: 200` 分页限制（订单过期/自动确认）✅ |
| 数据库死锁 | 多个 Serializable 事务互相等待 | 统一锁顺序（按 ID 排序加锁）+ 死锁重试（已做 P2034 重试）✅ |
| 长事务阻塞 | Serializable 事务持有锁时间过长 | 事务内避免外部调用（HTTP/Redis），控制事务执行时间 < 5 秒 |

## 四、API 安全

### 4.1 输入验证

| 风险 | 说明 | 防御 |
|------|------|------|
| SQL 注入 | 恶意 SQL 通过参数注入 | Prisma ORM 已做参数化查询（天然防御）；避免 `$queryRawUnsafe` |
| NoSQL 注入 | JSON 对象注入到查询条件 | Prisma 不是 NoSQL，但 `Json` 类型字段（`rules`/`meta`）读取后需类型校验 |
| XSS（跨站脚本） | 商品标题/评价/留言中嵌入 `<script>` | 输出时 HTML 转义；React 默认转义 JSX；富文本用 `DOMPurify` 过滤 |
| SSRF（服务端请求伪造） | 用户提供 URL（头像/图片）被服务端请求 | URL 白名单（只允许 OSS 域名）；不直接 fetch 用户提供的 URL |
| 路径穿越 | 文件上传路径 `../../etc/passwd` | 文件名 UUID 重命名；上传目录白名单；不使用用户提供的文件名 |
| 类型混淆 | `quantity: "99999999999999"` 字符串绕过 `@IsNumber` | DTO 使用 `@Type(() => Number)` + `enableImplicitConversion` |
| 数组炸弹 | `items` 数组传入 10 万个元素耗尽内存 | `@ArrayMaxSize(100)` 限制数组长度 |
| 深度嵌套 | 超深 JSON 嵌套导致解析 OOM | 全局中间件限制 `body` 大小（`app.use(json({ limit: '1mb' }))`） |

**当前代码状态（2026-02-25）**: `部分实现`。已启用全局 `ValidationPipe` 白名单/隐式转换/拒绝额外字段，并已对订单、售后、认证、AI 等高频入口补充数组长度/字符串长度/格式校验与 Token 长度约束。其余 DTO（管理端复杂查询/批量接口等）仍需持续覆盖（`建议未落地`）。

| `POST /orders` | 5 次/分钟/用户 | 防刷单 |

| `POST /upload` | 20 次/分钟/用户 | 防存储滥用 |
| 全局 | 100 次/分钟/IP | 防 DDoS |

## 五、数据保护与隐私

### 5.1 敏感数据

| 数据 | 风险 | 防御 |
|------|------|------|
| 手机号 | 泄露后可被社工/垃圾短信 | 数据库存明文（查询需要），API 返回时脱敏（`138****8888`）；日志中脱敏 |
| 身份证号 | 实名认证场景（企业入驻） | AES-256 加密存储，只在必要时解密；API 不返回完整身份证号 |
| 银行卡号 | 提现绑定的银行卡 | 只存后 4 位用于展示；完整卡号交给支付渠道处理 |
| 收货地址 | 精确到门牌号的隐私信息 | 物流面单脱敏（隐藏部分地址）；API 只返回给地址所有者 |
| 密码 | 管理员/卖家登录密码 | bcrypt 哈希（cost ≥ 10）；不存明文；不返回 hash |
| JWT Secret | 泄露后可伪造任意用户 Token | 环境变量管理；不提交到 Git；定期轮换 |
| 支付密钥 | 微信/支付宝商户私钥 | 服务端环境变量；不记入日志；文件权限 600 |

### 5.2 日志安全

| 风险 | 说明 | 防御 |
|------|------|------|
| 敏感信息泄露 | 日志中打印完整手机号、Token、密码、身份证号 | 日志中间件自动脱敏：正则匹配手机号/身份证号/Token/密码并替换 ✅ |
| 日志注入 | 用户输入中包含换行符伪造日志 | 日志输出时转义换行符和控制字符 |

## 十二、安全监控与应急

### 12.1 需要监控的关键指标
