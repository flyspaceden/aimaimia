# 管理后台前端整改方案

## 1. 文档目标

本文档用于指导管理后台前端整改，解决以下问题：

- 左侧目录层级混乱，页面归属不清
- 系统配置、VIP 配置、普通奖励配置边界不清
- 部分页面的信息密度失衡，主次不分
- 部分页面存在右侧内容溢出、表格过宽、详情堆叠的问题
- 存在已开发但未挂菜单/未挂路由的页面，信息架构不完整

本文档基于当前 `admin/` 代码逐页审阅整理，不是抽象建议，而是面向现有代码结构的整改计划。

### 金额可见性边界

配送管理后台的金额展示统一为四个标签：

- `买家金额`
- `商家供货`
- `商家应结`
- `平台差额`

要求：

- API 层统一保持 `cents` 单位，页面只通过 `formatMoney(cents)` 格式化。
- 商家应结 = 商家供货 + 运费分摊。
- 平台差额 = 买家金额 - 商家应结。
- 商家供货、运费分摊优先取后端明确字段，如 `supplyAmountCents`、`shippingFeeShareCents`；缺字段时只展示可得项。
- 平台差额只在前端能从现有字段确定时显示；不能确定时显示 `-`。
- 结算页只展示供货额、应结额、已结额，不在该页暴露平台定价策略。

## 2. 现状诊断

### 2.1 菜单层面

当前菜单定义在 `admin/src/layouts/AdminLayout.tsx`，主要问题：

- 一级菜单按“技术模块”堆叠，不按“运营任务流”组织
- `系统配置` 混入 VIP 参数、抽奖参数、定价参数、订单参数，边界失控
- `会员/奖励` 下同时放会员、提现、树、普通系统配置，但缺少 VIP 配置页
- `退款仲裁` 页面已存在，但没有挂路由/菜单
- `broadcast-window` 已废弃但仍保留路由，容易形成隐性入口

### 2.2 页面层面

当前页面大体可分为六类：

- 总览：工作台
- 用户奖励：用户、会员、提现、奖励树、奖励配置
- 商家商品：企业、商家商品、奖励商品、溯源
- 交易售后：订单、换货、退款、运费
- 运营活动：红包、抽奖、团购
- 客服中心：消息公告、客服对话、工单、FAQ、快捷入口、快捷回复、数据看板
- 系统管理：配置、管理员、角色、审计

现在的问题不是页面数量多，而是页面职责边界不统一。

### 2.3 布局层面

当前多个页面使用：

- `div style={{ padding: 24 }}`
- 大宽度表格但未统一配置 `scroll.x`
- 复杂页直接使用多列布局，缺少页面最大宽度和响应式收缩策略

因此会出现：

- 红包活动、抽奖管理、企业详情、树结构页在窄屏或侧边栏展开后易拥挤
- 表格列过多时挤压右侧内容
- 详情页信息块过多，阅读顺序不稳定

### 2.4 权限与角色视图层面

当前管理后台缺少多角色视图的考量：

- 所有管理员看到的菜单和页面内容完全一致，没有按角色差异化
- 后端已有 `PermissionGuard` + `@Permission()` 装饰器，但前端未充分利用 `PermissionGate` 组件做按钮级控制
- 超级管理员、运营、客服、财务等角色的菜单可见范围未定义
- 高频列表页缺少批量操作能力，影响运营效率（批量审核、批量导出、批量发货等属于 P2/P3 扩展项，需后端配套接口、权限模型和导出任务系统，不在前端整改范围内，单独排期）

## 3. 整改原则

### 3.1 信息架构原则

- 一级菜单按业务任务组织，不按技术模块组织
- 列表页进入菜单，详情页默认不进入菜单
- “配置类页面”必须按业务域拆分，不能继续聚合到一个万能配置页
- 同一业务域内，列表页、详情页、配置页命名风格统一

### 3.2 页面设计原则

- 每个页面只解决一个核心任务
- 页面头部固定为：标题 + 简介 + 主操作
- 列表页优先表格，详情页优先卡片分组
- 表格默认考虑横向滚动和列优先级
- 大页面优先拆 Tab 或拆子页面，不在单页中塞入过多业务

### 3.3 交互反馈原则

- 涉及资金、状态变更的操作（提现审核、封禁用户、退款仲裁、配置变更）必须二次确认弹窗
- 普通 CRUD 操作使用 `message.success` 轻提示，不弹窗
- 列表页操作成功后自动刷新当前页，不跳转
- 乐观更新仅限低风险操作（如标记已读），资金/审核类操作必须等待服务端确认
- 高频列表页（用户、订单、商品、企业、提现）后续应支持批量操作（P2/P3 扩展项，需后端配套批量接口和导出任务系统）

### 3.4 实施原则

- 先调整菜单和页面归位，再做 UI 优化
- 先做配置边界拆分，再做内容精修
- 先修高频页面，再修低频页面

### 6.2 工作台 ✅ 已完成（2026-07-04 V2 运营工作台；侧边栏徽标待做）

- 平台总览

**2026-07-04 V2 状态**

- 已新增 `GET /admin/stats/operations-overview` 聚合接口，工作台不再在前端散落调用商品、企业、提现、售后和奖励多个列表接口来拼待办数。
- 今日经营口径统一使用有效支付时间，并按 UTC+8 中国自然日取边界；优先使用 `paidAt`，仅对已付款、已发货、已签收、已完成但历史 `paidAt` 缺失的数据回退到 `createdAt`，避免工作台最近订单有数据但 GMV/订单数/销售趋势显示为 0；今日 GMV、订单数、客单价和支付渠道分布来自订单/支付真实表。
- 销售趋势与抽奖今日数据均按中国自然日口径统计，避免服务器时区与运营日历不一致；销售趋势 SQL 使用 `TO_CHAR(..., 'YYYY-MM-DD')` 固定返回日期字符串，避免 Postgres `DATE()` 经 Prisma 返回 Date 对象后与前端日期键不匹配；抽奖使用 `LotteryRecord.drawDate`。
- 红包有效活动统计同时校验 `status=ACTIVE`、开始时间和结束时间窗口；商品审核排除草稿商品。
- 页面分为「待办中心 / 今日经营 / 销售趋势 / 最近订单 / 奖励资金 / 数字资产概览 / 活动增长 / 经营脉搏 / 处理优先级」；待办项按数量和风险排序，空状态显示“暂无待办”，最近订单使用紧凑列表避免表格在窄屏裁切。
- 提现失败不列入待办中心：该状态由系统判定，管理员不能在工作台直接处理；后续只在提现明细或资金风险视图中查看。
- 图表区补齐「订单结构 / 支付渠道分布 / 待办分布 / 资金结构 / 活动转化」5 张图，全部从 `operations-overview` 派生，不新增前端 mock 统计；资金结构只统计真实奖励资金与提现处理中金额，不包含数字资产；资金结构饼图不使用外置标签，tooltip 使用中文“金额”，金额和占比放入图下方全宽明细列表，并补充每项资金含义，避免小扇区文字重叠或简称难懂。
- 数字资产是基于累计消费形成的虚拟权益指标，不属于真实资金结构；工作台单独放在「数字资产概览」中展示数字资产值、资产账户数、今日新增资产和对应累计消费，并保留跳转 `/digital-assets` 的入口。

## 9. 前端实现注意事项

### 9.1 菜单实现

- 一级菜单控制在 5 到 6 组
- 详情页不注册菜单 route
- `ProLayout` 建议从 `mix` 改为纯侧边结构，减少顶部混杂感

### 9.2 页面容器规范

统一建议：

- 列表页：`padding: 24`
- 复杂详情页：`maxWidth` 控制主体宽度
- 表格页默认补 `scroll.x`
- 抽屉和弹窗宽度统一分级

### 9.3 表格规范

- 所有宽表统一加横向滚动
- 操作列放最后并固定宽度
- 可稳定服务端排序的数字、金额、时间、计数类列优先开启表头排序；排序字段必须由后端白名单接收，不能直接透传任意列名
- 用户与奖励域的宽表使用通用列宽拖拽组件，拖拽后的列宽写入浏览器本地存储；列宽调整只改变当前管理员当前浏览器的显示，不改变业务数据
- 列优先级分为：
  - 必看列
  - 辅助列
  - 次要列

### 9.4 配置页规范

- 配置页一律按业务域拆分
- 每组参数都要有说明
- 比例类参数都要有合计校验
- 版本历史和回滚保留

### 9.5 搜索与筛选规范

- 筛选器统一放在表格上方（ProTable `search` 区域），不使用侧边抽屉
- 筛选状态建议同步到 URL query params（`react-router` `useSearchParams`），刷新页面不丢失筛选条件；高频列表页默认遵循，低频页视复杂度决定
- 时间范围筛选默认值（建议，各页面可按业务需要调整）：
  - 订单/发放记录/使用记录：最近 7 天
  - 审计日志：最近 24 小时
  - 统计页：最近 30 天
  - 用户/企业/商品列表：不限
- 模糊搜索输入防抖 300ms，最小输入长度 2 字符；**精确 ID/单号搜索（订单号、用户 ID、退款单号等）不设最小长度限制**
- 每个列表页的筛选项按使用频率排序，高频筛选项默认展示，低频筛选项收入"更多筛选"

### 9.6 响应式断点规范

统一断点定义（与 Ant Design 保持一致）：

| 断点 | 宽度 | 行为 |
|------|------|------|
| `xs` | < 576px | 不支持（管理后台最低 768px） |
| `sm` | ≥ 576px | 侧边栏自动折叠为图标模式 |
| `md` | ≥ 768px | 管理后台最低支持宽度 |
| `lg` | ≥ 992px | 侧边栏展开，详情页单列布局 |
| `xl` | ≥ 1200px | 详情页双列布局，表格完整展示 |
| `xxl` | ≥ 1600px | 页面主体 `maxWidth: 1400px` 居中 |

- ProLayout `breakpoint` 设为 `lg`（992px 以下自动折叠侧边栏）
- 表格横向滚动触发：当列总宽超过容器宽度时自动启用 `scroll.x`
- 树可视化页面（VIP/普通奖励树）：`lg` 以下详情面板自动折叠为抽屉模式

### 9.7 操作反馈规范

| 操作类型 | 确认方式 | 成功反馈 |
|---------|---------|---------|
| 普通 CRUD（创建/编辑/删除） | 删除需确认弹窗，创建/编辑不需要 | `message.success` |
| 审核操作（通过/拒绝） | 二次确认弹窗 + 原因输入 | `message.success` + 列表自动刷新 |
| 资金操作（提现审核） | 二次确认弹窗 + 金额复核 | `notification.success` + 列表自动刷新 |
| 封禁/解封用户 | 二次确认弹窗 + 原因输入 | `message.success` |
| 配置变更 | 二次确认弹窗 + 变更对比展示 | `notification.success` + 显示"已生效" |
| 回滚操作 | 二次确认弹窗 + 影响范围提示 | `notification.success` |
| 批量操作 | 确认弹窗 + 显示影响数量 | `message.success` + 显示成功/失败数量 |

### 9.8 空状态与错误状态规范

- **表格空状态**区分两种情况：
  - 无数据：展示空状态插图 + "暂无数据" + 引导操作按钮（如"创建第一个活动"）
  - 筛选无结果：展示"未找到匹配结果" + "清除筛选条件"按钮
- **接口异常**：
  - 网络错误：展示重试按钮 + 错误描述
  - 403 权限不足：展示"无权限访问此页面，请联系管理员"
  - 404 资源不存在：展示"资源已被删除或不存在" + 返回列表按钮
  - 500 服务端错误：展示通用错误页 + 重试按钮
- **权限不足时的 UI 处理**：
  - 菜单级：无权限的菜单项不显示（通过 `PermissionGate` 控制）
  - 按钮级：无权限的操作按钮隐藏或置灰 + tooltip 提示"无权限"
  - 页面级：直接访问无权限路由时跳转 403 页面

### 9.9 共享组件复用清单

以下结构在多个页面出现，应抽取为共享组件：

| 共享组件 | 使用页面 | 说明 |
|---------|---------|------|
| `TreeVisualization` | VIP奖励树、普通奖励树 | 搜索 + 树渲染 + 详情面板，通过 props 切换主题色和数据源 |
| `AuditReviewModal` | 企业审核、商品审核、提现审核、退款/换货仲裁 | 统一的审核弹窗：通过/拒绝 + 原因输入 + 凭证预览 |
| `ConfigFormGroup` | VIP配置、普通配置、平台设置 | 配置分组表单：标题 + 说明 + 字段列表 + 合计校验 |
| `StatusFilter` | 订单、商品、企业、提现、红包 | 快捷状态筛选 Tag 组，点击切换筛选 |
| `BatchActionBar` | 用户、订单、商品、企业、提现 | 批量选择后的浮动操作栏：已选 N 项 + 批量操作按钮 |
| `AuditTrail` | 会员详情、企业详情、订单详情、配置页 | "查看操作记录"入口，展示审计日志时间线 |

### 9.10 角色与菜单权限映射

不同角色的菜单可见范围建议：

| 菜单组 | 超级管理员 | 运营 | 客服 | 财务 |
|--------|-----------|------|------|------|
| 工作台 | ✅ | ✅ | ✅ | ✅ |
| 用户与奖励 | ✅ | ✅ 用户/会员只读 | ✅ 用户只读 | ✅ 提现审核 |
| 商家与商品 | ✅ | ✅ | ❌ | ❌ |
| 交易与售后 | ✅ | ✅ 订单只读 | ✅ | ✅ 退款只读 |
| 运营活动 | ✅ | ✅ | ❌ | ❌ |
| 系统管理 | ✅ | ❌ | ❌ | ❌ |

- 菜单可见性由前端 `PermissionGate` + 后端 `@Permission()` 双重控制
- 超级管理员绕过所有前端权限检查
- 工作台的待办卡片按角色过滤，每个角色只看到自己负责的待办
