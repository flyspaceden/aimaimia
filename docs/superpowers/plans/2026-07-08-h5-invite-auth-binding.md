# H5 Invite Auth Binding Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build an H5 scan flow where an invitee can use phone + SMS code to login or auto-register, bind to the inviter immediately if no existing relation exists, and later see the same relation in the App by logging in with the same phone.

**Status (2026-07-08):** Code implemented on branch `codex/h5-invite-auth-design`; backend/unit tests, website build, App TypeScript, and referral-link script tests pass. Remaining: staging deployment and normal/VIP真机链路验收.

**Architecture:** Add a backend `invite-h5` module for invite-code resolution, landing statistics, and post-auth binding orchestration. Auth owns phone/SMS login and token issuance, then delegates invite binding to the existing normal-share and VIP referral services so current business rules remain authoritative. The website adds `/invite/:code`, and the buyer App changes newly generated invite QR/share links to that H5 route while preserving old `/s/:code` and `/r/:code` compatibility.

**Tech Stack:** NestJS + Prisma + PostgreSQL, Jest, React/Vite website, React Native/Expo buyer App, TanStack Query, existing Result API envelope.

## Non-Negotiable Boundaries

- Do not create a second visible invite code per user.
- Normal inviters use `NormalShareProfile.code`.
- VIP inviters use `MemberProfile.referralCode`, but only when `MemberProfile.tier = VIP`.
- A normal user's hidden `MemberProfile.referralCode` must not be accepted by `/invite/:code`.
- H5 must not display inviter nickname, phone, buyer number, avatar, or team information.
- SMS verification is required for real login/register.
- If the invitee is already bound to any other inviter, login still succeeds and binding is not overwritten.
- Existing `/s/:code` and `/r/:code` must keep working for published old material.
- Binding writes must continue through `NormalShareService.bind()` and `BonusService.useReferralCode()`; do not duplicate their business rules.

const code = rawCode.trim().toUpperCase();
const normal = await prisma.normalShareProfile.findUnique({
  where: { code },
  select: { userId: true, status: true },
});

- `vip.tier === 'VIP'` and no normal match: `VIP_REFERRAL`.
- `normal` exists and `vip.tier === 'VIP'`: `CONFLICT`.
- all other cases: `INVALID`.

- [ ] **Step 5: Implement bind status mapping**

Before calling underlying bind service:

1. Resolve code.
2. If invalid/conflict, return `INVALID_CODE`.
3. If `inviterUserId === userId`, return `SELF_INVITE`.
4. Read current effective inviter using `referralLink`, active `normalShareBinding`, and `memberProfile.inviterUserId`.
5. If existing inviter differs, return `ALREADY_BOUND_OTHER`.
6. If existing inviter is same, return `ALREADY_BOUND_SAME`.
7. Call:
   - normal: `NormalShareService.bind(userId, { code, source: 'LANDING' })`
   - VIP: `BonusService.useReferralCode(userId, code)`

Map known exceptions:

| Message contains | Status |
|---|---|
| `已绑定` / `不能更换` | `ALREADY_BOUND_OTHER` |
| `自己的` | `SELF_INVITE` |
| `VIP 用户不使用普通分享码` / `已加入 VIP 团队` | `NOT_ELIGIBLE` |
| `无效` / `停用` / `不可用` | `INVALID_CODE` |
| other | `ERROR` |

async inviteLogin(dto: InviteLoginDto) {
  await this.enforceLoginAttemptRateLimit('PHONE', dto.phone);
  const session = await this.loginOrAutoRegisterByPhoneCode({
    phone: dto.phone,
    code: dto.code,

export async function apiPostJson<T>(path: string, body: unknown): Promise<T> {
  const res = await fetch(`${BASE_URL}${path}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(body),
  });
  if (!res.ok) {
    throw new Error(res.status === 429 ? '请求过于频繁，请稍后再试' : '服务器错误，请稍后重试');
  }
  const payload: ApiResponse<T> = await res.json();
  if (!payload.ok) throw new Error(payload.error || '请求失败');
  return payload.data;
}
