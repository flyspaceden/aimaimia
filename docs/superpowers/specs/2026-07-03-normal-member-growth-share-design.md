# 普通会员增长与普通分享码设计方案

> 状态：设计已确认，待实施计划
> 创建时间：2026-07-03
> 适用范围：买家 App / 官网落地页 / 后端 / 管理后台 / 平台红包 / 任务签到 / 普通奖励钱包
>
> **For agentic workers:** 本文档是“普通会员增长与普通分享码”的权威来源。本功能独立于现有 VIP 推荐码、VIP 树、普通分润树、团购分享码、平台红包和消费积分系统，但会复用其中已存在的能力。

## 1. 背景

当前 App 已有团购、抽奖、新人红包、首单红包、普通奖励钱包、普通树、任务、签到、积分、成长值和消费积分抵扣。但这些能力在买家 App 中没有形成面向普通用户的完整增长路径。

现有推荐码链路是 VIP 专属：

- `GET /bonus/member` 只在用户为 VIP 时返回 `referralCode`。
- `useReferralCode` 和延迟深链只允许 VIP 用户作为推荐人。
- 买家 App “我的推荐码”页面对非 VIP 用户只展示绑定推荐人和购买 VIP 引导。
- VIP 页明确提示“推荐码仅 VIP 用户可展示和分享”。

这会造成一个关键问题：普通用户可以消费、签到、做任务、获得普通奖励，但不能拥有一个可持续传播的“普通分享身份”。团购和抽奖是活动型入口，能制造短期转化，但不能替代每个普通用户都能发起的 App 级拉新机制。

本设计的目标是降低普通用户参与门槛，让新用户先进入“注册、领红包、首单、复购、分享”的路径。VIP 保持高阶权益和收益加速器定位，不再作为基础推广能力的入口门槛。

## 2. 核心结论

新增一套独立的“普通分享码 / 新人福利码”体系。

普通分享码用于普通用户拉新，不复用现有 VIP 推荐码字段，不复用 `/r/{code}` VIP 推荐链接，不接入 VIP 树，也不改变普通树自动入树算法。

第一版增长组合为：

1. 普通分享码：所有正常买家用户可生成和分享。
2. 新人福利路径：新用户通过普通分享码注册后领取新人红包。
3. 首单后奖励邀请人：被邀请人完成首笔普通商品订单并确认收货后，邀请人获得平台配置的奖励。
4. 普通会员成长中心：前置展示签到、任务、成长值、积分、普通奖励和消费抵扣。
5. VIP 转化桥：用户先看到普通会员价值，再提示升级 VIP 后可提高抵扣比例、获得 VIP 礼包、数字资产和 VIP 推荐权益。

推荐第一版奖励口径：

```text
新人：注册送新人红包，可叠加首单红包规则
邀请人：好友首单确认收货后，发普通邀请奖励
奖励类型：优先平台红包、积分、成长值；不默认发现金
```

## 3. 系统边界

| 系统 | 本设计关系 |
|---|---|
| VIP 推荐码 | 保持 VIP 专属，不下放，不复用字段 |
| VIP 树 | 不接入普通分享码 |
| 普通树 | 不按邀请关系插入，仍按首单后自动平衡入树 |
| 团购分享码 | 保持 `/gb/{code}` 独立链路，不复用 |
| 平台红包 Coupon | 复用新人、首单、邀请、分享、签到等触发能力 |
| 消费积分 Reward | 复用普通奖励钱包与普通商品抵扣能力 |
| 任务签到 | 复用现有 TaskService 和 CheckInService，补足 App 前台入口 |
| 数字资产 | 不下放给普通用户，作为 VIP 转化桥 |

## 4. 非目标

第一版不做以下内容：

- 不给普通用户开放现有 VIP 推荐码。
- 不允许普通用户推荐他人购买 VIP 后获得 VIP 推荐收益。
- 不按普通分享关系改变普通树节点位置。
- 不做多级邀请、团队关系、排行榜或层级展示。
- 不把普通分享码用于团购付款。
- 不新增现金提现型普通邀请奖励。
- 不改变团购、抽奖、VIP 礼包、数字资产和现有订单支付主流程。
- 不做复杂反作弊风控模型，第一版只做基础防刷限制和奖励延迟发放。

## 5. 已确认设计

### 5.1 普通分享码

每个正常买家用户可拥有一个普通分享码。

建议新增独立概念：

```text
NormalShareCode
```

不要复用 `MemberProfile.referralCode`，因为该字段当前已经被定义为 VIP 推荐码，并且后端依赖它判断 VIP 推荐关系。

普通分享码可使用买家公开编号 `buyerNo` 作为展示身份，也可新增随机短码。第一版推荐使用独立随机短码，避免公开编号承担增长归因和风控状态。

建议链接：

```text
https://app.ai-maimai.com/s/{code}
```

`/s/{code}` 表示普通分享落地页，和 VIP `/r/{code}`、团购 `/gb/{code}` 分离。

### 5.2 归因规则

用户打开 `/s/{code}` 后：

1. 官网落地页展示邀请人昵称或脱敏信息、App 下载入口和新人福利说明。
2. 未安装 App 时，写入 Cookie 与服务端延迟归因记录。
3. 已安装 App 时，Deep Link 打开 App 并保存 pending normal share code。
4. 用户注册或登录后，尝试绑定普通分享关系。

绑定限制：

- 不能绑定自己的普通分享码。
- 已绑定普通分享关系后不允许频繁换绑。
- 已有 VIP 推荐关系不被普通分享码覆盖。
- 普通分享关系不影响 VIP 购买时的 VIP 推荐人确认流程。
- 注销、封禁、非 ACTIVE 用户的分享码不可继续绑定。

### 5.3 奖励发放时机

普通分享不在注册时奖励邀请人，避免刷号。

邀请人奖励触发条件：

```text
被邀请人完成首笔普通商品订单
订单已确认收货
订单无取消、退款、退货、换货成功
订单不是 VIP 礼包订单
订单不是团购订单
订单不是纯奖品或 0 元订单
```

第一版默认按奖励类型区分发放时机：

- 平台红包、积分、成长值：确认收货后发放，保证普通拉新的反馈速度。
- 普通奖励钱包、可提现余额或其他资金型奖励：必须等售后窗口结束且无成功售后后发放。

第一版默认只开放平台红包、积分、成长值，因此邀请人奖励默认在好友首单确认收货后发放。

奖励幂等键建议：

```text
NORMAL_INVITE_FIRST_ORDER:{inviteeUserId}:{orderId}
```

### 5.4 奖励类型

第一版奖励优先使用低风险权益：

- 平台红包：适合新人和邀请人的明确优惠。
- 积分：适合任务、签到、成长路径。
- 成长值：适合等级、头像框、身份装饰。
- 普通奖励钱包：只在后续确认合规和成本模型后开放给邀请奖励。

默认不使用现金提现奖励。

建议管理后台配置项：

| 配置 | 默认建议 |
|---|---|
| 普通分享码开关 | 关闭，灰度打开 |
| 新人注册红包活动 | 复用 `REGISTER` |
| 首单红包活动 | 复用 `FIRST_ORDER` |
| 普通邀请奖励活动 | 新增或扩展 `INVITE`，限定普通分享场景 |
| 邀请人奖励发放时机 | 红包/积分/成长值为首单确认收货后；资金型奖励为售后窗口后 |
| 每人每日可触发奖励人数 | 后台可配 |
| 同设备/同手机号/同支付账号限制 | 后端基础限制 |

## 6. 普通会员成长中心

普通会员成长中心不是新资金系统，而是把现有能力集中展示。

页面入口建议：

- 我的页身份卡下方增加“普通会员成长”模块。
- 常用工具中“推荐关系”改为“新人福利码”或“我的分享”。
- 任务和签到从隐藏状态恢复到可见入口。
- 钱包空状态不再写“成为 VIP 推荐好友获得消费积分”，改成普通用户也能理解的权益说明。

普通会员可展示权益：

1. 每日签到：获得积分，触发签到红包。
2. 今日任务：浏览、收藏、分享、下单、评价等任务获得积分或成长值。
3. 普通奖励：普通商品消费后进入普通树，满足规则后获得普通奖励。
4. 消费抵扣：普通商品最高 10% 消费积分抵扣。
5. 新人福利码：邀请好友注册和首单后拿奖励。
6. 省钱记录：展示红包抵扣、积分抵扣和普通奖励释放。

VIP 展示为升级加速器：

| 普通会员 | VIP |
|---|---|
| 普通商品最高 10% 积分抵扣 | 普通商品最高 15% 积分抵扣 |
| 普通分享码拉新奖励 | VIP 推荐码和 VIP 推荐权益 |
| 签到、任务、成长值 | 更高身份展示和专属礼包 |
| 普通奖励钱包 | VIP 奖励钱包和数字资产 |

## 7. 新用户 7 日路径

新用户 7 日路径优先复用红包、任务、签到和普通商品订单能力。

| 时间 | 目标 | 用户动作 | 系统奖励 |
|---|---|---|---|
| 第 0 天 | 完成注册 | 通过普通分享码下载注册 | 新人红包 |
| 第 1 天 | 完成首次打开和签到 | 签到、浏览精选商品 | 积分、签到红包 |
| 第 2 天 | 建立兴趣 | 收藏商品或店铺 | 成长值、任务奖励 |
| 第 3 天 | 触发首单 | 使用新人红包或首单红包下单 | 首单红包抵扣 |
| 第 4 天 | 强化履约信任 | 查看订单、物流、客服入口 | 成长值 |
| 第 5 天 | 收货后释放价值 | 确认收货、评价 | 普通奖励、评价红包 |
| 第 6-7 天 | 引导复购和分享 | 分享新人福利码或复购商品 | 分享任务奖励、复购红包 |

这一路径的核心不是让用户马上购买 VIP，而是先完成普通用户闭环：注册、领券、下单、收货、复购、分享。

## 8. App 端设计

### 8.1 我的页

我的页需要从“VIP 转化中心”调整为“普通会员权益先展示，VIP 作为升级入口”。

建议结构：

1. 身份卡：展示昵称、买家编号、普通会员等级、积分。
2. 成长模块：签到、任务进度、成长值。
3. 新人福利码：普通分享二维码、复制、系统分享。
4. 钱包概览：消费积分、普通奖励、团购返还余额按账户说明。
5. VIP 升级提示：基于用户已有普通权益给出差异化提示。

### 8.2 新人福利码页

普通用户页面展示：

- 我的新人福利码。
- 二维码。
- 复制链接。
- 分享给好友。
- 已邀请注册人数。
- 已完成首单人数。
- 待发放奖励。
- 已到账奖励。
- 规则说明。

VIP 用户页面可以同时展示：

- 普通新人福利码：用于普通 App 拉新。
- VIP 推荐码：用于 VIP 推荐权益。

两者必须用不同标题、不同路径、不同规则说明。

### 8.3 VIP 页

VIP 页文案从“没有 VIP 就没有推荐码”调整为：

```text
普通会员也可分享新人福利码。升级 VIP 后，可额外获得 VIP 推荐权益、专属礼包、数字资产和更高抵扣比例。
```

这样 VIP 不再是基础推广门槛，而是升级后的增强权益。

## 9. 后端设计

### 9.1 数据模型建议

建议新增独立模型，命名可在实施计划中按现有 Prisma 风格调整。

```prisma
model NormalShareProfile {
  id          String   @id @default(cuid())
  userId      String   @unique
  code        String   @unique
  status      String   @default("ACTIVE")
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

model NormalShareBinding {
  id             String   @id @default(cuid())
  inviterUserId  String
  inviteeUserId  String   @unique
  code           String
  source         String
  boundAt        DateTime @default(now())
  firstOrderId   String?
  rewardStatus   String   @default("PENDING")
  rewardIssuedAt DateTime?
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt
}
```

状态建议：

```text
NormalShareProfile.status = ACTIVE / DISABLED
NormalShareBinding.rewardStatus = PENDING / QUALIFIED / ISSUED / VOIDED
```

### 9.2 API 建议

买家端：

```text
GET  /normal-share/me
POST /normal-share/bind
GET  /normal-share/stats
GET  /normal-share/records
```

官网/落地页：

```text
POST /normal-share/deferred/create
GET  /normal-share/deferred/resolve
```

管理后台：

```text
GET   /admin/normal-share/settings
PATCH /admin/normal-share/settings
GET   /admin/normal-share/bindings
GET   /admin/normal-share/reward-records
POST  /admin/normal-share/:id/disable
```

### 9.3 事件接入

建议在以下事件点接入：

- 注册完成：尝试绑定 pending 普通分享码，给新人发新人红包。
- 首单创建：只记录候选，不立即奖励邀请人。
- 确认收货：若奖励类型为红包、积分、成长值，校验订单有效后发放邀请人奖励。
- 售后窗口结束：若奖励类型为普通奖励钱包、可提现余额或其他资金型奖励，校验订单有效后发放邀请人奖励。
- 退款、退货、换货成功：若奖励未发放则置为 `VOIDED`；若红包、积分、成长值已发放，第一版不自动追回，但应进入异常统计；资金型奖励因售后窗口后发放，不应出现已发放后又成功售后的常规场景。

### 9.4 幂等与安全

涉及奖励、红包、积分、状态转换的写入必须具备幂等约束。

最低要求：

- `inviteeUserId` 对普通分享绑定唯一。
- 邀请奖励按 `inviteeUserId + orderId` 唯一。
- 发放邀请奖励使用事务。
- 同一用户不能绑定自己。
- 非 ACTIVE、已注销、被禁用分享码不可绑定。
- 同设备、同手机号、同实名/支付账号的异常重复邀请需要基础限制。
- 管理后台可禁用某个普通分享码。

如奖励涉及 `RewardAccount` 或资金类余额变动，必须使用 Serializable 隔离级别。

## 10. 管理后台设计

新增或扩展“增长配置”页面。

配置项：

| 配置 | 说明 |
|---|---|
| 普通分享码总开关 | 支持灰度上线 |
| 分享码生成规则 | 自动生成或手动补发 |

| 每人每日奖励上限 | 防止刷量 |
| 每个邀请人总奖励上限 | 控制成本 |
| 异常邀请查看 | 同设备、同手机号、短时间批量注册 |

运营数据：

- 普通分享码生成用户数。
- 分享落地页访问量。
- 注册转化人数。
- 首单转化人数。
- 邀请奖励发放金额。
- 新用户 7 日留存。
- 普通用户升级 VIP 转化率。

## 11. 合规与页面文案边界

普通分享码页面只使用以下表达：

- 新人福利码
- 分享给好友
- 好友注册
- 好友首单
- 平台奖励
- 红包
- 积分
- 成长值
- 普通会员权益

避免以下表达：

- 拉人赚钱
- 团队收益
- 下级
- 多级返利
- 躺赚
- 保证赚钱
- 佣金
- 提成

普通分享码是 App 拉新福利，不是分销关系。

## 12. 分阶段实施建议

### Phase 1：普通会员成长中心和文案调整

目标：不新增复杂经济规则，先降低 VIP 门槛感。

范围：

- 我的页恢复任务和签到入口。
- 钱包空状态、推荐关系页、VIP 页文案调整。
- 普通用户展示“消费积分最高 10% 抵扣”。
- 我的页增加普通成长模块。

### Phase 2：普通分享码和新人福利落地页

目标：让每个普通用户都能分享 App。

范围：

- 新增普通分享码模型和 API。
- 新增 `/s/{code}` 官网落地页。
- App 新增新人福利码页。
- 注册后绑定普通分享关系。
- 新人红包复用现有 `REGISTER`。

### Phase 3：好友首单后奖励邀请人

目标：建立可持续拉新激励，但避免注册刷号。

范围：

- 订单确认收货后识别首单。
- 发放普通邀请奖励。
- 管理后台配置奖励和风控上限。
- 用户端展示邀请记录和奖励状态。

### Phase 4：VIP 转化桥

目标：在普通价值被证明后再转化 VIP。

范围：

- 展示普通用户已省金额、已得积分、已获得奖励。
- 对比 VIP 可多省金额和可解锁权益。
- 数字资产入口继续作为 VIP 激活能力。

## 13. 验证清单

后端：

- 普通分享码生成唯一。
- 普通用户可绑定普通分享码。
- 自己绑定自己被拒绝。
- VIP 推荐关系不被普通分享码覆盖。
- 被邀请人首单确认收货后只奖励一次。
- 团购订单、VIP 礼包订单、0 元订单不触发普通邀请奖励。
- 注销或封禁用户的分享码不可继续绑定。
- 奖励发放幂等。

App：

- 普通用户能看到新人福利码。
- VIP 用户能区分普通新人福利码和 VIP 推荐码。
- 钱包空状态不再暗示必须成为 VIP 才有消费积分。
- 任务和签到入口可见。
- 大字体和小屏下二维码、按钮、规则文案不重叠。

管理后台：

- 能配置普通分享奖励。
- 能查看邀请绑定和奖励记录。
- 能禁用异常分享码。
- 能查看注册、首单、发奖和留存数据。

## 14. 推荐优先级

推荐先做 Phase 1 和 Phase 2。

原因：

1. Phase 1 主要是前端信息架构和文案重排，能最快降低 VIP 门槛感。
2. Phase 2 给普通用户建立 App 级分享身份，是拉新机制的核心。
3. Phase 3 涉及订单、奖励、风控和资金边界，应该在普通分享入口稳定后再做。
4. Phase 4 是转化优化，不应该早于普通用户价值展示。

最终产品口径：

```text
普通会员也可以分享新人福利码，领红包、省钱、做任务、拿积分。
VIP 不是入场门槛，而是普通会员体验后的升级权益。
```
