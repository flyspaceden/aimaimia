# 组合商品设计方案

## 1. 背景

当前卖家中心一次创建的是一个普通可售商品。商品可以有多个 SKU/规格，但商品之间没有“组合售卖”的关系。商家希望把商品库里已有的单品 SKU 组合成一个新的普通可售商品，例如“水果礼盒”包含苹果 5斤装、梨 3斤装和橙子 1份。

本方案把组合看作一个商品：组合商品本身仍然是 `Product`，买家购买、购物车、订单、售后和审核都围绕这个商品进行；区别是这个商品内部有多个单品 SKU 组成，用于库存扣减、买家展示和卖家拣货打印。

## 2. 目标

1. 卖家可以创建普通可售的组合商品。
2. 组合商品由本商户商品库里的具体 SKU/规格组成。
3. 组合商品像普通商品一样进入商品列表、详情、购物车、下单、订单、售后和审核流。
4. 组合商品买家售价沿用现有“成本价 x 加价率”的自动定价规则。
5. 卖家创建时看到组成 SKU 当前售价总和，作为填写组合成本价的参考。
6. 买家端和订单详情展示组合内容。
7. 卖家订单和打印清单展开组合内容，方便工人按 SKU 拣货。
8. 下单时按组成 SKU 扣库存，不维护独立组合库存。

## 3. 非目标

1. 第一版不支持跨商户组合商品。
2. 第一版不支持组合商品嵌套组合商品。
3. 第一版不支持对组合内某个子商品单独售后，售后按组合整体处理。
4. 第一版不支持商家手动填写买家售价，仍由成本价自动定价。
5. 第一版不重写 VIP 赠品组合系统；VIP 赠品组合继续使用现有 `VipGiftOption` / `VipGiftItem`。
6. 第一版不做批量导入组合商品。

## 4. 核心规则

### 4.1 商品定位

组合商品是一个普通可售商品。系统应在 `Product` 上区分商品类型：

- `SIMPLE`：普通商品，现有商品默认属于此类型。
- `BUNDLE`：组合商品，有内部组成 SKU。

组合商品保留现有商品字段和能力：

- 标题、副标题、详情、图片、分类、标签、产地、退货政策。
- 草稿、提交审核、审核通过、上下架。
- 成本价、自动售价、SKU、购物车、订单、售后。

### 4.2 组成项

新增组合明细表，例如 `ProductBundleItem`：

```prisma
model ProductBundleItem {
  id              String     @id @default(cuid())
  bundleProductId String
  bundleProduct   Product    @relation("ProductBundleItems", fields: [bundleProductId], references: [id], onDelete: Cascade)
  skuId           String
  sku             ProductSKU @relation(fields: [skuId], references: [id], onDelete: Restrict)
  quantity        Int
  sortOrder       Int        @default(0)
  createdAt       DateTime   @default(now())
  updatedAt       DateTime   @updatedAt

  @@unique([bundleProductId, skuId])
  @@index([bundleProductId])
  @@index([skuId])
}
```

`Product` 和 `ProductSKU` 需要补充对应关系：

```prisma
model Product {
  bundleItems ProductBundleItem[] @relation("ProductBundleItems")
}

model ProductSKU {
  bundleItems ProductBundleItem[]
}
```

规则：

1. `bundleProductId` 指向组合商品。
2. `skuId` 指向被组合的具体 SKU。
3. 同一个组合中同一个 SKU 只保留一行，重复选择自动合并数量。
4. `quantity` 必须大于 0。
5. 最终组成项只能来自本商户自己的普通商品 SKU。
6. 最终组成项不能来自草稿、未审核、下架商品，也不能来自组合商品。

### 4.3 组合售卖 SKU

组合商品仍需要一个自己的售卖 SKU，用于复用现有购物车、CheckoutSession、订单项、支付金额和售后主链路。

组合售卖 SKU 的规则：

1. 价格和成本来自组合商品自己的成本价和自动售价。
2. `stock` 只是数据库占位，不作为真实库存来源。
3. 可售数量、库存不足判断、库存扣减、库存回填均按 `ProductBundleItem` 展开计算。
4. `weightGram` 也不作为真实发货重量来源；组合发货重量按组成 SKU 的 `weightGram x quantity` 汇总。
5. 卖家端不暴露组合售卖 SKU 的独立库存编辑入口，避免商家误以为组合有独立库存。

### 4.4 从已有组合展开

商家创建或编辑组合商品时，可以选择一个已有组合商品作为来源。系统只把该组合内部的普通 SKU 展开并加入当前组合清单，不保存组合套组合关系。

示例：

- 组合 A = 苹果 SKU x2 + 梨 SKU x1
- 新组合 B 选择组合 A 作为来源
- B 实际新增的是苹果 SKU x2 + 梨 SKU x1
- 如果 B 已有苹果 SKU x1，则合并为苹果 SKU x3

## 5. 价格与成本

组合商品价格口径：

1. 组成 SKU 当前售价 x 数量汇总为“参考总价”。
2. 参考总价只展示给商家，帮助商家决定组合商品成本价。
3. 商家填写组合商品自己的成本价。
4. 买家售价沿用现有自动定价规则：`组合成本价 x markupRate`。
5. 组合商品的订单金额、消费积分抵扣、平台红包、分润、数字资产累计消费、发票金额等均按组合商品自己的成交价计算。
6. 组成 SKU 当前售价变化不自动改历史订单，也不自动改组合商品售价。

## 6. 运费与重量

组合商品运费重量按组成 SKU 汇总：

```text
单个组合重量 = sum(组成 SKU.weightGram x 组成数量)
订单组合重量 = 单个组合重量 x 购买组合数量
```

结算、运费锁价、顺丰面单和平台运费成本记录都必须使用组合展开后的重量。组合售卖 SKU 的 `weightGram` 只作为 schema 必填占位，不能参与组合商品的真实运费计算。

如果组成 SKU 缺失有效重量，提交审核和结算都应拦截，提示商家先补全组成商品重量。不能用默认重量静默放行组合商品，否则会导致组合包裹运费和顺丰面单重量偏差。

## 7. 卖家中心设计

### 7.1 创建和编辑

商品创建页增加商品类型选择：

- 普通商品：保持现有流程。
- 组合商品：在现有基础信息之外增加“组合内容”区域。

组合内容区域展示：

1. 商品图。
2. 商品名。
3. SKU/规格。
4. 当前售价。
5. 当前库存。
6. 组合数量。
7. 小计。
8. 底部参考总价。

创建规则：

1. 卖家只能选择本商户自己的 SKU。
2. 选择普通商品时必须落到具体 SKU。
3. 可以从已有组合商品展开内部 SKU。
4. 重复 SKU 自动合并数量。
5. 草稿阶段允许组合内容不完整。
6. 提交审核时强校验组合内容必须完整有效。

编辑规则：

1. 组合商品编辑后沿用现有商品重新审核规则。
2. 编辑组合内容时实时重算参考总价。
3. 第一版不支持普通商品和组合商品互相转换，避免历史订单和审核语义混乱。

### 7.2 草稿

组合商品支持草稿：

1. 创建草稿只要求标题。
2. 草稿可保存不完整的组合内容。
3. 提交审核时校验组成 SKU 是否仍可用、数量是否有效、是否属于本商户、是否为普通商品。
4. 如果组成 SKU 后来下架、未审核或被删除，提交审核时提示卖家修改。
5. 提交审核时校验组成 SKU 重量是否有效，重量缺失或小于等于 0 时提示补全。

## 8. 管理端审核

组合商品继续进入现有商品审核列表。

审核详情需要展示：

1. 组合商品基础信息。
2. 组合商品成本价和自动售价。
3. 组合内容清单：商品、SKU、数量、当前售价、小计。
4. 组成 SKU 当前售价总和。

审核规则：

1. 管理端审核时不能放行跨商户组成项。
2. 管理端审核时不能放行组合套组合。
3. 审核通过后组合商品可上架销售。
4. 草稿组合不进入审核列表。
5. 管理端审核时需要看到组合总重量，便于发现明显不合理的包裹重量。

## 9. 买家端展示

组合商品在买家端表现为一个商品：

1. 商品列表和搜索中正常展示组合商品。
2. 商品详情页增加“组合内容”模块，展示内含商品、SKU 和数量。
3. 购物车展示组合商品本身，并展示组合内容摘要。
4. 确认订单页展示组合内容摘要。
5. 订单详情页展示组合内容快照。
6. 售后入口针对整个组合商品，不允许只选择内部某个 SKU。

组合商品可售数量：

```text
可售数量 = min(floor(组成 SKU 当前库存 / 单个组合所需数量))
```

示例：

- 苹果库存 10，组合每份需要 2。
- 梨库存 3，组合每份需要 1。
- 组合可售数量为 `min(floor(10 / 2), floor(3 / 1)) = 3`。

## 10. 购物车与下单

### 10.1 购物车

买家加入购物车的是组合商品自己的售卖 SKU。购物车数量表示购买几个组合。

购物车校验：

1. 组合商品必须是 `ACTIVE` 且审核通过。
2. 组合商品必须有有效组成项。
3. 任一组成 SKU 库存不足时，提示组合商品库存不足。
4. 组合商品数量不能超过按组成 SKU 计算的可售数量。
5. 不能使用组合售卖 SKU 的 `stock` 判断库存。

### 10.2 CheckoutSession

创建 CheckoutSession 时重新校验组合商品：

1. 读取组合商品及其组成 SKU。
2. 校验所有组成 SKU 仍为本商户普通商品 SKU。
3. 校验组成 SKU 状态、商品状态、审核状态和库存。
4. 计算组合商品金额时使用组合商品自己的售卖 SKU 价格。
5. 库存占用或支付成功扣减时按组成 SKU 展开。
6. 运费重量按组成 SKU 展开汇总，并写入现有运费锁价链路。

如果买 2 个组合，且每个组合需要苹果 SKU x3，则苹果 SKU 实际需要扣减 6。

### 10.3 支付成功建单

支付成功回调建单时：

1. 订单主项显示组合商品本身。
2. `OrderItem.productSnapshot` 保存组合商品快照。
3. 额外保存组合内容快照，例如 `bundleItems`：

```json
[
  {
    "skuId": "sku_apple_5jin",
    "productId": "product_apple",
    "productTitle": "红富士苹果",
    "skuTitle": "5斤装",
    "quantityPerBundle": 2,
    "bundleQuantity": 3,
    "totalQuantity": 6,
    "unitPriceAtOrder": 19.9,
    "image": "https://..."
  }
]
```

库存扣减：

1. 在 Serializable 事务内按组成 SKU 扣库存。
2. 记录库存流水，流水原因标记为组合商品订单。
3. 结算前已知库存不足则拦截。
4. 支付回调阶段如果并发导致组成 SKU 库存变负，沿用当前普通商品超卖策略：允许建单成功，记录库存流水，并通知卖家补货。

## 11. 订单、发货与打印清单

卖家订单列表仍以组合商品作为订单商品展示。

订单详情展示两层信息：

1. 买家购买的组合商品，例如 `水果礼盒 x2`。
2. 组合内容快照，例如苹果 5斤装 x4、梨 3斤装 x2。

打印清单必须基于订单快照，不读当前商品配置，避免订单后商品修改导致工人拿错货。

打印清单展示：

```text
水果礼盒 x2
- 红富士苹果 / 5斤装 x4
- 皇冠梨 / 3斤装 x2

拣货汇总
红富士苹果 / 5斤装 x4
皇冠梨 / 3斤装 x2
```

如果订单同时包含多个组合和普通商品，打印清单应：

1. 保留每个组合商品下面的明细。
2. 额外生成按 SKU 汇总的拣货清单。
3. 普通商品也进入拣货汇总。

## 12. 售后

组合商品售后按整体处理：

1. 买家只能对整个组合订单项申请退货、退款或换货。
2. 第一版不支持只退组合里的某个子 SKU。
3. 售后退款金额按组合商品订单项成交金额计算。
4. 退货入库或换货发货时，库存回填/扣减需要按组合内容快照展开到组成 SKU。
5. 售后详情页展示组合内容快照，方便客服、卖家和买家确认。

## 13. 安全与一致性

本功能涉及库存、订单、售后、金额和审核状态，属于高风险链路。实现时必须对照 `docs/issues/tofix-safe.md` 的安全检查清单。

关键约束：

1. 涉及库存扣减、库存回填、订单创建、售后库存处理的事务必须使用 Serializable 隔离级别。
2. 后端必须校验商户隔离，不能只依赖前端筛选。
3. 后端必须禁止组合商品作为最终组成项。
4. 后端必须保存订单组合快照，历史订单不能依赖当前商品配置。
5. 组成 SKU 改价、下架、删除、审核状态变化不能破坏历史订单展示和打印。
6. 组合商品成本价和订单成交金额必须进入现有金额校验、分润、积分、红包、数字资产和发票链路。
7. 组合售卖 SKU 的占位库存和占位重量不能被普通链路误用；所有读写库存和运费计算入口都必须识别 `Product.type = BUNDLE`。

## 14. 验收标准

1. 卖家可以创建组合商品草稿。
2. 卖家可以从本商户普通商品 SKU 中选择组成项。
3. 卖家可以从已有组合商品展开内部 SKU 到新组合。
4. 重复 SKU 自动合并数量。
5. 卖家创建页展示组成 SKU 当前售价总和。
6. 组合商品提交审核后进入管理端审核列表。
7. 管理端审核详情展示组合内容。
8. 审核通过并上架后，买家端可以看到组合商品。
9. 买家详情页、购物车、确认订单、订单详情展示组合内容。
10. 购物车和结算按组成 SKU 库存计算可买数量。
11. 支付成功后扣减组成 SKU 库存，而不是扣组合商品独立库存。
12. 订单快照保存组合内容，商品后续修改不影响历史订单。
13. 卖家订单详情和打印清单从订单快照展开组合内容。
14. 售后只能按组合整体申请。
15. 组合商品不能跨商户选择 SKU。
16. 组合商品不能组合另一个组合商品。
17. 组合商品运费重量按组成 SKU 重量汇总。
18. 组合售卖 SKU 的占位库存不会影响购物车、结算和商品详情库存展示。

## 15. 实施拆分建议

1. Schema：新增商品类型和 `ProductBundleItem`。
2. 后端 seller products：组合商品草稿、创建、更新、提交审核、展开已有组合、校验本商户 SKU。
3. 后端 buyer product/cart/checkout：组合商品展示、可售数量、购物车校验、CheckoutSession 校验。
4. 后端订单：支付成功建单、库存扣减、订单快照、库存流水、超卖通知。
5. 后端售后：整体售后和库存回填按组合快照展开。
6. 卖家前端：商品类型、组合内容选择器、参考总价、从已有组合展开。
7. 管理前端：审核详情展示组合内容。
8. 买家 App：详情、购物车、确认订单、订单详情展示组合内容。
9. 卖家订单和打印：订单详情展开、打印清单 SKU 汇总。
10. 文档和验证：更新商品、订单、售后、响应式和发布前检查文档。
