# 订单收货信息纠错设计方案

## 背景

卖家在待发货订单中生成顺丰电子面单时，顺丰可能返回“对方电话或手机不合法”。这类问题通常来自买家下单时填写的收货手机号错误。当前 App 订单详情只展示脱敏收货信息，地址簿修改也不会回写已付款订单的地址快照，导致买家无法自助修正当前订单，卖家只能反复生成面单失败或线下联系买家。

## 目标

第一版建立“卖家发货失败 -> 买家收货信息待办 -> 买家修改当前订单快照 -> 卖家重新生成面单”的闭环。

## 非目标

- 不接短信通知，因为订单手机号本身可能错误。
- 不做快递已揽收后的改派、拦截或退回重发。
- 不允许修改商品、数量、价格、优惠、支付方式或买家留言。
- 不新增商家审批流程；未生成面单前由买家自助修正。

## 用户流程

1. 卖家在订单详情点击“生成面单（顺丰速运）”。
2. 若顺丰返回收件手机号/电话不合法，后端清理面单生成标记并发送交易类站内信给买家。
3. 买家在 App 消息中心或订单详情看到“请修改收货信息”的待办。
4. 买家点击“修改收货信息”，编辑收货人、手机号、所在地区、详细地址。
5. 后端校验订单属于该买家、订单仍处于 `PAID`、该商户尚未生成面单，并用 Serializable 事务更新订单地址快照。
6. 买家保存成功后，订单详情刷新，卖家可重新生成顺丰面单。

## 状态边界

- 允许修改：普通商品订单 `PAID` 且没有任何关联 `Shipment.waybillNo`。
- 不允许修改：已生成面单、已发货、已送达、已完成、已取消、已退款、售后处理中、VIP 礼包、团购订单。
- 若错误来自商家发件电话或企业地址，卖家端展示企业信息错误，不通知买家。
- 若买家仅修改地址簿，不影响历史订单；当前订单必须通过订单详情的“修改收货信息”更新订单快照。

## 数据与 API

新增买家端接口：

- `PATCH /orders/:id/receiver-info`

请求体：

```json
{
  "recipientName": "张三",
  "phone": "13800000000",
  "regionCode": "450481",
  "regionText": "广西壮族自治区/梧州市/岑溪市",
  "detail": "某街道 1 号"
}
```

后端更新 `Order.addressSnapshot`，保持现有加密存储方式；响应返回订单详情映射结果。

## 通知

复用 `InboxMessage`：

- `category`: `transaction`
- `type`: `order_receiver_info_required`
- `target`: `{ route: "/orders/[id]", params: { id: orderId } }`

消息文案：

> 商家发货时发现收货手机号无法生成快递面单，请修改收货信息。修改前商家无法发货。

## 前端体验

订单详情地址卡片下方在 `receiverInfoEditable=true` 或 `receiverInfoIssue` 存在时显示待办提示。按钮文案为“修改收货信息”。编辑页复用地址表单的字段和地区选择器，但保存的是当前订单快照，不更新地址簿。

地址簿手机号校验同步加严为大陆手机号正则，避免继续产生无效手机号订单。

## 安全与一致性

- 更新订单地址快照涉及履约状态转换边界，使用 Serializable 事务。
- 更新前检查当前订单没有任何 `Shipment.waybillNo`，防止卖家并发生成面单后买家仍能改地址。
- 接口仅允许订单买家本人调用。
- 记录订单地址修改站内信和后端日志，便于追踪。

## 验收

- 卖家生成顺丰面单遇到收件手机号不合法时，买家收到可跳转订单详情的站内信。
- 买家可在待发货且未生成面单的普通商品订单中修改收货手机号，保存后卖家可重新生成面单。
- 已生成面单、已发货、取消/退款/VIP/团购订单不能修改收货信息。
- 地址簿和订单收货信息表单都拒绝非大陆手机号。
