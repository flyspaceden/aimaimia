# 数字资产 V2 规则设计方案

> 状态：设计已确认，待实施计划
> 创建时间：2026-06-17
> 适用范围：买家 App / 管理后台 / 后端 / Prisma Schema / 法律文本
>
> **For agentic workers:** 本文档是数字资产 V2 规则权威来源，补充并覆盖 `2026-06-14-digital-asset-cumulative-spend-design.md` 中“VIP 礼包计入累计消费”和“仅展示累计消费金额”的旧口径。V2 只定义数字资产生成、记录、展示和规则配置，不定义数字资产使用方式。

## 背景

第一版数字资产已经建立了 `DigitalAssetAccount` + `DigitalAssetLedger` 的累计消费底座，买家 App 和管理后台可以展示“累计消费金额”。V2 需要在这个底座上升级为真正的数字资产体系：

1. 所有用户都记录普通商品累计消费金额。
2. 只有 VIP 用户拥有数字资产余额。
3. 数字资产由种子资产和消费资产组成。
4. 后台可配置 VIP 种子资产和消费资产倍率规则。
5. App 展示数字资产总额、资产构成、累计消费金额、消费记录和规则待开放模块。

数字资产的现金兑换、定期收益、未来权益折算等使用方式暂未确定，本期只做占位和法律边界。

## 核心定义

| 概念 | 定义 |
|---|---|
| 累计消费金额 | 所有用户都有。仅统计普通商品真实实付商品金额，不含运费、平台红包抵扣、消费积分抵扣、VIP 折扣等；退款/退货成功后扣回 |
| 种子资产 | 只有 VIP 用户拥有。来源于自己购买 VIP 和直接推荐别人购买 VIP |
| 消费资产 | 只有 VIP 用户拥有。来源于普通商品累计消费按后台倍率规则转化 |
| 数字资产总额 | `种子资产 + 消费资产`，只对 VIP 用户展示 |
| 消费记录 | App 独立页面，展示完整普通消费、退款扣回、VIP 种子资产、消费资产转入和后台调整记录 |

展示数字资产时只显示数字，不加“点”“份”“币”等单位；累计消费金额仍按人民币金额展示。

## 已确认决策

| 决策点 | 结论 |
|---|---|
| 系统方案 | 在现有 `digital-asset` 模块上升级为 V2，不拆新系统 |
| 普通用户 | 有累计消费金额，没有数字资产余额 |
| VIP 用户 | 展示数字资产总额、种子资产、消费资产和累计消费金额 |
| VIP 自购入账 | VIP 支付成功后立即入账种子资产，不等确认收货 |
| VIP 推荐入账 | 被推荐人 VIP 支付成功后，直接推荐人立即获得种子资产 |
| 推荐层级 | 只发给直接推荐人，不沿 VIP 树多级发放 |
| VIP 重复购买 | 现有代码和 `VipPurchase.userId` 唯一约束已限制每个用户只能购买一次 VIP |
| VIP 与累计消费 | VIP 礼包不计入累计消费金额，不产生消费资产；不参与普通/VIP 上溯有效消费计数 |
| 消费资产依据 | 以用户总累计消费金额跨档分段计算，不因成为 VIP 重新从 0 开始 |
| 普通转 VIP | 成为 VIP 时按当时累计消费总额和当前倍率规则一次性补发历史消费资产 |
| 历史补发 | 只执行一次，后续规则调整不追溯、不补差 |
| 既有 VIP 迁移 | V2 上线时对已是 VIP 的老用户一次性补发自购种子资产和历史消费资产 |
| 规则生效 | 后台修改规则立即生效，只影响未来新流水 |
| 规则快照 | 每笔资产流水保存当时规则快照 |
| 数字资产精度 | 整数入账。分段计算后整笔加总，四舍五入为整数 |
| 使用方式 | 现金兑换、定期收益、未来权益折算均未定，本期不实现 |
| 转让交易 | 不允许用户间转让、交易、赠送、倒卖 |
| 注销/封禁 | 数字资产清零作废，仅保留必要审计记录 |

## 非目标

V2 不做以下内容：

- 不做现金兑换。
- 不做定期收益发放。
- 不做未来权益折算。
- 不做资产锁定。
- 不做资产消费、抵扣或提现。
- 不做用户主动申请使用数字资产。
- 不承诺固定收益、固定兑换比例、股权或分红。
- 不追溯后台规则变更前的历史资产流水。

## 账户模型

继续复用现有 `DigitalAssetAccount`，但语义升级为：

```text
cumulativeSpendAmount: 所有用户的累计普通商品真实消费金额
seedAssetBalance: VIP 用户种子资产余额
creditAssetBalance: VIP 用户消费资产余额
totalAssetBalance: seedAssetBalance + creditAssetBalance，可存储也可查询时计算
historicalCreditGrantedAt: 历史累计消费转消费资产的执行时间
historicalCreditGrantLedgerId: 历史补发流水 ID
```

普通用户可以存在 `DigitalAssetAccount`，但 `seedAssetBalance` 和 `creditAssetBalance` 必须为 0。普通用户接口不得返回可被理解为数字资产余额的字段。

## 流水模型

`DigitalAssetLedger` 需要区分记录对象和业务来源：

```text
subjectType:
  SEED_ASSET        种子资产
  CREDIT_ASSET      消费资产
  CUMULATIVE_SPEND  累计消费金额记录

sourceType:
  CONSUMPTION_CONFIRMED       普通商品确认收货
  REFUND_REVERSAL             普通商品退款/退货扣回
  SELF_VIP_PURCHASE           自购 VIP 种子资产
  REFERRAL_VIP_PURCHASE       推荐 VIP 种子资产
  HISTORICAL_CONSUMPTION_GRANT 历史累计消费转消费资产
  ADMIN_ADJUSTMENT            后台人工调整
  BACKFILL                    历史补数
```

资产余额类流水保存 `balanceAfter`、`seedAssetBalanceAfter`、`creditAssetBalanceAfter` 或等价快照。累计消费类流水保存 `cumulativeSpendAfter`。实现时可以沿用现有方向字段，但需要确保每条流水可解释“改了哪个余额/金额”。累计消费金额不是数字资产余额，不得把 `CUMULATIVE_SPEND` 记为可使用资产。

所有自动入账/扣回必须有幂等键：

- 普通商品确认收货累计消费：`order:{orderId}:spend-credit`
- VIP 后普通消费消费资产：`order:{orderId}:credit-asset`
- 普通商品退款扣回：`refund:{refundId}:digital-asset-reversal`
- 自购 VIP 种子资产：`vip-purchase:{vipPurchaseId}:self-seed`
- 推荐 VIP 种子资产：`vip-purchase:{vipPurchaseId}:referral-seed`
- 历史消费转消费资产：`user:{userId}:historical-consumption-credit-grant`

## 种子资产规则

种子资产只和 VIP 相关。

### 默认规则

| VIP 档位 | 自购种子资产 | 推荐种子资产 |
|---|---:|---:|
| 399 | 1000 | 2000 |
| 699 | 2000 | 4000 |
| 999 | 3000 | 8000 |

### 后台配置

种子资产配置绑定现有 `VipPackage` 档位。每个 VIP 档位增加：

- 自购种子资产数量。
- 推荐种子资产数量。

后台修改后立即生效，只影响未来新支付成功的 VIP。每笔流水保存当时的 VIP 档位、金额、配置值、`packageId`、`vipPurchaseId`、购买人和推荐人快照。

### 入账时点

VIP 支付成功并激活成功后：

1. 给购买人发自购种子资产。
2. 如果购买人有直接推荐人，给直接推荐人发推荐种子资产。
3. 不给间接上级发数字资产。
4. 不改变现有 VIP 现金分润、普通树或 VIP 树分配逻辑。

## 消费资产规则

消费资产来自累计消费金额。累计消费金额只包括普通商品真实实付商品金额，不包括 VIP 礼包实付金额。

### 默认倍率

| 累计消费金额区间 | 消费资产倍率 |
|---|---:|
| 0-500 元 | x3 |
| 500-5000 元 | x5 |
| 5000 元以上 | x10 |

后台可自由增删改档位，默认初始化为上面三档。配置校验要求：

- 档位按起始金额升序。
- 不允许重叠。
- 不允许断档。
- 第一档从 0 开始。
- 最后一档可以无上限。
- 倍率必须大于 0。

### 普通用户阶段

普通用户确认收货普通商品订单时：

1. 更新累计消费金额。
2. 不发消费资产。
3. App 数字资产页只展示累计消费金额和 VIP 激活引导。

### 成为 VIP 时历史补发

用户成为 VIP 时：

1. 读取当前累计消费金额。
2. 按当时后台最新消费资产倍率规则计算历史消费资产。
3. 四舍五入为整数。
4. 写一笔 `HISTORICAL_CONSUMPTION_GRANT` 消费资产流水。
5. 标记该用户已执行历史补发。

历史补发只执行一次。后续后台改倍率不追溯、不补差。

示例：成为 VIP 时累计消费 5800 元：

```text
0-500: 500 * 3 = 1500
500-5000: 4500 * 5 = 22500
5000-5800: 800 * 10 = 8000
历史补发消费资产 = 32000
```

### 既有 VIP 迁移补发

V2 上线前已经是 VIP 的用户需要一次性补发：

1. 自购种子资产：根据该用户的 `VipPurchase.packageId` 优先匹配 VIP 档位；如果缺少 `packageId`，用 `VipPurchase.amount` 匹配档位价格；仍无法匹配时进入迁移异常清单，不自动发放。
2. 历史消费资产：读取用户当前累计消费金额，按迁移执行时后台最新消费资产倍率规则计算。
3. 迁移补发必须写独立流水和规则快照。
4. 迁移补发必须有幂等键，重复执行不得重复发放。
5. 迁移脚本默认 dry-run，必须显式执行写入模式。

既有 VIP 的推荐种子资产不做自动补发。原因是推荐关系和当时购买档位可能涉及历史链路核验，直接补发容易误发；如后续需要，应单独设计迁移规则。

### VIP 后普通消费

VIP 用户普通商品确认收货时：

1. 计算本次订单真实实付商品金额。
2. 用“入账前累计消费金额 -> 入账后累计消费金额”跨档分段。
3. 按当前倍率规则计算消费资产。
4. 整笔订单所有分段加总后四舍五入为整数。
5. 写累计消费流水和消费资产流水。
6. 流水保存规则快照、分段明细和订单商品分摊明细。

示例：入账前累计消费 480 元，本次确认收货 100 元：

```text
480-500: 20 * 3 = 60
500-580: 80 * 5 = 400
本次消费资产 = 460
```

### 退款扣回

普通商品退款/退货成功后：

1. 扣回累计消费金额。
2. 如果原订单曾产生消费资产，按原订单流水保存的分段和商品行分摊结果扣回消费资产。
3. 不按退款发生时的当前倍率规则重算。
4. 扣回不得超过原订单已产生且未扣回的消费资产。

VIP 礼包没有退款，也不产生消费资产；VIP 赠品售后不触发消费资产扣回。

如果退款渠道和退款业务状态已经成功，但数字资产扣回失败，不得回滚退款成功状态，也不得阻断用户到账通知。系统必须写入 `DigitalAssetRefundReversalFailure` 补偿记录，记录 `refundId`、`orderId`、`afterSaleId`、`userId`、来源、失败原因、重试次数和下次重试时间。定时任务每 10 分钟扫描待处理记录并调用同一个幂等的 `reverseRefund(refundId)`；成功后标记 `RESOLVED`，连续失败达到上限后标记 `FAILED`，等待人工核查。

## App 设计

### 普通用户数字资产页

普通用户可以进入数字资产页，但不显示数字资产余额。页面内容：

- 累计消费金额。
- VIP 激活引导。
- 当前消费资产规则说明。
- VIP 档位种子资产规则说明。
- 最近 5 条消费记录摘要。
- “查看全部消费记录”入口。
- 长期收益权规则待开放区域。

引导文案：

```text
让每一次消费，都成为你的数字资产基础
成为 VIP 后，累计消费可按规则转化为消费资产。
开通 VIP 激活资产
```

### VIP 用户数字资产页

顶部主数字显示“数字资产总额”。页面内容：

- 数字资产总额。
- 种子资产。
- 消费资产。
- 累计消费金额。
- 当前消费资产倍率和下一档进度。
- 当前 VIP 档位种子资产规则。
- 最近 5 条消费记录摘要。
- “查看全部消费记录”入口。
- 长期收益权规则待开放区域。

### 消费记录页

独立页面命名为“消费记录”。数字资产页只显示最近 5 条，完整列表进入消费记录页查看。

消费记录页展示：

- 普通商品消费累计记录。
- 退款/退货扣回记录。
- 自购 VIP 种子资产记录。
- 推荐 VIP 种子资产记录。
- 历史消费转消费资产记录。
- 后台人工调整记录。

可按实现复杂度支持筛选，但首版至少需要时间倒序完整列表和清晰类型标签。

## 管理后台设计

### 总览

数字资产后台总览展示：

- 数字资产总额。
- 种子资产总额。
- 消费资产总额。
- 累计消费总额。
- 今日种子资产入账。
- 今日消费资产入账。
- 今日累计消费新增。
- 今日扣回。

### 用户列表

列表字段：

- 用户身份。
- VIP 状态。
- 数字资产总额。
- 种子资产。
- 消费资产。
- 累计消费金额。
- 最近更新时间。

普通用户数字资产总额、种子资产、消费资产显示为 0 或 `未激活`，累计消费金额正常展示。

### 用户明细

明细展示：

- 资产构成。
- 累计消费金额。
- 当前倍率档位和下一档进度。
- 完整流水。
- 每笔流水的规则快照。

### 人工调整

后台人工调整必须选择资产类型：

- 种子资产。
- 消费资产。

不能直接调整数字资产总额。总额由系统汇总。调整必须填写原因，写审计日志，并保留调整前后余额快照。

### 规则配置

规则配置包括：

1. VIP 档位管理中配置自购种子资产和推荐种子资产。
2. 数字资产规则中配置消费资产倍率阶梯。
3. 未来权益模块只做中性占位配置和文案，不做计算。

本期使用简单规则模式：修改后立即生效，不做预约生效和历史版本页面。历史流水靠规则快照解释。

## 未来权益占位

数字资产如何使用还没有确定。App 和后台只展示中性占位，不承诺数字：

- 未来权益模块：规则待开放。
- 权益规则待开放：规则待开放。
- 后续使用方式：以平台后续公示规则为准。

禁止在 App 或法律文本中写成固定现金兑换比例、固定利息、固定股权或固定分红承诺。

## 法律与合规文案

法律文本和 App 规则说明需要同步：

- 数字资产不是现金余额。
- 数字资产不是消费积分、证券或可流通代币。
- 数字资产不支持用户间转让、交易、赠送、倒卖。
- 未来权益模块未开放前不构成平台承诺。
- 后续具体门槛、比例、周期、限制条件以平台公示规则为准。
- 账号注销或严重违规封禁时，数字资产清零作废，仅保留必要审计记录。

涉及未来用途的表达统一弱化为“未来权益模块 / 权益规则待开放 / 规则待开放”，不直接承诺现金、收益、股权或分红。

## 安全边界

所有写入必须收口到 `DigitalAssetService`，禁止业务模块直接更新账户余额。

必须使用 `Prisma.TransactionIsolationLevel.Serializable` 的场景：

- 累计消费金额增加或扣回。
- 消费资产入账或扣回。
- 种子资产入账。
- VIP 激活时的自购种子资产、推荐种子资产和历史消费资产补发。
- 后台人工调整。
- 注销/封禁清零。

错误处理要求：

- VIP 激活流程中的自购种子资产、推荐种子资产和历史消费资产补发应接入现有 VIP 激活重试状态机；任一必发资产写入失败时，不得静默标记 VIP 资产发放成功，必须可重试或进入人工核查。
- 普通商品确认收货后的累计消费和消费资产写入失败不得阻断订单履约终态，但必须写错误日志或死信记录，后续可补偿扫描。
- 退款/退货后的累计消费和消费资产扣回失败不得回滚退款终态，但必须写错误日志或死信记录，后续可补偿扫描。
- 后台人工调整失败必须整体失败，不得出现只写审计不改余额或只改余额不写审计。

并发和幂等要求：

- 确认收货重复触发不得重复增加累计消费或消费资产。
- 支付回调重复触发不得重复发种子资产。
- VIP 激活重试不得重复补发历史消费资产。
- 退款通知或售后状态重复推进不得重复扣回。
- 后台调整支持客户端幂等键。

## 迁移注意事项

现有第一版实现把数字资产账户理解为累计消费账户，且旧文档曾定义 VIP 礼包计入累计消费。V2 实施前必须审计线上 `DigitalAssetLedger` 和 `Order.bizType`：

1. 如已有 `VIP_PACKAGE` 被计入累计消费，需要制定一次性修正方案。
2. 普通商品历史累计消费继续保留。
3. 已有普通商品累计消费可作为普通用户/VIP 用户的累计消费基数。
4. 已有后台人工调整需要判断是累计消费校正还是资产余额调整，不能自动迁移为种子资产或消费资产。
5. V2 上线前已是 VIP 的用户需要执行一次性补发 dry-run，报告应列出可自动补发、缺少档位无法补发、疑似重复或异常账户三类结果。

迁移方案应在实施计划中单独列出，并在执行前做 dry-run 报告。

## 验收标准

- 普通用户数字资产页不展示数字资产余额，只展示累计消费和 VIP 激活引导。
- VIP 用户数字资产页展示数字资产总额、种子资产、消费资产和累计消费金额。
- 数字资产页只展示最近 5 条消费记录，完整列表通过“查看全部消费记录”进入消费记录页。
- VIP 自购种子资产在支付成功后只入账一次。
- 直接推荐人种子资产在被推荐人支付成功后只入账一次。
- 推荐种子资产不多级发放。
- 普通用户成为 VIP 时历史消费资产只补发一次。
- V2 上线前已是 VIP 的用户按迁移规则一次性补发自购种子资产和历史消费资产，重复执行脚本不重复发放。
- 消费资产跨档分段计算正确，整笔四舍五入为整数。
- 退款扣回按原订单流水快照执行，不按当前规则重算。
- 后台规则修改只影响未来新流水。
- 后台人工调整必须选择种子资产或消费资产，不能直接改总额。
- 数字资产不支持转让、交易、赠送、倒卖。
- 法律文本不承诺现金兑换、固定收益、股权或分红。
