# 团长利润分成与跨配置硬校验设计方案

## 1. 目标与边界

预包装海鲜团长计划、VIP 分润系统和普通用户分润系统继续保持三条独立业务路径：独立关系、独立配置、独立资格、独立账户和独立流水。团长比例不得并入 VIP 或普通用户的利润七分配置。

本设计只改变资金计算和安全控制：

1. 团长逐单及月度奖励由销售额比例改为可分润利润比例。
2. VIP/普通直推、树奖励和产业基金改用扣除订单优惠后的实际利润。
3. 后台保存配置、回滚配置或修改商品价格/成本时做硬校验；订单结算不临时缩减比例。
4. 同一订单内三条路径产生的金额必须资金守恒，不能在 VIP/普通七分已经分完 100%后再无来源地增加团长奖励。

团长继续只保留一层直接客户收益；本设计不得恢复二级团长、间接佣金或团队向上计酬。

## 2. 已确认的业务决策

1. 团长是独立的 `SEAFOOD_PREPACKAGED` 项目，不修改 VIP 树、普通树、VIP 码、普通分享码或团长码的关系语义。
2. 团长逐单佣金按利润百分比计算。
3. 团长管理津贴、增长奖、有效成交辅导奖和经营绩效奖也按直接客户利润百分比计算。
4. 团长资格和档位仍使用直接客户 GMV、有效客户数、新增有效客户数及退款率。
5. VIP/普通分润利润必须先扣除 VIP 折扣、平台红包、消费积分抵扣、团购返还余额抵扣及其他商品级优惠。
6. 平台占比、慈善、科技和备用金暂归平台，本次作为平台留存资金，不作为对外承诺扣减。
7. 产业基金进入卖家 OWNER 账户，属于对外承诺资金，必须计入资金守恒与安全校验。
8. 配置不安全时拒绝保存，不自动修改管理员输入；订单结算不做动态比例裁剪。

## 3. 利润术语与计算顺序

为了避免把商品毛利和真正可分润金额混为一谈，系统统一使用以下术语。

### 3.1 商品实收

订单支付成功时创建不可变利润快照：

```text
商品实收 = max(0, 订单实付总额 - 买家支付运费)

商品优惠合计 =
  VIP 折扣
  平台红包抵扣
  消费积分抵扣
  团购返还余额抵扣
  其他已计入订单实付的商品级抵扣
```

必须满足分级守恒：

```text
订单商品原价合计 - 商品优惠合计 = 商品实收
Σ 订单项原价 = 订单商品原价合计
Σ 订单项优惠 = 商品优惠合计
Σ 订单项实收 = 商品实收
```

所有金额先转换为分，以整数完成分摊和加减；禁止使用 Float 直接分摊。末分采用最大余数法分配，排序键固定为 `OrderItem.id`，保证同一输入得到同一结果。

### 3.2 优惠分摊顺序

1. 明确绑定商品或订单项的优惠先分配给目标订单项。
2. 订单级 VIP 折扣、消费积分、团购返还余额及无商品归属的红包，按剩余非奖品订单项原价比例分摊。
3. 每个订单项的优惠分摊以其“原价 - 已分摊优惠”为容量上限。一次比例分配超过容量的部分必须重新分配给仍有容量的订单项，循环到全部优惠分完或全部容量耗尽。
4. 已经被优惠抵扣到零的订单项不再分摊后续优惠；任何订单项实收都不得小于零。
5. 若优惠总额超过全部非奖品商品容量，利润快照进入对账状态，不得把超额优惠写成负商品收入。
6. 奖品项不分摊优惠，不进入商品收入、成本或利润。
7. `groupBuyRebateDeductionAmount` 必须在利润快照中独立保存，不能混入无法解释的其他优惠。

任何守恒式不成立时，利润快照进入 `RECONCILIATION_REQUIRED`，不得生成任何用户奖励。

### 3.3 商品毛利与可分润利润

每个非奖品订单项固定支付时成本：

```text
订单项商品毛利 = 订单项实收 - 支付时 SKU 成本快照 × 数量
订单商品毛利 = Σ 订单项商品毛利
```

亏损订单项不得单独截断为零后再求和。必须先把所有正负订单项毛利相加，最后计算订单级金额，避免用盈利商品掩盖亏损商品。

履约成本、风险预留和目标平台净利按订单项实收比例分摊：

```text
订单项可分润贡献 =
  订单项商品毛利
  - 订单项履约成本预留
  - 订单项风险预留
  - 订单项目标平台净利预留

订单可分润利润 D = max(0, Σ 订单项可分润贡献)
```

为了给退款提供严格上限，支付快照把 `D` 重新分配给正贡献订单项：

```text
正贡献合计 P = Σ max(0, 订单项可分润贡献)

若 D > 0：
订单项可分润份额 = D × max(0, 订单项可分润贡献) / P

若 D = 0：
所有订单项可分润份额 = 0
```

按分使用最大余数法补差，必须满足 `Σ 订单项可分润份额 = D`。亏损订单项通过降低 `D` 等比例压低全部正贡献份额，不会被忽略，也不会在退款时冲回超过订单实际可分润金额。

VIP 七分、普通七分和团长利润分成都以 `D` 或其订单项份额子集为基数。这样平台先保留履约、风险和目标净利，再计算任何推广奖励。

### 3.4 团长适用利润

混合订单中，团长只取得符合范围订单项已经分配到的可分润份额：

```text
团长利润基数 C = Σ 符合团长范围的订单项可分润份额
```

因此团长利润基数永远不会大于订单可分润利润，非团长商品的亏损也不会被忽略。

## 4. 不可变利润与规则快照

### 4.1 订单利润快照

新增 `OrderProfitSnapshot`，支付建单事务内创建，至少保存：

```text
orderId（唯一）
status: READY | RECONCILIATION_REQUIRED
grossGoodsAmount
shippingAmount
vipDiscountAmount
couponDiscountAmount
rewardDeductionAmount
groupBuyRebateDeductionAmount
otherGoodsDiscountAmount
netGoodsRevenue
productCostAmount
fulfillmentReserveAmount
riskReserveAmount
targetNetProfitAmount
distributableProfitAmount
captainEligibleProfitAmount
calculationVersion
itemBreakdown
errorCode / errorMeta
createdAt / reconciledAt
```

`itemBreakdown` 固定订单项原价、专项优惠、订单级优惠分摊、实收、成本、三项预留、可分润贡献及团长适用标记。

### 4.2 订单规则快照

利润快照同时固定该订单后续分配所需规则，至少包括：

```text
buyerTierAtPayment
directInviterUserIdAtPayment
directInviterTierAtPayment
directRelationSourceAtPayment
vipTreeNodeIdAtPayment
normalTreeNodeIdAtPayment
vipTreeAncestorPathAtPayment（最多配置层数）
normalTreeAncestorPathAtPayment（最多配置层数）
captainUserIdAtPayment
captainRelationIdAtPayment
captainConfigVersion
vipNormalConfigVersion
validatedSafetyVersion
所有本单适用费率
月度资格门槛、GMV 档位和退款率门槛
```

支付时直推奖、收货后树奖励和月度团长奖励必须读取同一快照。收货时根据有效消费序号从支付时保存的祖先路径选择受益人，不重新查询当前树位置。支付后用户升级 VIP、树节点调整、邀请关系变化或后台修改比例/门槛，都不得改变该订单的路径、候选祖先、费率和所属配置批次。

### 4.3 成本异常的运行时处理

支付订单不能因为奖励成本资料缺失而阻断买家付款，但必须资金失败关闭：

1. 成本缺失、成本非正数、守恒失败或无法计算时，订单正常创建。
2. 利润快照写为 `RECONCILIATION_REQUIRED`，`D=0`、`C=0`。
3. VIP/普通直推、树奖励、产业基金和团长奖励全部暂停，不创建用户可得流水。
4. 创建可检索的财务对账任务和审计记录。
5. 管理员补齐经审计的成本后，在任何奖励释放前执行幂等重算；不得继续沿用当前“成本缺失按零计算”的逻辑。

## 5. 三条独立路径与资金来源

### 5.1 团长路径

```text
团长逐单佣金 = C × 团长直接利润分成率
```

逐单金额支付成功后冻结，继续受确认收货、售后窗口、退款冲回、幂等键和 Serializable 事务保护。

月度资格继续按直接客户净 GMV、有效客户数、新客户数和退款率判断。月度奖金按每个订单自己的利润及配置版本分段累加：

```text
订单月度最高预留 = C ×（该订单快照内全部可叠加月度利润率之和）
订单实际月度奖金 = C ×（最终档位命中的该订单快照费率之和）
```

不能用月末最新配置回算整月订单。

### 5.2 VIP/普通路径

VIP 或普通路径仍按各自七分配置分配 `D`。一个订单只选择一条买家路径：

```text
树奖励 = D × 买家路径奖励率
产业基金 = D × 买家路径产业基金率
```

直接推荐奖仍按直接推荐人支付时层级选择费率：

```text
直接推荐奖 = D × 直接推荐人层级对应直推率
```

没有有效直接推荐人时，不产生对外直推奖励，对应金额留在平台留存资金。

### 5.3 团长奖金不得成为第八份额

VIP/普通七分名义比例继续合计 100%，团长配置仍完全独立，但团长金额必须有平台资金来源。

平台留存资金定义为七分中实际仍归平台控制的部分：平台占比、慈善、科技、备用金，以及因为没有有效直接推荐人而未对外发放的直推份额。产业基金、树奖励、有效直接推荐奖属于对外资金。

每个订单创建一条平台留存资金汇总及团长资金占用：

```text
平台留存资金 R = D - 树奖励 - 产业基金 - 实际对外直推奖励

团长资金占用 F = 团长逐单佣金 + 团长月度最高预留

必须满足 F <= R

平台最终留存 = R - F + 月度未使用预留返还
```

团长流水仍写入独立 `CaptainAccount/CaptainCommissionLedger`；同时写入等额 `PLATFORM_RETAINED_FUNDING` 资金占用记录。七分流水、团长流水和平台资金占用净额合计必须严格等于 `D`，不得重复创造余额。

## 6. 月度预留、结算与跨版本规则

1. 每笔团长订单支付时，根据该订单规则快照冻结逐单佣金并预留最高月度奖金。
2. 月度预留只占用平台留存资金，不计入团长可见余额。
3. 月末先按整月净 GMV、有效客户、新客和退款率确定资格及档位。
4. 按配置版本对订单分组，每组使用订单支付时快照费率计算实际月度奖金。
5. 资格和最终档位先按该配置批次快照中的门槛分别判断；不能用月末最新门槛判断旧订单批次。
6. 实际奖金不得超过该订单剩余月度预留；差额返还平台留存资金。
7. 同月修改配置只影响修改生效后的新支付订单，不能让高逐单旧配置与高月度新配置同时作用于同一订单利润。
8. 月度结算、预留释放和退款冲回必须具有订单级幂等键。

## 7. 后台硬校验

### 7.1 触发边界

以下写操作必须在实际写入前调用同一个 `ProfitSafetyValidator`：

1. 保存、启用或停用团长配置。
2. 单项或批量保存 VIP 七分比例。
3. 单项或批量保存普通七分比例。
4. 修改 `MARKUP_RATE`、履约成本率、风险预留率或目标净利率。
5. 修改 `VIP_DISCOUNT_RATE` 或平台级消费积分最高抵扣比例。
6. 回滚包含上述参数的配置版本或审计日志。
7. 任意普通商品 SKU 的价格、成本、状态、类目或商户归属变化。
8. 新商品进入 VIP/普通分润范围，或进入已启用的团长类目、商品、商户范围。

平台红包活动本身的预算安全属于 Coupon 系统；本设计保证红包实际抵扣后奖励基数同步下降，不承诺一个脱离预算的红包活动本身仍能满足目标净利。

订单支付和确认收货不再执行配置安全裁决，只计算不可变利润快照并读取已验证规则。

### 7.2 单一有效毛利率

配置校验先应用会自动作用于商品的全局优惠（当前至少包括 VIP 折扣），再使用同一个有效毛利率 `g`。用户自主选择的红包、消费积分和团购返还余额会在实际利润快照中降低 `D`，但不作为静态 SKU 毛利参数重复扣减。

```text
标准优惠后收入倍率 = MARKUP_RATE × 最低强制成交折扣率
标准优惠后毛利率 = (标准优惠后收入倍率 - 1) / 标准优惠后收入倍率

SKU 最低强制成交价 = price × 该 SKU 适用的最低强制成交折扣率
SKU 优惠后毛利率 = (SKU 最低强制成交价 - cost) / SKU 最低强制成交价

g = min(标准优惠后毛利率, 所有活跃普通商品 SKU 优惠后毛利率)

d = g - 履约成本率 - 风险预留率 - 目标净利率
```

`d <= 0` 时，任何对外奖励比例或团长项目都不能启用；错误必须直接显示基础单位经济模型没有可分润利润。

缺失成本、售价不大于成本或无法确定范围时必须失败关闭。不得按成本为零通过后台校验。

### 7.3 四种最坏组合

团长配置的最高利润率：

```text
captainMax =
  directProfitRate
  + baseManagementProfitRate
  + growthBonusProfitRate
  + cultivationBonusProfitRate
  + performanceBonusProfitRate
```

四种真实组合：

| 买家路径 | 直接推荐人层级 | 外部资金比例（相对 D） |
| --- | --- | --- |
| VIP | VIP | VIP 奖励率 + VIP 产业基金率 + VIP 直推率 + captainMax |
| VIP | 普通 | VIP 奖励率 + VIP 产业基金率 + 普通直推率 + captainMax |
| 普通 | VIP | 普通奖励率 + 普通产业基金率 + VIP 直推率 + captainMax |
| 普通 | 普通 | 普通奖励率 + 普通产业基金率 + 普通直推率 + captainMax |

平台、慈善、科技和备用金不计入外部资金；产业基金计入。

每种组合必须同时满足：

```text
外部资金比例 <= 100%
captainMax <= 100% - 树奖励率 - 产业基金率 - 实际直推率
captainMax <= maxTotalIncentiveProfitRate
```

第一条保证 `D` 内资金守恒，第二条保证团长资金全部来自平台留存，第三条保留团长项目自己的独立封顶。

### 7.4 配置并发安全

所有相关单项更新、批量更新、版本回滚、商品/SKU 更新必须：

1. 进入 Serializable 事务。
2. 获取同一个 PostgreSQL 事务级 advisory lock，例如 `pg_advisory_xact_lock(hashtext('profit-safety-config-v1'))`。
3. 在锁内读取完整当前快照并合并候选修改。
4. 在锁内完成 `ProfitSafetyValidator` 校验。
5. 只在校验通过后写入配置或商品，并创建版本/审计记录。

这样并发修改团长、VIP、普通比例和商品成本时，不能分别读取旧状态后同时通过。

### 7.5 纯函数接口

`OrderProfitSnapshotCalculator`：

```text
输入：订单金额字段、订单项、支付时 SKU 成本、优惠明细、单位经济配置、团长范围
输出：READY 利润快照，或 RECONCILIATION_REQUIRED + 稳定错误码
保证：分级金额守恒、D >= 0、0 <= C <= D、相同输入得到相同结果
```

`ProfitSafetyValidator`：

```text
输入：完整候选 RuleConfig、活跃 SKU 最低毛利、候选商品变更
输出：四场景测算和安全摘要；不安全时抛 CAPTAIN_PROFIT_SAFETY_VIOLATION
保证：无数据库写入、无缓存副作用、可独立单元测试
```

## 8. 团长配置 V3 与切换

`CAPTAIN_SEAFOOD_CONFIG` 升级为 schemaVersion 3，完整保留项目、范围、订单、资格、档位、税务和风控字段，并替换费率字段：

```text
perOrderCommission.directProfitRate

monthlyRewards.baseTierGmv
monthlyRewards.baseManagementProfitRate
monthlyRewards.growthTierGmv
monthlyRewards.growthBonusProfitRate
monthlyRewards.excellentTierGmv
monthlyRewards.cultivationBonusProfitRate
monthlyRewards.performanceBonusProfitRate

unitEconomics.fulfillmentCostRate
caps.maxTotalIncentiveProfitRate
caps.targetNetProfitRate
caps.coldChainRiskReserveRate
```

原 15.5% 是 V2 的销售额激励上限，不得直接解释为 V3 利润率上限。V3 初始值可以在后台展示 V2 数字作为编辑建议，但 `enabled=false`，必须由管理员明确确认并保存一份通过硬校验的 V3 配置。

V3 只能从自然月第一天 00:00 起生效。部署发现启用中的 V2 时，立即暂停新的团长订单归因，保留历史 V2 订单的冻结、释放、退款和该月旧结算；管理员配置的 V3 从下一自然月开始创建新归因。一个 `captain/month/program` 结算不得同时包含 V2 销售额订单和 V3 利润订单。

自然月边界只约束首次 V2→V3 切换。V3 生效后的后续安全配置可以按管理员指定时间生效，新订单按第 6 节的配置批次规则使用新快照，旧订单继续使用原批次。

新设计明确取代一层团长旧规格中“11%销售额直接佣金”和“15.5%销售额总封顶”，不改变一层直接关系、无二级和历史审计规则。

所有配置版本回滚入口，包括 RuleVersion 回滚和审计日志回滚，都必须拒绝恢复启用状态的 V2、缺字段 V3 或未通过安全校验的快照。历史 `RuleVersion` 若不是完整配置快照则直接拒绝全局回滚，不得与当前值猜测合并；单字段审计回滚必须把该字段旧值合并到当前完整快照后，在共享锁内重新校验并原子写入。

## 9. 退款和跨月冲回

### 9.1 退款利润比例

退款先维护每个订单项的累计退款实收和累计已冲回利润。退款明确到订单项和数量时：

```text
累计冲回比例 = min(1, 累计退款数量 / 原购买数量)
```

只有退款金额、没有明确数量时：

```text
累计冲回比例 = min(1, 累计商品退款金额 / 订单项原始实收)
```

每次只冲回增量：

```text
累计目标利润冲回 = 订单项原始可分润份额 × 累计冲回比例
本次利润冲回 = 累计目标利润冲回 - 历史已冲回利润
```

负贡献订单项的原始可分润份额为零，不产生奖励冲回。累计冲回不得超过订单项原始可分润份额；所有订单项累计冲回不得超过原订单 `D`，团长适用项累计冲回不得超过原订单 `C`。每次按分计算，最后一次退款吸收剩余尾差。

### 9.2 奖励与预留

退款后先得到剩余 `D`、`C`，再使用支付时规则快照计算每类流水的累计目标存量。VIP/普通直推、树奖励、卖家产业基金、团长逐单佣金、团长月度预留和 `PLATFORM_RETAINED_FUNDING` 全部按“累计目标存量 - 当前存量”生成本次增量冲回，保证退款后净流水仍合计为剩余 `D`。

1. 冻结期退款，减少对应冻结流水和月度预留。
2. 已可用但未提现/未支付时，扣回可用余额并创建负向流水。
3. 月度奖励已批准但未支付时，冲销后重新审核。
4. 已提现或已支付后退款，相关账户创建 `CLAWBACK_PENDING`；团长不足部分作为下月负向结转，卖家产业基金及用户奖励进入各自追偿流程，不得直接写出负账户余额。
5. 平台留存资金按其他流水实际冲回结果同步返还或扣减。
6. 退款、冲回和负向结转使用 `orderId + refundId + sourceLedgerId` 幂等键。

月末仍存在 `RECONCILIATION_REQUIRED` 的订单时，这些订单不计入奖金草稿，但后台必须展示待处理数量和金额；该团长当月结算不得审核通过。财务补齐成本并完成幂等重算后，系统重新生成月度指标和草稿，再进入审批。已批准或已支付结算不得通过事后补录偷偷并入新奖励，只能走显式补差或追偿流程。

## 10. 管理后台

团长配置页面继续独立存在：

1. “直接佣金率”改为“直接利润分成率”。
2. 四个月度奖励明确标注“按直接客户可分润利润计提”。
3. 展示 `MARKUP_RATE`、最低实际毛利率、履约、风险、目标净利和基准 `d`。
4. 展示四种最坏组合、平台留存比例、团长最大可配置率和剩余空间。
5. 每个字段继续提供问号说明；关联字段说明共同参与哪条公式。
6. 保存失败展示冲突场景、当前比例、允许上限、缺口和受影响 SKU，不只显示“参数不合法”。

VIP、普通配置页保存失败时，明确提示该修改与团长配置或商品利润底线冲突，并提供团长配置页入口。

## 11. 错误和审计

稳定错误码：

```text
CAPTAIN_PROFIT_SAFETY_VIOLATION
ORDER_PROFIT_COST_MISSING
ORDER_PROFIT_CONSERVATION_FAILED
ORDER_PROFIT_RECONCILIATION_REQUIRED
CAPTAIN_CONFIG_V2_NOT_ACTIVE
```

安全校验失败不得写 RuleConfig、RuleVersion、商品或缓存。成功配置版本保存四场景安全摘要和最低毛利证据。利润快照、规则快照、平台资金占用、月度预留、释放及退款冲回均保留配置版本和金额前后值。

## 12. 验收标准

1. 团长、VIP、普通关系、配置、账户和业务流水保持独立。
2. 团长新逐单和月度奖励全部按可分润利润计算。
3. VIP/普通直推、树奖励和产业基金扣除全部商品优惠后计算。
4. 订单项和订单级金额按分守恒，亏损项不会被逐项截断后忽略。
5. 支付直推、收货树奖励、团长逐单和月度奖励共用不可变利润及规则快照。
6. 团长逐单和月度最高预留全部由平台留存资金覆盖，所有净流水合计不超过 `D`。
7. 四种买家路径/推荐人层级组合全部覆盖。
8. 配置、回滚、商品价格/成本变更在同一共享锁下硬校验，不能并发绕过。
9. 订单结算不动态改比例；实际金额只由支付快照和对应配置版本决定。
10. V2 不会被静默当成利润比例，新订单在安全 V3 生效前暂停团长归因，历史流水继续完成。
11. 部分退款、跨月退款和已支付奖金冲回资金守恒且幂等。
12. 运行时成本异常不阻断支付，但奖励失败关闭并进入可恢复对账状态。
13. 自动化测试包含按分优惠容量重分配、正负贡献混合订单、连续部分退款、四场景配置组合及所有流水净额等于 `D` 的黄金向量。

本设计保证推广分润和产业基金不会额外穿透已预留的履约、风险和目标平台净利。平台红包活动本身的预算、税费和实际冷链账单仍需由各自系统及财务规则控制。
