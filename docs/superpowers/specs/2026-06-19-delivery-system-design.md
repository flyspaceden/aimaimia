# 配送系统设计方案

> 状态：设计已确认，待实施计划
> 创建时间：2026-06-19
> 适用范围：买家 App / 配送管理后台 / 配送中心 / 后端 / 配送数据库 / Prisma / 部署
>
> **For agentic workers:** 本文档是配送系统第一版的权威设计来源。配送不是新 App，也不是新品牌系统；它是在现有爱买买框架下复制出一条独立配送业务线。大部分业务逻辑直接复用当前 App、管理后台、卖家中心和后端成熟逻辑，只在数据隔离、账号隔离、主题、路由、模块删减和配送特有能力上做改造。

## 1. 背景与目标

平台需要新增一套面向单位和大批量购买场景的配送系统。用户不希望重新申请新的 App、新主域名、新备案或新的审核流程，因此配送系统应嵌入现有爱买买技术体系：

- 买家端仍使用同一个爱买买 App。
- 管理端和商家端使用新子域名，但不新建主域名体系。
- 后端仍使用同一个 NestJS API 进程和同一个 API 域名。
- 配送业务数据使用独立数据库，和现有普通商城完全隔离。

目标是复制一套现有爱买买电商能力给配送业务使用，同时删除配送不需要的模块，并增加配送单位、灵活加价、配送清单 PDF/Excel、配送商家结算等特有能力。

## 2. 总体原则

### 2.1 复用原则

能复制当前系统逻辑的地方直接复制当前逻辑，不重新设计一套电商系统。

默认复制：

- App 商品浏览、商品详情、购物车、结算、支付、订单、物流展示。
- 管理后台工作台、用户管理、商品审核、订单管理、客服、配置、审计等通用交互。
- 卖家中心商品草稿、商品提交、审核流、库存、订单、发货、员工权限等通用逻辑。
- 支付宝、微信支付、顺丰、OSS、SMS 等基础设施对接。

只改以下内容：

- 数据源从主业务数据库改为配送数据库。
- 路由前缀改为 `delivery` / `delivery-admin` / `delivery-seller`。
- 账号、Token、权限、审计独立。
- 删除配送不需要的 VIP、红包、消费积分、分润、退款退货等模块。
- 增加配送单位、配送定价规则、配送清单 PDF/Excel、配送结算等特有能力。
- 配送 App 和配送中心使用橙色主题，配送管理后台使用浅蓝色主题。

### 2.2 隔离原则

配送系统的数据、账号、权限和业务记录必须和现有普通商城隔离：

| 范围 | 隔离要求 |
|---|---|
| 买家账号 | 配送用户不和普通 App 用户账号打通 |
| 管理员账号 | 配送管理后台账号不和爱买买管理后台账号打通 |
| 商家账号 | 配送中心账号不和爱买买卖家中心账号打通 |
| 商品 | 配送商品不进入普通商城，普通商品不进入配送商城 |
| 订单 | 配送订单不进入普通订单体系 |
| 数据库 | 配送业务写独立配送数据库 |
| 权益 | 配送订单不计入 VIP、红包、消费积分、数字资产、分润 |
| 客服 | 配送客服数据独立 |
| 审计 | 配送审计日志独立 |

## 3. 业务场景边界

普通爱买买商城用于普通用户小量购买，继续保留现有 VIP、红包、消费积分、分润、数字资产、普通订单和售后退款体系。

配送系统用于单位或批量购买场景，典型使用者包括学校、食堂、门店、项目单位等。配送系统强调：

- 先选择配送单位，再下单。
- 商品可设置起订量和购买步长。
- 后台按商家、商品、SKU、数量阶梯灵活设置最终售价。
- 付款后生成配送清单 PDF/Excel。
- 配送中心按子订单履约和导出商家结算清单。

## 4. 三端入口与系统切换

### 4.1 买家 App

配送 App 指现有买家 App 内部的配送模块，不是单独下载安装的新 App。

入口：

```text
正常爱买买 App
我的 > 常用工具 > 配送
进入配送模块
```

进入配送模块后：

```text
未登录配送账号 -> 配送注册/登录
已登录但没有配送单位 -> 创建配送单位
已登录且已有配送单位 -> 选择当前配送单位后进入配送商品页
```

配送模块返回普通爱买买平台：

```text
配送模块
我的 > 常用工具 > 返回爱买买平台
回到正常爱买买 App
```

### 4.2 管理后台

```text
爱买买管理后台登录页
按钮：切换配送管理后台
跳转到 delivery-admin.ai-maimai.com 登录

配送管理后台登录页
按钮：切换爱买买管理后台
跳转到 admin.ai-maimai.com 登录
```

两边只做跳转，不做单点登录，不共享登录态。

### 4.3 配送中心

展示名称统一为“配送中心”。技术命名仍使用已确认的 `delivery-seller`。

```text
爱买买卖家中心登录页
按钮：切换配送中心
跳转到 delivery-seller.ai-maimai.com 登录

配送中心登录页
按钮：切换爱买买卖家中心
跳转到 seller.ai-maimai.com 登录
```

两边只做跳转，不共享登录态、账号或数据。

## 5. API 与域名

所有前端最终仍调用同一个 API 域名：

```text
生产：https://api.ai-maimai.com/api/v1
测试：https://test-api.ai-maimai.com/api/v1
```

配送 API 路由前缀：

```text
买家 App 配送模块  -> /api/v1/delivery/*
配送管理后台       -> /api/v1/delivery-admin/*
配送中心           -> /api/v1/delivery-seller/*
```

前端域名：

```text
配送管理后台生产：delivery-admin.ai-maimai.com
配送中心生产：delivery-seller.ai-maimai.com

配送管理后台测试：test-delivery-admin.ai-maimai.com
配送中心测试：test-delivery-seller.ai-maimai.com
```

需要在阿里云添加 DNS 解析，并在宝塔配置站点、反代或静态目录、SSL。API 不新增独立域名。

## 6. 数据库与 Prisma

配送使用独立数据库，建议在同一台 PostgreSQL/RDS 实例中新建独立 database 和独立数据库账号。后续配送业务量变大时再迁移到独立 RDS 实例。

技术结构：

```text
backend/prisma-delivery/schema.prisma
backend/prisma-delivery/migrations/
独立 Delivery Prisma Client
独立 DeliveryPrismaService
环境变量 DELIVERY_DATABASE_URL
```

主业务 Prisma 继续服务普通爱买买业务，配送模块不得通过主 Prisma 读写配送业务数据。

配送数据库保存：

- 配送用户
- 配送单位
- 配送地址
- 配送管理员
- 配送商家
- 配送中心员工和角色
- 配送商品、SKU、分类、单位
- 配送价格规则
- 配送购物车
- 配送订单、子订单、订单项
- 配送支付记录
- 配送物流和顺丰记录
- 配送清单 PDF/Excel 和模板配置
- 配送客服、会话、工单、FAQ、快捷回复
- 配送结算
- 配送配置
- 配送审计日志

## 7. 编号规则

配送数据库内使用可读业务编号作为主要展示编号，并可作为配送业务表主键或唯一业务主键。编号由数据库序列或等价原子计数机制生成，必须并发安全，不回收、不复用。

建议格式：

| 对象 | 编号格式示例 |
|---|---|
| 配送用户 | `PSYH0000000000001` |
| 配送商家 | `PSSJ0000000000001` |
| 配送商品 | `PSSP0000000000001` |
| 配送订单 | `PSDD0000000000001` |
| 配送子订单 | `PSZDD000000000001` |
| 配送支付 | `PSZF0000000000001` |
| 配送清单 | `PSQD0000000000001` |

编号前缀使用拼音首字母，便于业务人员一眼识别。

## 8. 账号、权限与认证

### 8.1 配送用户

配送用户不和普通 App 用户账号打通。配送用户使用手机号/微信注册登录，数据存配送数据库。

第一版不支持配送用户自助注销；后台可禁用/冻结配送用户。

### 8.2 配送单位

一个配送用户可管理多个配送单位，并在配送模块中切换当前配送单位。

规则：

- 首次进入配送模块，如果没有配送单位，必须先创建配送单位。
- 配送单位第一版不审核，创建后可直接下单。
- 第一版不做配送单位多人协作或邀请成员。
- 配送管理后台可禁用、冻结、备注配送单位。
- 购物车、订单、清单按配送用户和当前配送单位隔离。

配送单位固定履约字段：

- 单位名称
- 联系人姓名
- 联系人手机号
- 省 / 市 / 区
- 详细地址

地址选择复用当前 App 地址逻辑，省市区可选择，详细地址手填。

配送管理后台可配置配送单位扩展字段：

- 字段名称
- 排序
- 是否显示
- 是否必填
- 是否在 App 创建单位页显示
- 是否在配送管理后台显示
- 是否进入 PDF/Excel

### 8.3 配送管理后台

配送管理后台账号完全独立：

- 独立 `DeliveryAdminUser`
- 独立角色和权限
- 独立 Token
- 独立审计日志
- 独立登录页

建议环境变量：

```text
DELIVERY_ADMIN_JWT_SECRET
```

### 8.4 配送中心

配送中心账号完全独立：

- 独立配送商家
- 独立员工账号
- 独立角色权限
- 独立 Token
- 独立登录页

展示名称为“配送中心”。大部分逻辑复制现有卖家中心。建议环境变量：

```text
DELIVERY_SELLER_JWT_SECRET
```

配送中心登录页提供“申请入驻”。申请进入配送管理后台审核；审核通过后创建配送商家并创建或绑定 OWNER 账号。

建议配送中心角色沿用现有卖家中心逻辑：

- OWNER
- MANAGER
- OPERATOR

### 8.5 配送 App 用户 Token

建议环境变量：

```text
DELIVERY_USER_JWT_SECRET
```

配送用户、配送管理员、配送中心三套认证互相独立，也和现有普通 App、admin、seller 认证独立。

## 9. 配送 App 功能范围

配送模块第一版只保留两个底部 Tab：

```text
商品
我的
```

不做：

- 独立首页
- Banner 推荐区
- 分类金刚区
- 独立消息中心
- 底部购物车 Tab

### 9.1 商品 Tab

“商品”复用现有发现页、商品列表和商品详情基础逻辑，但页面名称改为“商品”。

第一版能力：

- 配送商品分类
- 配送商品搜索
- 配送商品列表
- 配送商品详情
- SKU 选择
- 起订量校验
- 购买步长校验
- 加入配送购物车
- 立即购买/结算

第一版不接：

- AI 搜索
- 标签筛选
- 溯源
- VIP 商品
- 奖品/抽奖
- 红包/优惠券
- 消费积分抵扣
- 数字资产
- 分润

### 9.2 购物与订单

配送购物流程复制现有普通商城流程：

```text
浏览配送商品
选择 SKU 和数量
加入配送购物车
购物车结算
选择配送单位
选择或填写收货地址
填写配送备注
支付宝/微信支付
支付成功后生成配送订单
生成配送清单 PDF/Excel
查看配送订单和物流
```

购物车按配送用户和当前配送单位隔离。

多商家结算逻辑和现有商城一致：

- 一个配送购物车可包含多个配送商家商品。
- 用户一次结算、一次支付。
- 支付成功后按配送商家拆成配送子订单。

配送订单支持再次购买：

- 只能复购当前配送单位下的配送订单。
- 仍可售且库存足够的商品重新加入当前配送单位购物车。
- 不可售或库存不足的商品给出提示。
- 价格按复购时最新配送价格重新计算，不沿用历史锁价。

配送下单支持“配送备注”，不支持期望送达日期或时间段。

### 9.3 我的页

配送“我的”页基本复制现有“我的”页结构，删除不属于配送的入口。

删除：

- VIP
- 红包/优惠券
- 消费积分/钱包
- 数字资产
- 推荐/分润
- 抽奖/奖品
- 普通商城订单入口

保留或新增：

- 配送单位
- 配送订单
- 配送清单
- 配送客服
- 收货地址
- 账号与安全
- 常用工具：返回爱买买平台

### 9.4 主题

配送 App 使用橙色主题：

```text
主色：#F97316
强调深橙：#EA580C
浅背景：#FFF7ED
```

页面结构可以复制现有 App，但视觉上要让用户一眼识别当前处于配送模块。

## 11. 配送中心功能范围

配送中心是现有爱买买卖家中心的配送版。展示名称统一叫“配送中心”，但技术域名和 API 使用 `delivery-seller`。

```text
独立域名：delivery-seller.ai-maimai.com
独立登录
独立配送商家
独立员工账号和角色权限
独立配送数据库
API：/api/v1/delivery-seller/*
```

配送中心大部分业务逻辑直接复制现有卖家中心。不同点仅包括：

- 数据源改为配送数据库。
- 登录体系独立。
- 展示名称改为“配送中心”。
- 主题色改为橙色。
- 删除售后、退款、退货相关模块。
- 严格限制最终售价、加价率和平台利润不可见。

第一版菜单：

- 工作台
- 商品管理
- 订单管理
- 发货 / 物流
- 库存管理
- 客服 / 工单
- 财务结算
- 企业信息
- 员工与权限
- 账号安全

删除：

- 售后
- 退款
- 退货
- 普通商城专属能力

工作台指标：

- 待发货订单
- 今日订单
- 商品审核状态
- 库存预警
- 待结算金额
- 已结算金额

- 自己的库存
- 自己的订单数量
- 自己的供货小计
- 自己的可结算/已结算金额

配送中心不可见：

- 平台最终售价
- 平台加价率
- 平台利润
- 用户完整支付金额
- 其他配送商家的数据

## 12. 商品、定价与库存

### 12.1 商品体系

配送商品体系独立于现有普通商品体系：

- 普通商品不进入配送商城。
- 配送商品不进入普通商城。
- 配送商品、SKU、分类、单位、库存全部存在配送数据库。

分类和商品单位第一版可以从现有分类/单位复制一份初始数据，之后不再同步。

商品上架流程基本复制现有卖家中心和管理后台：

```text
配送中心创建商品草稿
编辑商品和 SKU
填写供货价/成品价
填写库存
填写重量
填写起订量和购买步长
提交审核
配送管理后台审核
审核通过后配送 App 可见
驳回后配送中心修改再提交
支持上下架
```

草稿逻辑按现有卖家中心复制：

- 支持自动保存。
- 草稿不进入审核列表。
- 草稿不对用户可见。
- 草稿数量限制按现有逻辑复制。

第一版不做：

- 视频
- AI 搜索
- 标签筛选
- 溯源
- 批量导入商品/SKU
- 分类加价率
- 配送单位专属价
- 询价模式

### 12.2 起订量和购买步长

配送 SKU 支持：

```text
minOrderQuantity：起订量
orderStepQuantity：购买步长
```

用户购买数量必须大于等于起订量，并符合购买步长。

### 12.3 定价

配送中心只填写供货价/成品价。配送用户看到的最终售价由配送管理后台规则计算。

定价优先级：

```text
SKU固定售价
> SKU数量阶梯加价率
> 商品级数量阶梯加价率
> 商家级数量阶梯加价率
> 商家默认加价率
> 平台默认加价率
```

规则说明：

- SKU 固定售价直接指定该 SKU 的平台最终销售价。
- 数量阶梯加价率按购买数量匹配不同加价率。
- 商家默认加价率是该商家全部商品默认规则。
- 平台默认加价率是兜底规则。

定价变更规则：

- 价格规则变更只影响新订单。
- 历史订单保留支付时的价格快照。
- 所有价格规则变更写入审计日志。
- 第一版不做价格变更审批流。
- 第一版不支持人工改订单价格。

金额存储和计算建议使用整数分/cents，前端展示为元并保留两位小数。

### 12.4 库存

库存逻辑：

- 配送中心管理库存。
- 配送管理后台只查看库存和库存流水，不直接改库存。
- 付款成功创建配送订单时扣库存。
- 未付款不扣库存。
- 库存不允许扣成负数。

如果支付成功但扣库存、建单或清单生成失败：

- 进入配送管理后台“支付异常”。
- 后台人工处理。
- 不自动退款。

库存流水记录：

- 配送中心调整库存。
- 付款成功扣减库存。
- 订单异常处理。
- 后台修正记录。

顺丰发货用的 SKU 重量字段直接复制现有系统逻辑，只要求 `weightGram`，不新增长宽高、体积重或顺丰运输类型选择。

## 13. 订单、支付、顺丰与结算

### 13.1 订单流程

配送订单流程整体复制现有商城订单流程：

```text
配送购物车
生成配送 CheckoutSession
锁定商品价格、运费、地址、配送单位、备注快照
支付宝/微信支付
支付回调成功
创建配送订单和配送子订单
扣减配送库存
生成配送清单 PDF/Excel
```

多商家订单：

- 用户一次支付。
- 后台按配送商家拆分配送子订单。
- 配送用户看到整单和分组明细。
- 配送管理后台看到整单和所有子订单。
- 配送中心只看到自己的子订单。

配送订单状态第一版简化为：

- 已支付 / 待发货
- 已发货
- 已签收 / 已送达
- 已完成
- 已取消

取消规则：

- 只有配送管理后台可以取消未发货配送订单或子订单。
- 必须填写取消原因。
- 配送用户不能自助取消。
- 配送中心不能自助取消。
- 第一版不做自动退款。

自动确认收货天数默认按配置控制。如果存在顺丰签收事件，从签收时间开始计算；否则从发货时间开始计算。

### 13.2 支付

支付宝和微信支付全部复用现有配置和现有逻辑，不重新申请支付应用，不重新设计支付链路。

隔离点：

- 配送订单号独立。
- 配送支付单号独立。
- 配送支付记录写配送数据库。
- 支付回调识别配送支付单后进入配送支付模块。
- 普通订单和配送订单不混。

支付记录需要保存：

- 支付渠道
- 商户订单号
- 渠道交易号
- 支付时间
- 支付金额
- 回调摘要或必要原始摘要
- 异常处理状态

### 13.3 顺丰

顺丰逻辑完全复制现有系统：

- 复用现有顺丰配置。
- 复用顺丰月结账号。
- 复用发货、面单、轨迹、对账记录逻辑。
- 不新增大件计费。
- 不新增体积重。
- 不新增运输类型选择。

运费规则复制现有平台顺丰风格计费：

- 平台按规则计算配送用户应付运费。
- 支付时锁定运费。
- 顺丰实际月结成本用于后台记录和对账。
- 不向配送用户二次补收差价。

### 13.4 结算

平台向配送用户收取最终销售价，配送商家按供货价/成品价结算，平台保留加价差额。

配送商家结算进入可结算的时机：

```text
配送订单确认收货 / 自动完成后
对应配送商家的供货金额进入可结算
```

配送管理后台支持：

- 查看待结算金额。
- 查看已结算金额。
- 按商家导出结算单。
- 手动标记已结算。
- 记录结算操作日志。

第一版不做：

- 商家自动提现。
- 平台自动打款。
- 退款退货售后资金链路。

配送中心可导出自己的财务清单，只显示自己可见的供货价、供货小计、可结算金额、已结算金额，不显示平台最终售价、加价率和平台利润。

### 14.5 模板配置和版本

每套模板支持：

- 列名
- 排序
- 是否显示
- 固定列保护
- 是否导出 PDF
- 是否导出 Excel

后台可以给某一笔配送单手动添加：

- 自定义列
- 自定义内容

版本规则：

- 付款成功自动生成 v1。
- 后台单笔重新生成 v2/v3。
- 用户默认看到最新版本。
- 后台可查看历史版本。
- 历史 PDF/Excel 不物理删除。
- 版本记录生成时间、操作人、原因。

第一版只支持单笔重新生成，不做批量重新生成。

后台可上传公司章或页脚章图片，生成 PDF 时放入模板指定位置。第一版不做 CA 电子签章。

文件存储复用现有 OSS bucket，但配送文件必须存入 `delivery/` 前缀，例如：

```text
delivery/products/
delivery/manifests/
delivery/chat/
delivery/waybills/
```

配送模块不能读写非 `delivery/` 前缀文件。

## 16. 技术落地结构

整体采用“同仓库、同后端进程、独立配送数据库”的方案。

```text
买家 App
爱买买管理后台
爱买买卖家中心
配送管理后台
配送中心
        ↓
同一个 API 域名
https://api.ai-maimai.com/api/v1
        ↓
同一个 NestJS 后端进程
        ↓
主业务数据库 + 配送数据库
```

后端模块建议按配送边界拆分：

```text
backend/src/modules/delivery/
  buyer/
  admin/
  seller/
  auth/
  users/
  units/
  merchants/
  products/
  pricing/
  cart/
  checkout/
  orders/
  payments/
  shipping/
  manifests/
  settlement/
  customer-service/
  audit/
  config/
```

可以复用现有服务逻辑，但要避免直接依赖主业务表。支付、顺丰、OSS、SMS 这些基础设施可以复用适配器；配送业务数据必须写配送数据库。

前端落地：

```text
买家 App：
  app 内新增配送模块路由
  我的 > 常用工具 > 配送
  配送模块两个 Tab：商品 / 我的
  主题：橙色

配送管理后台：
  新增 delivery-admin 前端构建目标或独立入口目录
  复制现有 admin 结构
  删除不需要模块
  加入配送特有模块
  主题：浅蓝色

配送中心：
  新增 delivery-seller 前端构建目标或独立入口目录
  复制现有 seller 结构
  展示名称：配送中心
  主题：橙色
```

后端部署仍然是当前 PM2 后端服务，但新增配送数据库环境变量和配送 Prisma migration 流程。

## 18. 安全与一致性

涉及支付、库存、订单创建、价格快照、结算、状态转换的操作必须使用事务和并发保护。配送付款回调、建单、扣库存、生成清单必须幂等。

要求：

- 支付回调重复到达不得重复建单或重复扣库存。
- 支付成功但建单、扣库存、清单生成失败时必须进入异常记录。
- 库存扣减不得扣成负数。
- 历史订单价格快照不得被后续价格规则变更影响。
- 结算标记必须留审计日志。
- 模板变更、自定义列变更、PDF/Excel 重生成必须留审计日志。
- 配送 OSS 文件必须限制在 `delivery/` 前缀内。
- 配送中心的接口和导出文件不得泄露平台最终售价、加价率或平台利润。

实现涉及金额、库存、支付、结算、状态转换时，需要按项目安全规则检查并使用足够强的事务隔离级别。

## 19. 第一版不做

第一版明确不做：

- 新的独立 App
- 新的主域名或新品牌官网
- 普通 App 用户账号打通
- 普通商家账号打通
- 普通管理员账号打通
- 共享普通商城数据库
- VIP
- 红包/优惠券
- 消费积分/钱包
- 数字资产
- 推荐/分润
- 抽奖/奖品
- 退款/退货/售后退款流程
- AI 搜索
- 标签筛选
- 溯源
- 首页 Banner/推荐区
- 独立消息中心
- 配送单位多人协作
- 配送单位审核
- 配送单位专属价
- 分类加价率
- 商品/SKU 批量导入
- 期望送达日期/时间段
- 商家自动提现/自动打款
- CA 电子签章
- PDF/Excel 批量重新生成
- 新的顺丰计费、大件运输、体积重、运输类型

## 20. 验收标准

1. 普通 App“我的 > 常用工具”中有配送入口，配送模块中有返回爱买买平台入口。
2. 配送用户可独立手机号/微信登录，未创建配送单位时必须先创建单位。
3. 配送 App 只有“商品”和“我的”两个主 Tab，主题为橙色。
4. 配送管理后台和配送中心可通过独立域名登录，且不复用现有后台/卖家中心登录态。
5. 配送管理后台主题为浅蓝色，配送中心主题为橙色。
6. 配送数据库独立，配送业务数据不写入主业务数据库。
7. 配送商品、购物车、订单、支付、物流、客服、清单、结算都走配送数据库。
8. 配送中心只能看到自己的供货价、订单、履约和结算数据，不能看到平台最终售价、加价率和平台利润。
9. 配送定价优先级符合 `SKU固定售价 > SKU数量阶梯加价率 > 商品级数量阶梯加价率 > 商家级数量阶梯加价率 > 商家默认加价率 > 平台默认加价率`。
10. 支付宝、微信支付、顺丰逻辑复用现有配置和现有逻辑。
11. 付款成功后生成用户完整清单 PDF/Excel，后台可单笔重生成并保留版本。
12. 配送中心履约 PDF 不包含任何金额字段。
13. 配送中心财务导出只包含商家自己可见的供货价和结算金额，不包含平台最终售价、加价率或平台利润。
14. 配送订单不计入现有 VIP、红包、消费积分、数字资产、分润。
15. 配送系统有独立协议、审计日志、基础统计和测试 seed。
