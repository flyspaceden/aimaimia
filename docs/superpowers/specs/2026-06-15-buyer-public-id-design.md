# 买家公开编号设计方案

## 1. 背景

当前 `User.id` 使用 Prisma `cuid()`，例如 `cmqc65zt2003rt7ki4i0e89cx`。该字段是系统内部主键，被订单、售后、客服、分润树、推荐关系、数字资产、卖家员工、JWT 会话等大量数据引用。直接改写 `User.id` 会触发全库主键和外键迁移，风险过高。

本方案新增买家公开编号 `buyerNo`，用于 App、管理后台、卖家中心的业务展示、复制和搜索。内部主键 `User.id` 保持不变，仅用于数据库关系、鉴权和技术排查。

## 2. 目标

1. 只给买家用户生成公开编号。
2. 编号格式固定为 `AIMM` + 14 位数字，例如 `AIMM00000000000001`。
3. 历史买家按注册时间从 1 开始回填：`User.createdAt ASC, User.id ASC`。
4. 新买家在买家端注册或首次自动注册时生成下一个编号。
5. App、管理后台、卖家中心面向业务人员的用户识别优先使用 `buyerNo`。
6. 管理后台和卖家中心支持输入 `AIMM...` 定位对应买家数据。

## 3. 非目标

1. 不修改 `User.id`，不重写现有外键。
2. 不给管理后台 `AdminUser` 生成 `buyerNo`。
3. 不给纯卖家员工账号生成 `buyerNo`。
4. 不把 `buyerNo` 用作 JWT `sub`、支付回调主键、数据库外键或审计主键。
5. 不因为卖家中心显示 `buyerNo` 而放开买家手机号、真实姓名等隐私信息。

## 4. 数据模型

在 `User` 表新增：

```prisma
buyerNo String? @unique
```

编号规则：

- 正则：`^AIMM\d{14}$`
- 最小值：`AIMM00000000000001`
- 最大值：`AIMM99999999999999`
- 数字部分不复用、不回收；账号注销后原编号仍保留在依法留存的用户记录上，用于历史订单和客服排查。

为避免并发注册重复编号，新增数据库级序列或等价的原子计数机制。推荐使用 PostgreSQL sequence，例如 `buyer_no_seq`，后端在事务内取 `nextval` 后格式化为 `AIMM${n.padStart(14, '0')}`。

## 5. 买家判定

### 5.1 新用户

以下路径生成 `buyerNo`：

- 买家 App 手机号注册：`POST /auth/register`
- 买家 App 手机验证码登录自动注册
- 买家 App 微信首次登录自动注册
- 未来任何买家端新增的自动注册入口

以下路径不生成 `buyerNo`：

- 管理后台账号创建
- 卖家中心员工邀请或创建
- 管理后台绑定商户 OWNER / MANAGER / OPERATOR 时创建的用户

如果同一个手机号或微信账号先作为卖家员工存在，之后首次进入买家 App 并完成买家登录流程，则在买家登录事务中补生成 `buyerNo`。

### 5.2 历史用户回填

历史回填不能只依赖 `MemberProfile`，因为商户员工创建路径也可能写入用户资料。回填脚本应按以下顺序识别历史买家：

1. 有买家订单、购物车、收货地址、发票抬头、优惠券实例、抽奖记录、客服会话、数字资产账户等买家侧业务记录的用户。
2. 通过买家 App 注册/登录路径创建，且没有任何商户员工关系的用户。
3. 对同时具备买家业务记录和卖家员工关系的用户，视为买家，生成 `buyerNo`。
4. 纯商户员工、无买家侧业务记录的用户不回填。

回填排序以 `User.createdAt ASC, User.id ASC` 为准。并发上线前应在生产库克隆环境跑一次 dry-run，输出待回填数量、跳过数量、首尾编号和冲突检查结果。

## 6. 后端 API 设计

### 6.1 返回字段

所有返回买家摘要或买家详情的 DTO 增加 `buyerNo`：

- 买家 App 我的页面资料接口
- 管理后台用户列表和详情
- 管理后台订单、售后、优惠券、抽奖、客服、分润、数字资产等涉及买家的接口
- 卖家中心订单、售后、发货等涉及买家的接口

内部仍返回 `userId` 给需要路由或调试的管理端接口，但业务展示应优先使用 `buyerNo`。

### 6.2 查询输入

管理后台和卖家中心涉及买家查询的接口支持 `buyerNo`：

- 如果输入符合 `^AIMM\d{14}$`，先通过 `User.buyerNo` 解析为内部 `User.id`。
- 如果输入不符合 `buyerNo` 格式，保持原有手机号、昵称、原始 `userId` 查询逻辑。
- 路由参数短期内仍使用内部 `userId`，但搜索框和列表操作可以接受 `buyerNo` 后跳转到解析后的内部路由。

## 7. App 前端

买家 App 的「我的」页调整：

1. 头像右侧原 `profile.level` 展示位改为 `buyerNo`。
2. `buyerNo` 旁增加复制按钮，点击复制完整编号。
3. 复制成功提示：`用户编号已复制`。
4. 若 `buyerNo` 暂未返回，显示 `用户编号生成中`，不展示内部 `User.id`。
5. `新芽会员` 等成长等级不在该位置展示；如后续仍需展示等级，应另开独立 UI 位置。

## 8. 管理后台

管理后台改造原则：

1. 所有当前展示原始 `userId` 的买家场景，增加 `buyerNo` 展示和复制。
2. 原始 `User.id` 保留为「内部ID」，供技术排查使用。
3. 用户列表、用户详情、订单、售后、优惠券、抽奖、客服工作台、分润会员/树、数字资产等页面均需覆盖。
4. 搜索框支持输入 `AIMM...`，能定位到对应买家。
5. 对没有 `buyerNo` 的非买家用户，展示 `-` 或 `非买家账号`，不得伪造编号。

## 9. 卖家中心

卖家中心现有 `buyerAlias` 是店铺维度匿名编号，仍保留。

改造原则：

1. 订单、售后、发货详情可以展示买家 `buyerNo`，用于跨客服和平台协同。
2. 列表页可在 `buyerAlias` 下方展示较弱样式的 `buyerNo`。
3. 卖家中心不展示内部 `User.id`。
4. 卖家中心不因 `buyerNo` 增加而暴露买家手机号、真实姓名等隐私字段。
5. 卖家订单/售后搜索可支持 `AIMM...`，解析后仍受商户数据权限限制，只能查本商户相关订单和售后。

## 10. 安全与一致性

本改动涉及身份标识和管理端检索，不涉及资金金额写入，但会影响分润、订单、客服、数字资产等高价值数据的查询入口。实现时必须：

1. `buyerNo` 生成使用数据库原子机制，避免并发重复。
2. 回填脚本可重复执行，已存在 `buyerNo` 的用户跳过。
3. `buyerNo` 解析为内部 `userId` 后，继续复用原有权限校验，不能绕过商户隔离或管理端权限。
4. 日志和审计建议同时记录 `buyerNo` 与内部 `userId`，便于客服和技术排查关联。
5. 账号注销不清空 `buyerNo`，确保历史订单、售后、发票和审计链路可追溯。

## 11. 验收标准

1. 新买家注册后立即拥有 `AIMM` + 14 位编号。
2. 历史买家回填编号连续、唯一、按注册时间排序。
3. 纯卖家员工和管理员账号没有 `buyerNo`。
4. App「我的」页头像旁显示 `buyerNo`，复制按钮可复制完整编号。
5. 管理后台所有买家 `userId` 展示点同时显示并可复制 `buyerNo`。
6. 管理后台可用 `AIMM...` 搜索用户、订单、售后、优惠券、抽奖、客服、分润、数字资产等相关数据。
7. 卖家中心可显示和搜索 `buyerNo`，但不展示内部 `User.id`，且不突破商户数据权限。
8. 原有登录、订单、支付、推荐、分润、售后、客服、数字资产链路不因新增编号回归失败。

## 12. 实施拆分建议

1. Schema + 生成器 + 历史回填脚本。
2. 买家 Auth 注册/自动注册路径接入。
3. 买家 App 我的页展示和复制。
4. 管理后台用户、订单、售后、优惠券、抽奖、客服、分润、数字资产页面覆盖。
5. 卖家中心订单、售后、发货页面覆盖。
6. 验证与文档同步。
