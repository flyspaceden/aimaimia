# 爱买买 — 版本管理指南

## 一、三个版本概览

| 版本 | 用途 | 谁在用 |
|------|------|--------|
| **开发版本** | 日常写代码、调试功能 | 开发者本人 |
| **测试版本** | 功能验证、Bug 测试、上线前验收 | 开发者 + 测试人员 |
| **上线版本** | 正式对外服务，真实用户使用 | 所有用户 |

## 二、GitHub 分支管理

### 三个分支

```
dev         ← 日常开发，写新功能、修 Bug
staging     ← 测试验证，确认没问题后准备上线
main        ← 正式上线，只接受从 staging 合并的代码
```

### 代码流转流程

```
    开发新功能          测试通过           验收通过
dev ──────────→ staging ──────────→ main
 ↑                                    │
 └────── 如果线上发现 Bug，从 main 拉修复分支 ──┘
```

### 日常操作命令

```bash
# ═══ 开发阶段 ═══

# 切到 dev 分支开发
git checkout dev

# 写代码...改完后提交
git add -A
git commit -m "feat: 新增购物车功能"
git push origin dev

# ═══ 提交测试 ═══

# 合并 dev 到 staging
git checkout staging
git merge dev
git push origin staging
# → 自动部署到测试环境

# ═══ 正式上线 ═══

# 测试通过后，合并 staging 到 main
git checkout main
git merge staging
git push origin main
# → 自动部署到生产环境

# ═══ 紧急修复线上 Bug ═══

# 从 main 拉 hotfix 分支
git checkout main
git checkout -b hotfix/修复描述

# 修完后合并回 main 和 dev
git checkout main
git merge hotfix/修复描述
git push origin main

git checkout dev
git merge hotfix/修复描述
git push origin dev
```

### 提交信息规范

```
feat: 新功能        例：feat: 新增商品搜索
fix: 修复 Bug       例：fix: 修复支付金额计算错误
update: 改进功能     例：update: 优化商品列表加载速度
style: 样式调整      例：style: 调整首页按钮颜色
docs: 文档更新       例：docs: 更新 API 接口文档
```

## 五、App 版本管理

App 不部署到服务器，通过应用商店分发。但通过切换 API 地址区分环境。

### 三个阶段

#### 1. 开发阶段

```bash
# 启动 Expo 开发服务器
npx expo start
```

- 手机装 **Expo Go**，扫码实时预览
- App 连接 `http://localhost:3000`（本地后端）
- 改一行代码，手机上立刻更新

#### 2. 测试阶段

打**测试包**安装到手机：

| 平台 | 分发方式 | 说明 |
|------|---------|------|
| iPhone | TestFlight | 苹果官方测试分发，邀请测试人员安装 |
| Android | APK 直接安装 | 或通过蒲公英 / fir.im 分发给测试人员 |

- 测试包连接 `https://test-api.ai-maimai.com`
- 使用测试数据库，不影响正式环境
- 可以邀请多人同时测试

#### 3. 上线阶段

提交到应用商店审核：

| 平台 | 提交到 | 审核时间 |
|------|--------|---------|
| iPhone | App Store Connect | 1-3 天 |
| Android | 华为/小米/应用宝等 | 1-5 天 |

- 上线包连接 `https://api.ai-maimai.com`
- 使用正式数据库

### App 环境切换

App 内通过环境变量自动切换 API 地址：

```
开发模式 → http://localhost:3000/api/v1
测试包   → https://test-api.ai-maimai.com/api/v1
正式包   → https://api.ai-maimai.com/api/v1
```

## 六、版本号规范

### App 版本号

格式：`主版本.次版本.修订号`，例如 `1.2.3`

| 部分 | 什么时候升 | 例子 |
|------|-----------|------|
| 主版本 | 大改版、不兼容的更新 | 1.0.0 → 2.0.0 |
| 次版本 | 新增功能 | 1.0.0 → 1.1.0 |
| 修订号 | 修复 Bug | 1.0.0 → 1.0.1 |

### 后端 API 版本

API 路径已包含版本号：`/api/v1/...`

如果 API 有不兼容的改动，新增 `/api/v2/...`，旧版本保留一段时间。

## 七、完整工作流程示例

### 场景：开发"商品评价"功能

```
1. [开发] git checkout dev
           写评价功能代码（App + 后端 + 管理后台）
           本地测试通过
           git push origin dev

2. [测试] git checkout staging && git merge dev
           git push origin staging
           → 后端自动部署到 test-api.ai-maimai.com
           → 管理后台自动部署到 test-admin.ai-maimai.com
           打 App 测试包 → 连接测试 API → 给团队测试

3. [上线] 测试通过，无 Bug
           git checkout main && git merge staging
           git push origin main
           → 后端自动部署到 api.ai-maimai.com
           → 管理后台自动部署到 admin.ai-maimai.com
           打 App 正式包 → 提交应用商店审核

4. [发布] 应用商店审核通过 → 用户更新 App → 使用新功能
```

## 八、版本回退

上线后发现严重问题，需要立即回退到之前的版本。

### 网站/管理后台/卖家中心回退（秒级）

网站类项目回退最快，因为 GitHub Actions 自动部署，只需要把代码回退到上一个版本重新推送。

```bash
# 1. 查看最近的提交记录，找到要回退到的版本
git log --oneline -10

# 输出示例：
# a1b2c3d fix: 修复支付bug（← 有问题的版本）
# e4f5g6h feat: 新增评价功能（← 要回退到这里）
# h7i8j9k update: 优化首页

# 2. 回退到指定版本（用新提交的方式，不丢失历史）
git revert a1b2c3d
git push origin main
# → GitHub Actions 自动重新部署，1-2 分钟恢复

# 或者：回退到某个版本之前的所有改动
git revert a1b2c3d..HEAD
git push origin main
```

### 后端 API 回退（分钟级）

```bash
# 方法一：代码回退（推荐）
# 和上面一样用 git revert，推送后 GitHub Actions 自动部署

# 方法二：服务器直接回退（紧急情况）
# SSH 到服务器，用 PM2 回退
pm2 deploy production revert 1    # 回退到上一个部署版本
```

#### 数据库回退

如果新版本修改了数据库结构（migration），回退时需要额外处理：

```bash
# 在服务器上回退数据库迁移
cd /www/wwwroot/api
npx prisma migrate resolve --rolled-back "迁移名称"
```

**重要：** 数据库回退可能导致数据丢失，操作前必须备份：
```bash
pg_dump aimaimai > backup_$(date +%Y%m%d_%H%M%S).sql
```

### App 回退（最慢，需要审核）

App 回退是最复杂的，因为已经发布到应用商店的版本无法直接撤回。

| 方案 | 操作 | 时间 |
|------|------|------|
| **发布修复版本** | 修复 Bug → 打新包 → 提交审核 | 1-5 天 |
| **服务端降级** | 后端 API 做兼容处理，让旧版本正常工作 | 几分钟 |
| **强制更新提示** | App 内弹窗提示用户更新到修复版本 | 依赖用户操作 |

**最佳实践：** App 上线前充分测试，因为回退成本最高。如果出问题，优先通过后端 API 做兼容处理，同时提交修复版本审核。

### 快速回退命令速查

```bash
# ═══ 查看历史 ═══
git log --oneline -10              # 查看最近 10 次提交
git log --oneline --graph          # 图形化查看分支历史

# ═══ 回退单个提交（安全，推荐） ═══
git revert <commit-id>             # 撤销某次提交的改动
git push origin main               # 推送触发自动部署

# ═══ 回退到某个版本（会丢失之后的提交） ═══
git reset --hard <commit-id>       # 本地回退（危险操作！）
git push -f origin main            # 强制推送（会覆盖远程历史！）

# ═══ 临时切回旧版本查看（不修改任何东西） ═══
git checkout <commit-id>           # 查看旧版本代码
git checkout main                  # 切回最新版本
```

### 回退决策流程

```
发现线上问题
    │
    ├── 影响小，可以等？
    │       → 在 dev 修复 → staging 测试 → main 上线（正常流程）
    │
    ├── 影响大，需要立即处理？
    │       → git revert 回退 → 推送自动部署（1-2 分钟恢复）
    │
    └── 数据库也改了？
            → 先备份数据库 → 回退代码 → 回退数据库迁移
```

## 九、注意事项

1. **永远不要直接在 main 分支写代码**，所有开发在 dev 分支进行
2. **测试环境和生产环境数据库完全隔离**，测试不会影响真实用户
3. **上线前必须在 staging 测试通过**，不要跳过测试直接上线
4. **.env 文件不提交到 GitHub**，每个环境有各自的配置
5. **数据库操作要谨慎**，生产库的增删改需要备份后再执行
6. **App 发布需要时间**，应用商店审核 1-5 天，提前规划
