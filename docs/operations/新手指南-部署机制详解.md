# 新手指南：部署机制详解

> **用途**：第一次做完整全栈项目时需要理解的部署/CI/CD 全套概念
> **读法**：从头到尾读一遍，再按需查阅。每一节是独立的，可以反复重读
> **目标**：读完后你能独立解释整个发布链路、自己定位部署故障、敢于独立改 workflow

## 第一部分：你问过的核心问题

### Q1：整套部署是怎么工作的？

```
┌─────────────────────────── 测试环境 (staging 分支) ───────────────────────────┐
│  https://test-admin.ai-maimai.com  →  Nginx → /www/wwwroot/test-admin/      │
│  https://test-seller.ai-maimai.com →  Nginx → /www/wwwroot/test-seller/     │
│  https://test-api.ai-maimai.com    →  Nginx → 反代到 PM2 端口 3001          │
│                                          ↓                                   │
│                                   aimaimai-api-test (NestJS)                 │
│                                          ↓                                   │
│                                   PostgreSQL: testaimaimai 库                │
└──────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────── 生产环境 (main 分支) ─────────────────────────────┐
│  https://admin.ai-maimai.com       →  Nginx → /www/wwwroot/admin/           │
│  https://seller.ai-maimai.com      →  Nginx → /www/wwwroot/seller/          │
│  https://api.ai-maimai.com         →  Nginx → 反代到 PM2 端口 3000          │
│                                          ↓                                   │
│                                   aimaimai-api-prod (NestJS)                 │
│                                          ↓                                   │
│                                   PostgreSQL: aimaimai 库                    │
└──────────────────────────────────────────────────────────────────────────────┘
```

**核心要点**：两套环境用同一台服务器，但目录、PM2 进程、数据库、域名全部独立，互不影响。

### Q2：哪些代码控制 main 和 test 环境？

**只有一个文件**：`.github/workflows/deploy-website.yml`。

它的工作流程：

```
你 git push origin staging  (或 main)
        ↓
GitHub 检测 push 事件
        ↓
触发 workflow .github/workflows/deploy-website.yml
        ↓
Job 1: detect-changes
   ├─ 看你这次 commit 改了哪些目录 (website/admin/seller/backend)
   └─ 看你 push 到哪个分支 (staging/main)
        ↓ 把决策写入"输出变量"
        ↓
   ┌────────────┬────────────┬────────────┬────────────┐
   ↓            ↓            ↓            ↓
 deploy-       deploy-      deploy-       deploy-
 website       admin        seller        backend
 (并行)        (并行)       (并行)        (并行)
```

**关键的"分支 → 环境"开关**（workflow 第 60-83 行）：

```yaml
if [ "${{ github.ref }}" = "refs/heads/main" ]; then
  echo "api_base=https://api.ai-maimai.com/api/v1"        # 生产 API
  echo "admin_target=/www/wwwroot/admin/"                  # 生产部署路径
  echo "backend_pm2_name=aimaimai-api-prod"                # 生产 PM2 进程
  echo "backend_branch=main"                               # 服务器拉哪个分支
else
  echo "api_base=https://test-api.ai-maimai.com/api/v1"   # 测试 API
  echo "admin_target=/www/wwwroot/test-admin/"             # 测试部署路径
  echo "backend_pm2_name=aimaimai-api-test"                # 测试 PM2 进程
  echo "backend_branch=staging"                            # 服务器拉 staging
fi
```

**这就是为什么 staging 分支自动走测试环境、main 分支自动走生产**。没有第二个开关。

### Q3：admin / seller / website 三个前端是怎么部署的？

它们都是 **Vite 静态网站**，部署方式完全相同：

```
GitHub Actions Runner (临时虚拟机)
   ↓
   1. git clone 你的仓库
   2. cd admin && npm ci
   3. npm run build  ← 关键：构建时把 VITE_API_BASE_URL 烧进 JS bundle
   4. rsync admin/dist/ → 服务器:/www/wwwroot/test-admin/   (或 admin/)
   ↓
浏览器拿到的 JS 文件里，API 地址已经是字符串常量，写死在 bundle 里
```

**这就是为什么 admin bundle 里能找到字符串 `test-api.ai-maimai.com/api/v1`**——它在编译期就被注入了。运行时浏览器再也无法改 API 地址。

### Q4：backend 是怎么部署的？

不是构建静态文件，而是**远程 SSH 到服务器执行命令**：

```yaml
ssh root@8.163.16.32 << EOF
  cd /www/wwwroot/aimaimai-staging-src      # (或 -prod-src)
  git fetch origin staging                  # (或 main)
  git reset --hard origin/staging
  cd backend
  npm ci
  npx prisma generate
  npx prisma migrate deploy                 # 数据库迁移
  npm run build
  pm2 reload aimaimai-api-test              # (或 -prod) 平滑重启
EOF
```

后端自己不用"打包成可执行文件"，而是直接在服务器上拉源码、装依赖、起进程。

### Q5：App 怎么测？

**App 和 Web 完全不同**——它不是一个 URL，而是装在手机上的原生应用。**所以 App 不能像 admin/seller 那样"自动部署到测试环境"**。

#### App 怎么知道要连哪个后端？

看 `src/repos/http/config.ts`:

```ts
export const API_BASE_URL =
  process.env.EXPO_PUBLIC_API_BASE_URL || 'http://localhost:3000/api/v1';
```

环境变量从根目录的 `.env` 读取。

#### 测试 App 的 4 种方式（按推荐顺序）

**方式 A：本地 Expo + 手机扫码连测试 API（最快，最常用）**

```bash
# 1. 改根目录 .env
echo "EXPO_PUBLIC_USE_MOCK=false" > .env
echo "EXPO_PUBLIC_API_BASE_URL=https://test-api.ai-maimai.com/api/v1" >> .env

# 2. 启动 Expo
npx expo start

# 3. 手机装 Expo Go App，扫终端二维码
```

代码在你电脑上跑，修改即时刷新，但数据来自真实测试后端，可以联调。开发期主力方式。

**方式 B：EAS Build 打"测试包"分发给测试人员**

测试人员不可能装 Expo Go + 扫你的二维码。所以打独立的内部版 App：

```bash
# 一次性配置 eas.json：
# - "preview" profile: 注入 EXPO_PUBLIC_API_BASE_URL=test-api...
# - "production" profile: 注入 EXPO_PUBLIC_API_BASE_URL=api...

eas build --profile preview --platform ios   # 打测试包
eas build --profile production --platform ios # 打生产包
```

iOS 走 TestFlight，Android 走内部分发链接。

> 你现在还没建 `eas.json`，所以这步还没准备好。等正式有测试人员要测时再做。

**方式 C：浏览器跑 Expo Web 版**

```bash
npx expo start --web
```

很多原生组件（相机、扫码、原生地图）web 不支持。只适合临时验证少数页面。

**方式 D：在 App 里做一个"环境切换"开关（高级）**

在 App 设置页加一个隐藏入口，让测试人员自己切 API base。这样一个 App 安装包就能测两套环境。v1.0 不必做。

### Q6：App 怎么上线？

App 不会被 GitHub Actions 自动部署，所有 App 改动都需要打新包，用户重新装。流程：

```
代码改完
   ↓
EAS Build production profile
   ↓
拿到 .ipa (iOS) / .apk (Android)
   ↓
iOS: 上传到 App Store Connect → TestFlight 内测 → 提审 → 上架
Android: 上传到华为/小米/OPPO/vivo/应用宝 → 各自审核 → 上架
   ↓
用户在手机应用商店看到"有更新" → 点更新 → 装新版
```

这是原生 App 的天然限制，不是我们选择。

## 第二部分：你可能马上会问的基础概念

### Q7：为什么要有两个环境？直接 push 生产不行吗？

**不行。** 直接 push 生产意味着：

- 你写的 bug 直接被真实用户撞到（损失订单、损失信任）
- 数据库 migration 写错可能导致生产数据丢失
- 第三方支付/物流回调用真实 API，错误的话真的会扣钱
- 想验证 bug 修没修好，只能等用户帮你测

测试环境的本质是 **"给开发自己当用户的免费试错场"**。代价：服务器多跑一份进程、多一个数据库——基本免费。收益：避免一个生产事故就赚回 1000 倍。

### Q8：什么是 PM2？为什么需要它？

NestJS 后端是一个 Node.js 进程。如果直接 `node dist/main.js` 启动：

- 终端关闭进程就死了
- 进程崩溃没人重启
- 没有日志归档
- 升级代码要手动停旧进程、起新进程

PM2 是 Node.js 的**进程管理器**，解决这些问题：

```bash
pm2 start dist/main.js --name aimaimai-api-test  # 后台运行
pm2 list                                          # 看所有进程状态
pm2 logs aimaimai-api-test                        # 实时看日志
pm2 reload aimaimai-api-test                      # 平滑重启（零停机）
pm2 restart aimaimai-api-test                     # 强制重启
pm2 stop aimaimai-api-test                        # 停止
```

**最关键的是 `pm2 reload`**：它先启动新进程，等新进程就绪再杀旧进程，用户感觉不到中断。这就是"平滑重启"或"零停机部署"。

### Q9：什么是 Nginx？为什么不让 Node 直接对外？

Node 本身可以监听 80/443 端口直接对外，但有几个问题：

1. **不能同时挂多个域名**：Node 一个进程绑一个端口，但你有 8 个域名
2. **HTTPS 证书管理复杂**：Node 处理 SSL 不如 Nginx 成熟
3. **静态文件慢**：Node 发 HTML/JS/CSS 比 Nginx 慢 10 倍
4. **没有访问日志/限流/Gzip 压缩**

Nginx 是**反向代理**：

```
浏览器 → 443 端口 → Nginx → 解开 SSL → 看 Host 头是哪个域名
                                ↓
       ┌─────────────────────────┼─────────────────────────┐
       ↓                         ↓                         ↓
   admin.ai-maimai.com      api.ai-maimai.com       seller.ai-maimai.com
       ↓                         ↓                         ↓
   静态文件目录                反代到 :3000              静态文件目录
```

Nginx 一个进程同时挂 8 个域名，每个域名走不同规则（静态文件 / 反代到 Node / 重定向）。

### Q10："构建（build）"是什么意思？为什么前端要构建？

你写的 React/Vue 代码：

```tsx
import { Button } from 'antd'
export default function Page() {
  return <Button>点我</Button>
}
```

浏览器**不认识** `import`、不认识 `JSX`、不认识 `TypeScript`。需要把这些转成浏览器能跑的：

```js
// 构建后（简化）
import{B as a}from"./vendor.js";export default function(){return a({children:"点我"})}
```

**构建做的事**：
1. TypeScript → JavaScript
2. JSX → React.createElement 调用
3. 多个 .ts 文件合并成几个大 .js（减少 HTTP 请求）
4. 压缩、混淆、删空格（文件更小）
5. **替换环境变量**：把代码里的 `process.env.VITE_API_BASE_URL` 替换成实际字符串

**最后一步是关键**：构建时 `VITE_API_BASE_URL=https://test-api...`，构建后这个字符串就**永久写死**在 JS 文件里。所以同一份源码可以构建出测试版（连 test-api）和生产版（连 api）两个不同的 dist。

### Q11：什么是 rsync？为什么不直接 cp？

`rsync` 是"智能复制"工具：

- **只传变化的文件**（cp 是全量复制）
- **可以删除目标里多余的文件**（用 `--delete`）
- **可以跨机器**（用 SSH）
- **断点续传**（网断了重传只传没传完的）

GitHub Actions 把构建产物 rsync 到服务器：

```bash
rsync -avz --delete admin/dist/ root@8.163.16.32:/www/wwwroot/test-admin/
#       │             │              │
#       │             │              └── 远端目标
#       │             └── 本地源（注意末尾 /）
#       └── -a 保留权限 -v 详细输出 -z 压缩传输
# --delete = 远端有但本地没有的文件全删（保持完全一致）
```

**为什么用 --delete**：上次构建产生了 `index-OLDHASH.js`，这次构建变成 `index-NEWHASH.js`。不删旧的会越积越多，半年后服务器磁盘爆了。

### Q12：什么是 SSH？为什么用它部署？

SSH 是"加密的远程登录"协议。GitHub Actions 部署后端时：

```
GitHub Runner (临时虚拟机)
   ↓ 用 SSH 私钥（存在 GitHub Secrets 里）
   ↓ 加密连接到服务器
   ↓
服务器 (8.163.16.32)
   ↓ 用 SSH 公钥（存在 ~/.ssh/authorized_keys 里）验证私钥
   ↓ 验证通过，让 Runner 执行 shell 命令
```

**为什么用 SSH 不用密码**：
- 私钥比密码长 256 倍，暴力破解需要几万年
- 私钥不在网络上传输（密码会传），不会被中间人截获
- 私钥可以加密存储，丢了文件也不一定能用

**SSH 密钥对**：
- **私钥** (`id_ed25519`)：你保管，相当于钥匙，**绝不外泄**
- **公钥** (`id_ed25519.pub`)：放服务器上，相当于锁

公钥放再多地方都没事（不能反推出私钥），但私钥泄露 = 服务器拱手送人。

### Q13：什么是 GitHub Secrets？

GitHub 仓库里你可以存"密钥"，workflow 运行时能读取，但仓库里其他人**看不到内容**（只能改、不能读）：

- `SERVER_SSH_KEY`：服务器 SSH 私钥
- `SERVER_HOST`：服务器 IP `8.163.16.32`
- `SERVER_USER`：登录用户 `root`

为什么不直接写在 workflow yml 里？因为 yml 会被 commit 到仓库，任何人能看到。

**注意**：GitHub 网页粘贴 secret 时会**吃掉末尾换行**。SSH 私钥必须以换行结尾，否则 OpenSSH 拒绝加载。所以建议用命令行设置：

```bash
gh secret set SERVER_SSH_KEY --repo flyspaceden/aimaimia < ~/.ssh/aimaimai_deploy
```

### Q14：`npm ci` 和 `npm install` 的区别？

| 命令 | 行为 | 何时用 |
|------|------|--------|
| `npm install` | 按 `package.json` 装，可能更新 `package-lock.json` | 本地开发，新增/升级依赖 |
| `npm ci` | 严格按 `package-lock.json` 装，不修改 lock，速度更快 | CI/部署，**保证版本完全一致** |

**部署必须用 `npm ci`**：你本地装的版本和服务器一致，避免"我电脑能跑，服务器跑不起来"的坑。

### Q15：`prisma migrate deploy` 在做什么？

数据库表结构会随业务演进变化（加字段、改类型、加索引）。Prisma 用 **migration 文件** 管理变化：

```
backend/prisma/migrations/
├── 20260101_init/migration.sql              ← 初始建表
├── 20260215_add_user_phone/migration.sql    ← 给 User 表加 phone 字段
├── 20260330_add_after_sale/migration.sql    ← 新增售后表
└── ...
```

每个 migration 是一段 SQL。Prisma 维护一个 `_prisma_migrations` 表记录"已经执行过哪些"。

`prisma migrate deploy` 做的事：
1. 连数据库，读 `_prisma_migrations` 表
2. 找出 migration 目录里"还没执行过的"
3. 按时间顺序逐个执行
4. 执行完登记到 `_prisma_migrations`

**关键**：这是**只前进不后退**的。一旦 migration 执行了，想撤销只能写新的反向 migration。所以**写 migration 必须谨慎**：
- 不删字段（改成 nullable，让旧代码还能跑）
- 不改字段类型（先加新字段，迁数据，再删旧字段）
- 不重命名（同上）

### Q16：测试库和生产库是怎么隔离的？

PostgreSQL 是一个数据库**服务器**，里面可以建多个独立的"库"：

```
PostgreSQL 服务 (端口 5432)
├── 库: testaimaimai      ← 测试用，密码 wPjtreEp2GfxGrwS，用户 testaimaimai
└── 库: aimaimai          ← 生产用，密码 FwfAjyYY7TxyNrXM，用户 aimaimai
```

不同库的表、数据、用户权限完全独立。即使测试环境跑 `DROP TABLE users;`，生产库纹丝不动。

后端通过 `.env` 里的 `DATABASE_URL` 决定连哪个库：

```bash
# /www/wwwroot/aimaimai-staging-src/backend/.env
DATABASE_URL="postgresql://testaimaimai:wPjtreEp2GfxGrwS@127.0.0.1:5432/testaimaimai"

# /www/wwwroot/aimaimai-prod-src/backend/.env
DATABASE_URL="postgresql://aimaimai:FwfAjyYY7TxyNrXM@127.0.0.1:5432/aimaimai"
```

### Q17：`EXPO_PUBLIC_*` 前缀是什么意思？

Expo 区分两类环境变量：

- 没有前缀的：**只在构建时可见**，源码里读不到（用于服务端密钥）
- `EXPO_PUBLIC_` 前缀：**会被打进 App 包**，源码里能读到

为什么要区分？因为 App 装在用户手机上，**用户能反编译看到所有代码**。如果你把 `ALIPAY_PRIVATE_KEY` 直接放进 App，所有用户都能拿到你的支付宝私钥，能假装是你的服务器收钱。

**铁律：永远不要把任何密钥放进 `EXPO_PUBLIC_*`**。所有涉密操作必须走后端 API。

## 第三部分：实操技能

### Q18：怎么看部署成功了没？

**方法 1：GitHub 网页**
仓库 → **Actions** 标签 → 看最新一行：
- ✅ 绿色对勾 = 成功
- ❌ 红色叉 = 失败（点进去看哪个 step 红的）
- 🟡 黄色圆 = 进行中

**方法 2：直接访问域名**
浏览器打开 `https://test-admin.ai-maimai.com/`，看版本号或新功能是否生效。

**方法 3：查后端进程**
```bash
ssh root@8.163.16.32
pm2 list                           # 看 PM2 进程状态
pm2 logs aimaimai-api-test         # 实时日志（Ctrl+C 退出）
pm2 logs aimaimai-api-test --lines 200   # 最近 200 行
```

### Q19：部署失败了怎么办？

**先定位是哪一步失败**：GitHub Actions 页面 → 点失败的 run → 展开红色 step。

**常见失败原因**：

| 错误信息 | 原因 | 修复 |
|---------|------|------|
| `npm ci: lock file out of sync` | 你改了 package.json 但忘了 commit lock | 本地 `npm install` 后重新 commit |
| `tsc: error TS...` | TypeScript 编译错 | 本地 `npm run build` 复现 |
| `Permission denied (publickey)` | SSH 密钥配置错 | 重设 GitHub Secret |
| `pm2: command not found` | 服务器环境出问题 | SSH 上去重装 PM2 |
| `Migration failed` | Prisma migration SQL 错 | 看错误信息修 SQL，本地用测试库验证 |

**部署到一半失败，环境会怎样？**

- 前端：rsync 是原子的，只有完全成功才会替换。失败 → 旧版本还在跑
- 后端：可能停在 npm ci / build / migrate 任一步。需要 SSH 上去手动检查

### Q20：怎么回滚一个坏的部署？

**前端回滚**：
```bash
git revert <bad-commit-hash>      # 创建一个"反向 commit"
git push origin staging           # 触发新一次部署
```

不要用 `git reset --hard` 强推（会丢历史，多人协作时会打架）。

**后端回滚**：

```bash
ssh root@8.163.16.32
cd /www/wwwroot/aimaimai-staging-src
git log --oneline -10              # 看最近 10 个提交
git reset --hard <good-commit>     # 回退到好的版本
cd backend
npm ci
npm run build
pm2 reload aimaimai-api-test
```

**数据库回滚**：**没有简单方法**。所以 migration 必须设计成向后兼容。如果非要回滚：
1. 写一个反向 migration（把加的字段删掉之类）
2. 或者从备份恢复（pg_dump 备份必须提前做）

### Q21：怎么读后端日志？

```bash
ssh root@8.163.16.32

# 实时滚动看
```

### Q22：怎么紧急修复线上 bug（hotfix）？

正常流程是 staging → main，但生产着火时：

```bash
# 1. 直接在 main 修
git checkout main
git pull origin main
# ... 改代码 ...
git add -A && git commit -m "hotfix: 紧急修复 X 问题"
git push origin main          # 触发生产部署

# 2. 修完立刻同步回 staging（避免下次合并时冲突）
git checkout staging
git merge main
git push origin staging
```

**只在真的紧急时这么干**。能等的话还是走标准流程。

## 第四部分：安全与凭据

### Q23：如果不小心 commit 了密钥怎么办？

**坏消息**：Git 是永久历史，单纯 `git rm` 删除文件后，旧 commit 里还能看到。

**做法**：
1. **立刻撤销这个密钥**（去阿里云/支付宝后台重生成）
2. 用新密钥替换 `.env`
3. 提交带新密钥的 .env（如果该 .env 不该 commit，先把它加到 `.gitignore`）
4. 旧密钥即使被人翻历史拿到也已经失效

**不要试图用 `git filter-branch` 重写历史**——push 上去会让所有协作者地狱。直接换密钥是唯一靠谱方法。

**预防**：
- 任何 `.env*` 都加 `.gitignore`
- commit 前 `git diff --staged` 扫一眼
- 用 `git-secrets` 工具自动检测

### Q24：密码本怎么管？

`docs/operations/密码本.md` 是**唯一可以明文存凭据的地方**，且：
- 已经在 `.gitignore` 里（不会被 commit）
- 只在你本机存
- 任何文档要引用密钥，**必须用占位符** `<TEST_DB_PASSWORD>`

**把密码本备份**：你 Mac 坏了密码本就没了。建议：
- iCloud Drive / 坚果云同步
- 1Password / Bitwarden 之类密码管理器
- 至少打印一份锁抽屉

### Q25：服务器被入侵了怎么办？

**症状**：
- 莫名 CPU/内存 100%（被挖矿）
- 阿里云告警异常出网流量（被当跳板）
- 数据库被加密 + 勒索信

**应急步骤**：
1. 阿里云控制台 → 安全组 → **关掉所有入站规则**（断网隔离）
2. 看 `pm2 logs` / `/var/log/nginx/access.log` 找入侵时间点
3. 重置所有密码、SSH key、JWT secret、数据库密码
4. 从数据库**异机备份**恢复
5. 重装系统、重新部署

**预防**：
- SSH 禁用密码登录，只允许密钥
- 防火墙只开必要端口（80/443/SSH/宝塔）
- 定期 `pg_dump` 备份到 OSS（在另一个账号下）
- 不要给 root 用户跑应用（用专门的低权限用户）

## 第五部分：常见疑问

### Q26：为什么测试环境用 mock 短信，生产用真实短信？

发短信花钱（一条几分钱）。开发期反复测注册流程，可能一天发几百条。看看 `.env.staging`：

```
SMS_MOCK=true   ← 测试环境不真发，控制台打印验证码
```

测试时打开后端日志能直接看到验证码，不用真的拿手机收。生产环境必须 `SMS_MOCK=false`。

---

### Q27：为什么改了 workflow 文件 push 没触发部署？

看 workflow 的 `paths`：

```yaml
on:
  push:
    paths:
      - 'website/**'
      - 'admin/**'
      - 'seller/**'
      - 'backend/**'
      - '.github/workflows/deploy-website.yml'
```

只改这 5 个目录之一才会触发。如果你只改了 `docs/`，workflow 不会跑（这是故意的，文档改动不需要部署）。

但 workflow 自己改了会触发——这次 detect-changes 会看，4 个 deploy-* job 全部 skip（因为代码没变）。所以你看到 "成功但只跑了 7 秒"。

### Q28：手动触发部署什么时候用？

GitHub → Actions → Deploy Sites & Backend → **Run workflow**：

适用场景：
1. 改了配置但代码没变（路径检测不会触发，需要手动）
2. 服务器异常重新跑一遍部署
3. 强制重新构建某个站点（比如怀疑 dist 损坏）
4. 第一次部署一个新环境（之前没 commit 触发）

`deploy_target` 选项：
- `all` 全部
- `website` / `admin` / `seller` / `backend` 单选

### Q29：环境变量改了怎么生效？

**前端**：必须**重新构建**（重新 push 触发，或手动 workflow_dispatch）。因为变量被烧进 bundle 了。

**后端**：改服务器上的 `.env` 后，`pm2 reload aimaimai-api-test --update-env`。

```bash
ssh root@8.163.16.32
nano /www/wwwroot/aimaimai-staging-src/backend/.env
# ... 改完保存 ...
pm2 reload aimaimai-api-test --update-env   # 必须加 --update-env，否则不读新 .env
```

### Q31：怎么知道服务器要扩容了？

```bash
ssh root@8.163.16.32

# CPU 使用率
top
# 看右上角 load average，3 个数字代表 1/5/15 分钟平均
# 数字 < CPU 核数（你是 4 核）= 健康
# 长期 > 8 = 该升 8 核了

# 内存
free -h
# available 行剩多少，剩 < 1G 该升内存

# 磁盘
df -h
# 看 / 和 /www 用了多少，剩 < 20% 该清理或扩容

# PM2 进程内存
pm2 list
# 看 mem 列，单进程长期 > 500MB 可能有内存泄漏
```

### Q32：什么是"灰度发布"？

直接所有用户都用新版本风险高。"灰度"= 先放 1%~10% 用户用新版本，观察一阵没问题再全量。

实现方式：
1. **按用户 ID 取模**：`if (userId % 100 < 10) 用新逻辑`
2. **Feature Flag 服务**：管理后台开关控制
3. **Nginx 流量切分**：10% 请求路由到新后端

v1.0 你不需要灰度（用户少，直接全量）。等用户量大了再考虑。
